01初動トリアージLive Response Triage

侵害の疑いがある端末を最初に触るときの原則は「揮発性の高いものから、書き換えを最小に」。プロセス・ネットワーク接続・ログオンセッションはシャットダウンで失われるため、電源を落とす前に取得する。KAPE と Velociraptor が Windows ライブトリアージの事実上の標準になっている。

揮発性データの取得順序

RFC 3227 の Order of Volatility に沿う。上位ほど先に取る。

  • CPU レジスタ・キャッシュ (取得不可、諦める)
  • RAM (memory dump)
  • プロセス / ネットワーク / ARP / ログオンセッション
  • ディスク上の一時ファイル / ページファイル
  • ファイルシステム (MFT / USN / レジストリ / イベントログ)
  • リモートログ / バックアップ

KAPE トリアージ収集

KAPE (Kroll Artifact Parser and Extractor) は Target (収集) と Module (解析) の 2 段構成。まず Target で必要成果物を一括取得。

cmd
:: 基本的なトリアージ収集 (KapeTriage は必要成果物をまとめた target)
kape.exe --tsource C: --tdest <OUT_DIR>\triage --target KapeTriage --vhdx triage

:: 収集 → 解析まで一気通貫 (Module: !EZParser で EZ tools 全体を実行)
kape.exe --tsource C: --tdest <OUT_DIR>\triage --target KapeTriage ^
  --mdest <OUT_DIR>\parsed --module !EZParser --mflush
tip --vhdx で VHDX にまとめると事後解析側で仮想マウントできる。取得側 (被害端末) にはできる限りツール本体だけを置き、成果物は共有先や外付けメディアへ書き出す。

Velociraptor による群制御

複数ホストを跨いだトリアージなら Velociraptor。VQL でクエリを書き、Windows.KapeFiles.Targets のようなアーティファクトを配布して並列収集できる。

bash
# オフライン (単独ホスト) 収集: KAPE 相当のトリアージパッケージを生成
velociraptor.exe --config <CLIENT_CONFIG.yaml> \
  artifacts collect Windows.KapeFiles.Targets \
  --args Device=C: --args _SANS_Triage=Y \
  --output <OUT_DIR>\triage.zip

最小構成のワンライナー系

KAPE も Velociraptor も持ち込めない場面用の最低限。順序を崩さないこと。

powershell
# ネットワーク接続 + プロセスの相関
Get-NetTCPConnection | Where-Object State -eq Established |
  Select-Object LocalAddress,LocalPort,RemoteAddress,RemotePort,OwningProcess,
    @{n='Process';e={(Get-Process -Id $_.OwningProcess).Path}}

# ログオンセッション (対話・ネットワーク・キャッシュ含む)
qwinsta ; query user
Get-CimInstance Win32_LogonSession | Select LogonId,LogonType,StartTime

# 直近スケジューラ実行 / サービス状態
Get-ScheduledTask | Where-Object State -eq Ready |
  Select TaskName,TaskPath,@{n='LastRun';e={($_ | Get-ScheduledTaskInfo).LastRunTime}}
要点 / key takeaways
  • 取得順序は Order of Volatility。RAM を後回しにしない。
  • ライブでもディスク書き込みは最小に。ツールと成果物は別領域。
  • 1 台なら KAPE、複数台なら Velociraptor が現実解。

02メモリ解析Memory Forensics

ディスクに残らない痕跡 — 復号された文字列、注入コード、reflectively loaded assembly、消えたプロセスの子孫関係 — はメモリにしか無い。取得と解析を分けて考える。

メモリ取得

winpmem / DumpIt / Magnet RAM Capture のいずれか。raw または AFF4 で保存。取得は必ず外部媒体へ書き出す。

cmd
:: winpmem: raw 形式で外付けメディアへ
winpmem_mini.exe <EXTERNAL_DRIVE>\<HOST>.raw

:: DumpIt: プロンプト無しで実行
DumpIt.exe /Q /O <EXTERNAL_DRIVE>\<HOST>.dmp

Volatility 3

Volatility 3 はプロファイル指定が不要になった。頻用プラグイン。

bash
# プロセスツリー
vol -f <MEMORY.raw> windows.pstree

# DLL 一覧 (特定 PID)
vol -f <MEMORY.raw> windows.dlllist --pid <PID>

# 隠しプロセス / unlinked / EPROCESS 差分
vol -f <MEMORY.raw> windows.psscan
vol -f <MEMORY.raw> windows.psxview

# 注入コード検出 (RWX / private memory + executable など)
vol -f <MEMORY.raw> windows.malfind

# ネットワーク接続
vol -f <MEMORY.raw> windows.netscan

# コマンドライン (プロセス起動引数)
vol -f <MEMORY.raw> windows.cmdline

# 資格情報関連 (LSA secrets / hashes / cachedump)
vol -f <MEMORY.raw> windows.hashdump
vol -f <MEMORY.raw> windows.lsadump

# プロセス実行イメージのダンプ (二次解析用: 逆アセンブラ / strings)
vol -f <MEMORY.raw> -o <OUT> windows.pslist --pid <PID> --dump

# プロセスがオープンしているファイル (DLL / 開いたドキュメント等)
vol -f <MEMORY.raw> -o <OUT> windows.dumpfiles --pid <PID>

MemProcFS

メモリイメージを FUSE / ドライブレターとしてマウントし、ファイルシステム的にアクセスできる。ad-hoc 探索と grep の併用が強い。

cmd
:: メモリを M: にマウント
MemProcFS.exe -device <MEMORY.raw> -mount M:

:: プロセスごとに /pid/<PID>/ 以下でツリー展開される
dir M:\name\lsass.exe
type M:\registry\hive_list.txt

読み方の指針

  • 親子関係の異常: svchost.exe の親が services.exe でない、rundll32.exe が Office から派生している、など。
  • 署名無し + 高権限: NT AUTHORITY\SYSTEM で動く非署名モジュール、想定外の lsass.exe 内 DLL。
  • ネットワーク + 短命プロセス: netscan で外向き接続を持つが pstree で見えない → psscan で拾い直す。
  • malfind の false positive: JIT (Chrome / .NET) は通常 RWX を持つ。文字列 / import / エントリポイント命令列で切り分ける。
要点 / key takeaways
  • 取得と解析は分離。取得は最短時間で、解析は複製で。
  • pslist / psscan / psxview の差分こそがメモリ解析の意義。
  • malfind は起点、結論ではない。文字列と挙動で裏を取る。

EvtxECmd による正規化

Eric Zimmerman の EvtxECmd は evtx を CSV / JSON に平坦化。Timeline Explorer や後段の SIEM に流し込むための基本前処理。

cmd
:: 単一ファイル → CSV
EvtxECmd.exe -f <EVTX_FILE> --csv <OUT> --csvf events.csv

:: フォルダ再帰 + Map (Sysmon / PS など既知イベントを構造化)
EvtxECmd.exe -d <EVTX_DIR> --csv <OUT> --csvf all.csv

chainsaw / hayabusa による Sigma スイープ

Sigma ルールを evtx に直接ぶつけて既知パターンを一気に検出する。ハントの初手として費用対効果が高い。

bash
# chainsaw: Sigma + built-in ハンティング
chainsaw hunt <EVTX_DIR> -s sigma/ --mapping mappings/sigma-event-logs-all.yml \
  --output results.csv --csv

# hayabusa: 日本製、独自 Yaml ルールを組み込み
hayabusa csv-timeline -d <EVTX_DIR> -o timeline.csv

読み方の指針

  • 4624 は LogonType で分ける: 2=対話, 3=ネットワーク, 10=RemoteInteractive (RDP), 9=NewCredentials (runas /netonly, 横展開兆候)。
  • 4688 はコマンドライン記録の有効化が前提。無効なら Sysmon 1 で補う。
  • PowerShell 4104 はスクリプト難読化の平文化に効く。Base64 でも復号済み文字列がここに残ることが多い。
  • Sysmon 10 の対象が lsass.exe: 資格情報窃取 (mimikatz 系) の代表兆候。
要点 / key takeaways
  • Security チャネル単独では見えない挙動が Sysmon で埋まる。
  • evtx はまず正規化 (EvtxECmd) → Sigma スイープ (chainsaw) の順で費用対効果を最大化。
  • 4624 LogonType と 4688 CommandLine の 2 つが読めれば大半のシナリオは追える。

04ファイルシステム痕跡Filesystem Artifacts

ディスクは削除されても消えない情報の宝庫。NTFS の $MFT / $LogFile / $UsnJrnl、Windows 実行痕跡の Prefetch / ShimCache / AmCache を押さえる。

MFT: ファイルの母台帳

$MFT はボリューム上の全ファイルのメタデータを持つ。削除ファイルもエントリが再利用されるまで残る。MFTECmd で解析。

cmd
:: $MFT を CSV に
MFTECmd.exe -f <PATH>\$MFT --csv <OUT> --csvf mft.csv

:: $J (USN Journal) — ファイルの変更ログ (作成/削除/リネーム)
MFTECmd.exe -f <PATH>\$J --csv <OUT> --csvf usn.csv

MFT レコードには $STANDARD_INFORMATION (SI) と $FILE_NAME (FN) の 2 種類のタイムスタンプがある。SI はユーザ空間から書き換え可能、FN はカーネル空間で更新される。SI と FN の乖離は timestomping の代表兆候。

Prefetch: 実行された事実

%SystemRoot%\Prefetch\*.pf に、実行された EXE のメタデータと参照ファイル一覧が記録される。「実行された」という事実の一次証拠。

cmd
PECmd.exe -d C:\Windows\Prefetch --csv <OUT> --csvf prefetch.csv

取れる情報: 実行回数、最終 8 回分の実行時刻、参照した DLL / ファイル、実行元パス。Windows Server では既定で無効なので注意。

ShimCache (AppCompatCache)

アプリ互換性のためのキャッシュだが、EXE のフルパスとタイムスタンプが残る。実行された事実の記録ではないが「システムが認識した」証跡になる。SYSTEM ハイブに格納。

cmd
AppCompatCacheParser.exe -f <PATH>\SYSTEM --csv <OUT> --csvf shimcache.csv

AmCache

%SystemRoot%\AppCompat\Programs\Amcache.hve。実行された EXE のフルパス、SHA1、コンパイル時刻、初回実行時刻を含む。ShimCache より情報量が多い。

cmd
AmcacheParser.exe -f <PATH>\Amcache.hve --csv <OUT>

その他の実行痕跡

  • UserAssist (NTUSER): GUI から起動されたプログラム。実行回数と最終実行時刻。
  • BAM/DAM (SYSTEM): Background Activity Moderator。バックグラウンド実行の記録。
  • RecentDocs / OpenSaveMRU (NTUSER): 開かれたドキュメント。
  • LNK / JumpList: LECmd / JLECmd で解析。参照先ファイルの元パス、ボリューム情報。
  • ShellBags (NTUSER / UsrClass): エクスプローラで開いたフォルダ。削除フォルダの存在証拠にもなる。
要点 / key takeaways
  • 「実行された」の証拠は Prefetch > AmCache > UserAssist の順で強い。
  • ShimCache は「システムが見た」レベル。実行の証明にはならない。
  • MFT の SI / FN 乖離、USN Journal の削除記録が anti-forensics 発見の起点。

05レジストリ解析Registry Forensics

Windows のレジストリはハイブファイルとしてディスクに存在する。オンライン端末での reg query ではなく、コピーしたハイブをオフラインで解析するのがフォレンジックの原則。

ハイブの所在

ハイブパス用途
SYSTEM%SystemRoot%\System32\config\SYSTEMサービス, ShimCache, USBSTOR, ComputerName
SOFTWARE%SystemRoot%\System32\config\SOFTWAREインストール済み / Run キー / OS 情報
SAM%SystemRoot%\System32\config\SAMローカルアカウント
SECURITY%SystemRoot%\System32\config\SECURITYLSA secrets / ポリシー
NTUSER.DAT%UserProfile%\NTUSER.DATユーザ固有 (Run, UserAssist, RecentDocs)
UsrClass.dat%UserProfile%\AppData\Local\Microsoft\Windows\UsrClass.datShellBags, COM 関連

RegRipper

プラグイン式のハイブ解析ツール。ハイブごとに -p でプラグイン指定、-a で全プラグイン一括。

bash
# 全プラグイン一括 (SOFTWARE ハイブ)
rip.pl -r <PATH>/SOFTWARE -a > software_all.txt

# プロファイル指定 (定型プラグイン群)
rip.pl -r <PATH>/SYSTEM -f system

# 単発: Run キーだけ
rip.pl -r <PATH>/SOFTWARE -p run

RECmd (Eric Zimmerman)

Batch file (YAML) で複数キーを一括抽出。KAPE から呼ぶ場合の標準。

cmd
RECmd.exe --f <PATH>\SOFTWARE --bn BatchExamples\Kroll_Batch.reb ^
  --csv <OUT> --csvf software.csv

頻用キー

  • SOFTWARE\Microsoft\Windows\CurrentVersion\Run|RunOnce: 起動時実行 (機械単位)。
  • NTUSER.DAT\Software\...\Run: 起動時実行 (ユーザ単位)。
  • SYSTEM\CurrentControlSet\Services: サービス定義。ImagePath と Start=2 (Auto) を注視。
  • SYSTEM\Select\Current: どの ControlSet が現行か。
  • SOFTWARE\Microsoft\Windows NT\CurrentVersion\Winlogon\Userinit|Shell: ログオン時実行の書き換え。
  • SYSTEM\ControlSet001\Enum\USBSTOR: USB 接続履歴。
  • SOFTWARE\Microsoft\Windows NT\CurrentVersion\Image File Execution Options\<exe>: Debugger 値による永続化 (Sticky keys 系)。
要点 / key takeaways
  • 解析は常にオフラインハイブで。オンラインの reg query は書き換えを混入させる。
  • SYSTEM の Select を読まないと ControlSet を取り違える。
  • Run / Services / Winlogon / IFEO の 4 箇所は必ず舐める。

06タイムライン統合Super Timeline

個別成果物の解析結果を時系列に統合する工程。侵入起点 (patient zero) と横展開の順序が可視化される。plaso / log2timeline が事実上の標準。

plaso による super timeline

bash
# 収集 (mount 済みイメージ or ライブディスクを解析)
log2timeline.py --storage-file <OUT>/timeline.plaso <MOUNTED_IMAGE>

# CSV (l2tcsv) 出力
psort.py -o l2tcsv -w <OUT>/timeline.csv <OUT>/timeline.plaso

# 期間絞り (2026-07-01 〜 2026-07-10)
psort.py -o l2tcsv -w <OUT>/timeline_slice.csv <OUT>/timeline.plaso \
  "date > '2026-07-01 00:00:00' AND date < '2026-07-10 23:59:59'"

Timeline Explorer での読み方

Eric Zimmerman の Timeline Explorer は CSV を高速に絞り込む閲覧環境。plaso / EvtxECmd / MFTECmd の CSV を同じインタフェースで扱える。

  • 疑わしい 1 事象を起点に前後 5 分を精読 (前後関係で因果を推定)。
  • ホスト単位の色分けを維持し、複数ホストの CSV を統合しない。
  • タグ付け (フラグ列) で「確定」「疑い」「除外」を分けて減衰させる。

タイムライン整形の要点

  • TZ の一貫化: 収集元はローカル TZ、解析側で UTC に正規化。log2timeline / psort ともに TZ 指定を明示 (出力モジュールにより --output-time-zone 等)。
  • タイムスタンプの種類: MFT SI / FN、Prefetch 実行時刻、EventRecord の TimeCreated は意味が違う。混ぜない。
  • ノイズ削減: OS 更新や Defender スキャンは大量イベントを生む。ブラックリストで抑える。
要点 / key takeaways
  • super timeline はゴールではなく作業台。読み方が全て。
  • TZ を先に決める。混在は後戻り不能なコストになる。
  • タイムスタンプの意味を混ぜない (書換可 SI と不変 FN、実行時刻とファイル時刻)。

07永続化ハンティングPersistence Hunting

侵害端末で「排除しきれたか」を判断する最終フェーズ。MITRE ATT&CK の TA0003 (Persistence) を網羅的に確認する。オフラインハイブでの照合が原則。

確認すべき箇所 (優先順)

  1. Run / RunOnce キー (HKLM / HKCU)
  2. Services (ImagePath, ServiceDLL, Start=2)
  3. Scheduled Tasks (%SystemRoot%\System32\Tasks\* の XML)
  4. WMI Event Subscription (ROOT\subscription: __EventFilter, EventConsumer, FilterToConsumerBinding)
  5. Winlogon (Userinit, Shell, Notify)
  6. IFEO Debugger (Image File Execution Options\<exe>\Debugger)
  7. COM Hijacking (HKCU\Software\Classes\CLSID)
  8. AppInit_DLLs / LSA Provider / Netsh Helper
  9. Startup フォルダ (%AppData%\...\Startup, %ProgramData%\...\Startup)
  10. Print Monitor / SSP (資格情報経路の乗っ取り)

Autoruns (オフライン)

Autoruns は -z でオフラインシステムを解析できる。ライブでは書き換え混入があるため IR 目的ではオフラインを推奨。

cmd
:: オフライン: 収集済みディスクをマウントして指定
autorunsc.exe -z <MOUNTED_WIN_DIR> <USER_PROFILE_DIR> -a * -c -h -s -nobanner > autoruns.csv

Scheduled Tasks の解析

powershell
# ライブ (念のため):
Get-ScheduledTask | ForEach-Object {
  [PSCustomObject]@{
    TaskName = $_.TaskName
    Path     = $_.TaskPath
    Author   = $_.Author
    RunAs    = $_.Principal.UserId
    Command  = ($_.Actions | Select-Object -First 1).Execute
    Args     = ($_.Actions | Select-Object -First 1).Arguments
  }
} | Export-Csv scheduled_tasks.csv -NoTypeInformation

# オフライン: Tasks フォルダ配下の XML を直接パース
Get-ChildItem <MOUNTED_WIN_DIR>\System32\Tasks -Recurse -File |
  ForEach-Object { [xml]$x = Get-Content $_.FullName;
    [PSCustomObject]@{ Name=$_.Name; Exec=$x.Task.Actions.Exec.Command; Args=$x.Task.Actions.Exec.Arguments } }

WMI Event Subscription

WMI 永続化は再起動を跨ぐ、ファイルレス、Autoruns では見落とされやすい。オフラインでは OBJECTS.DATA を解析する。

powershell
# ライブ確認 (要ローカル admin)
Get-CimInstance -Namespace root\subscription -ClassName __EventFilter
Get-CimInstance -Namespace root\subscription -ClassName __EventConsumer
Get-CimInstance -Namespace root\subscription -ClassName __FilterToConsumerBinding

# オフライン (PyWMIPersistenceFinder 等): OBJECTS.DATA を直接パース
# %SystemRoot%\System32\wbem\Repository\OBJECTS.DATA
warning Autoruns は WMI Event Subscription の 一部 しか見せないケースがある。WMI 永続化は必ず個別確認する。
要点 / key takeaways
  • 永続化確認はオフラインで。ライブは補助。
  • WMI / COM Hijacking / IFEO は Autoruns 単体では取りこぼしうる。
  • 「見つけた」で止めず、同じ手口の全展開先を潰す (単一ホストで終わらない前提)。

08アンチフォレンジックAnti-Forensics Awareness

攻撃側が痕跡を消しにかかる前提で防御側は動く。何が消され、何が残るかを知っておくことが早期発見につながる。

タイムスタンプ改ざん (Timestomping)

SI (StandardInformation) は書換可能。SetFileTime API や timestomp 系ツールで秒未満まで自然な値に改ざんできる。ただし FN (FileName) はカーネルが更新するため一般ユーザ空間からは触れない。

  • 兆候: SI と FN の乖離、SI の秒以下が .0000000 (改ざんツールが精度を落とすことがある)、フォルダの MFT エントリの Modified が子ファイルより古い。
  • 検出: MFTECmd 出力で SI0/FN0 列を比較。差分ソートで一気にリスト化できる。

ログ改ざん / 消去

  • evtx 全消去: wevtutil cl <channel>。この操作自体が Security 1102 として記録される。
  • 個別イベントの選択削除: 通常は不可 (evtx はレコード単位で書き換えると破損)。DanderSpritz / 3gstudent 系の PoC は存在するが、破損検知で概ね見抜ける。
  • Sysmon の一時停止: Service 停止は System 7036 に記録される。設定書き換えは Sysmon 16。

ファイル削除・上書き

  • 単純削除: MFT エントリは残る (再利用されるまで)。$UsnJrnl に削除記録が残る。
  • SDelete 系: セクタ上書き。MFT エントリ・$UsnJrnl・親フォルダのタイムスタンプに痕跡は残る。
  • USN Journal のクリア: fsutil usn deletejournal。この操作の実行痕跡が Prefetch / AmCache / ShimCache に残ることがある。

ファイルレス・インメモリ手法

  • ディスクに落ちないため MFT / Prefetch / AmCache には残らない。
  • 残るのは PowerShell 4104 (ScriptBlock), Sysmon 1/7/8/10, WMI event subscription, メモリダンプ。
  • 再起動で消失する痕跡が多いため、疑わしい端末の即時電源断は原則避ける。
tip アンチフォレンジックが疑われる時こそ、複数の独立した情報源を突き合わせる (ディスク・メモリ・ネットワーク・別ホスト側のログ)。単一ソースで結論しない。
要点 / key takeaways
  • SI は書換可、FN は不変。この非対称性が timestomping 検出の梃子。
  • ログ全消去は静かではない。1102 / 7036 / Sysmon 16 で必ず何か残る。
  • インメモリは強いが「再起動しない」IR 判断が成立の前提。
