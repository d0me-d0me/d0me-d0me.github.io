Linux Hardening

A defender's field checklist for Linux + LAMP/LEMP hosts under active adversarial pressure — from baseline audit through kernel/PAM hardening, network isolation, auditing, and continuous monitoring (Hardening Project / Micro Hardening style).状態調査 → 退避確保 → カーネル・システム基本設定 → サービス強化 → ネットワーク隔離 → 監査・監視までの、Linux + Web スタック向け防御チェックリスト。

## Hardening Flow 強化の順序 assess → backup → baseline → harden → isolate → monitor
01 Assess 状態調査 02 Backup 退避確保 03 Baseline カーネル基本 04 Harden Services サービス強化 05 Isolate Network ネットワーク隔離 06 Monitor & Audit 監視・監査
Order matters / 実施順の含意

Backup を最初に取らずに hardening を始めると、権限変更ミスや設定破壊で復旧手段を失う。Kernel/sysctl の反映は再起動不要だが、systemctl mask による削減や PAM 変更は必ず別セッションを保持したまま反映する。Network isolation はサービス単位で段階的に締める(すべての UFW ルールを一括で入れると疎通確認が難しくなる)。

01Assess & Audit状態調査

侵害仮定で入った直後にまず把握するのは、動いているサービス、存在するアカウント、緩すぎる権限。防御側のスタート地点であると同時に、攻撃者にとっての初動情報でもある。

Service and account enumeration

bash
# 稼働サービス一覧
service --status-all
systemctl list-units --type=service --state=running

# ユーザ / グループ
cat /etc/passwd
cat /etc/group

# 昇格対象になり得る sudoers 設定
sudo cat /etc/sudoers
sudo ls /etc/sudoers.d/
sudo grep -RIn 'NOPASSWD\|ALL=(ALL)' /etc/sudoers /etc/sudoers.d/

# cron エントリ (root 含む)
crontab -l -u root
ls -la /etc/cron.*
cat /etc/crontab

Permission audit

bash
# world-writable なディレクトリ (真っ先に潰す)
find / -type d -perm -0002 -ls 2>/dev/null

# mode 777 のディレクトリ (旧手順互換の書き方)
find / -type d -perm 777 -ls 2>/dev/null

# SUID / SGID ファイル (privesc の主戦場)
find / -perm -4000 -type f 2>/dev/null
find / -perm -2000 -type f 2>/dev/null

# root 所有だがユーザが書ける不整合
find /var/www -user root -perm -o+w 2>/dev/null

# 空パスワードのアカウント検出
sudo awk -F: '($2 == "" || $2 == "!" ) {print $1}' /etc/shadow

Corrective examples

EC-CUBE 系アプリの標準的な締め方(ディレクトリ 705 / ファイル 604 を出発点に、書込みが必須な cache/log/plugin/template を個別に緩めるパターン)。

bash

sudo find /var/www/html/shop -type f -exec chmod 604 {} \;

# 書き込みが必要な領域を個別に開ける
sudo chmod 707 /var/www/html/shop
sudo chmod 707 /var/www/html/shop/app
sudo chmod 707 /var/www/html/shop/app/Plugin/
sudo chmod 707 /var/www/html/shop/app/template/
sudo chmod 707 /var/www/html/shop/vendor/
sudo chmod 606 /var/www/html/shop/composer.json
sudo chmod 606 /var/www/html/shop/composer.lock

# template / cache / log / config は 750-755 に絞る
chmod -R 755 /root/www/html/shop/app/template
chmod -R 755 /root/www/html/shop/app/Plugin
chmod -R 750 /root/www/html/shop/app/config
chmod -R 750 /root/www/html/shop/app/cache
chmod -R 750 /root/www/html/shop/app/log
chmod -R 755 /root/www/html/shop/html/template
chmod -R 755 /root/www/html/shop/html/plugin
chmod -R 755 /root/www/html/shop/html/user_data
chmod -R 755 /root/www/html/shop/html/upload
Attacker's view状態調査の直後にサービスを止める・パーミッションを落とすと、既に潜んでいる攻撃者のセッションを一時的に維持したまま「見た目だけ堅い」状態にしてしまう。ss -tnp / last / w で確立中の接続とログインを先に把握しておく。

02Backup & Restore退避と復旧

Hardening 手順の最初に置くべき工程。設定を書き換えた瞬間にサービスが止まって、切り戻し手段が無いという事故を防ぐ。タイムスタンプ付きで撮っておき、時系列で状態を比較可能にする。

Timestamped snapshots

対象バックアップ復元
/etc cp -pr /etc /backup/etc-$(date +%H-%M) rm -rf /etc && cp -pr /backup/etc-* /etc
/var cp -pr /var /backup/var-$(date +%H-%M) rm -rf /var && cp -pr /backup/var-* /var
/var/www cp -pr /var/www /backup/www-$(date +%H-%M) rm -rf /var/www && cp -pr /backup/www-* /var/www
MySQL / MariaDB mysqldump -u root -p -x --all-databases > /backup/mysql-$(date +%H-%M) mysql -u root -p < /backup/mysql-*
PostgreSQL pg_dumpall -U admin > /backup/postgresql-$(date +%H-%M) psql < /backup/postgresql-*

Quick working copy (/b pattern)

作業卓上の /b に「今この瞬間の断面」を置いておくと、設定破壊時の即時ロールバックが 1 コマンドで済む。

bash
sudo mkdir /b
cd /b
sudo cp -r /var/www/html/shop /b/
sudo cp -r /var/www/nginx/blog /b/
sudo cp /etc/apache2/apache2.conf /b/
sudo cp /etc/nginx/nginx.conf /b/

# ロールバック
sudo cp -r /b/shop /var/www/html/shop
sudo cp -r /b/blog /var/www/nginx/blog
sudo cp    /b/apache2.conf /etc/apache2/apache2.conf
sudo cp    /b/nginx.conf   /etc/nginx/nginx.conf
Pitfall / 落とし穴cp -pr は元の owner/group/permission/timestamp を保つ。-p を落とすと権限が root 所有で復元されてサービス起動不能になる。ダンプ系(mysqldump)は結果ファイルに認証情報が残るため、バックアップ先自体のパーミッションを chmod 600 に絞る。

03Kernel & System Baselineカーネル・システム基本

Kernel パラメータ、ulimit、モジュール制限、サービス最小化、自動パッチはサービス個別の hardening より先に「土台」として揃える。再起動なしで反映できるものが多く、影響範囲も予測しやすい。

sysctl — kernel & network

conf
# /etc/sysctl.d/99-hardening.conf

# ---- Network ----
# SYN flood 対策
net.ipv4.tcp_syncookies = 1
# ICMP redirect (偽装経路)
net.ipv4.conf.all.accept_redirects = 0
net.ipv4.conf.all.send_redirects = 0
net.ipv6.conf.all.accept_redirects = 0
# Source routing (経路表迂回)
net.ipv4.conf.all.accept_source_route = 0
net.ipv6.conf.all.accept_source_route = 0
# Reverse path filter (spoofing 抑止)
net.ipv4.conf.all.rp_filter = 1
net.ipv4.conf.default.rp_filter = 1
# ICMP ブロードキャスト (smurf 反射)
net.ipv4.icmp_echo_ignore_broadcasts = 1
net.ipv4.icmp_ignore_bogus_error_responses = 1
# 不正パケットの kern.log 記録
net.ipv4.conf.all.log_martians = 1

# ---- Kernel ----
# ASLR 完全有効
kernel.randomize_va_space = 2
# 一般ユーザからの dmesg / kernel pointer 隠蔽
kernel.dmesg_restrict = 1
kernel.kptr_restrict = 2
# ptrace の範囲を子プロセスに限定 (root は例外)
kernel.yama.ptrace_scope = 1
# SUID プログラムの core dump 抑止
fs.suid_dumpable = 0
# hardlink / symlink 保護 (/tmp race condition 対策)
fs.protected_hardlinks = 1
fs.protected_symlinks = 1
fs.protected_fifos = 2
fs.protected_regular = 2
bash
# 反映 (即時、再起動不要)
sudo sysctl -p /etc/sysctl.d/99-hardening.conf

# 適用値の確認
sudo sysctl -a 2>/dev/null | grep -E 'tcp_syncookies|rp_filter|randomize_va_space|ptrace_scope'

Resource limits — limits.conf

conf
# /etc/security/limits.d/99-hardening.conf

# core dump を全ユーザ無効化 (機密漏洩防止)
*  hard  core   0
*  soft  core   0

# fork bomb 対策
*  soft  nproc  1024
*  hard  nproc  4096

# 開けるファイル数 (実サービスの実測値に合わせて調整)
*  soft  nofile 4096
*  hard  nofile 8192

Kernel module blacklist

使わないファイルシステムやプロトコルをカーネルに読み込ませない。攻撃サーフェス削減と過去 CVE の予防的無効化を兼ねる。

conf
# /etc/modprobe.d/blacklist-hardening.conf

# 未使用ファイルシステム
install cramfs    /bin/false
install freevxfs  /bin/false
install jffs2     /bin/false
install hfs       /bin/false
install hfsplus   /bin/false
install udf       /bin/false

# 稀少 or 未使用の L4 プロトコル (CVE 頻出領域)
install dccp /bin/false
install sctp /bin/false
install rds  /bin/false
install tipc /bin/false

# 物理経路を締める必要があれば
install usb-storage /bin/false
install firewire-core /bin/false

Service minimization

bash
# 有効化されているサービス一覧
systemctl list-unit-files --type=service --state=enabled

# LISTEN しているポートと紐付くプロセスの棚卸し
sudo ss -tulnp

# 無効化 + マスク (誤って再有効化されないように)
sudo systemctl disable --now <SERVICE>
sudo systemctl mask <SERVICE>

# 現場でよく落とす候補 (要件を確認してから)
#   bluetooth / cups / avahi-daemon / snapd / ModemManager / rpcbind

Automatic security updates

bash
# --- Debian / Ubuntu ---
sudo apt install -y unattended-upgrades apt-listchanges
sudo dpkg-reconfigure -plow unattended-upgrades

# /etc/apt/apt.conf.d/50unattended-upgrades を編集
#   Allowed-Origins に security を残す
#   Automatic-Reboot / Automatic-Reboot-Time は運用ポリシー次第

sudo unattended-upgrades --dry-run --debug

# --- RHEL / AlmaLinux / Rocky ---
sudo dnf install -y dnf-automatic
sudo systemctl enable --now dnf-automatic.timer
# /etc/dnf/automatic.conf を編集 (apply_updates = yes)
Countermeasures
  • sysctl は永続化ファイルで管理: sysctl -w の即時変更は再起動で消える。/etc/sysctl.d/ にファイルを置いて sysctl -p で反映するのが恒久策。
  • module blacklist は install ... /bin/false で: 単なる blacklist ディレクティブでは依存モジュールの automatic load を止められない。
  • 自動 reboot は業務時間外に: Unattended-Upgrade::Automatic-Reboot-Time で明示。

04SSH & PAMSSH と認証強化

root 直接ログイン抑止・空パスワード禁止・試行回数制限が SSH の最小三点セット。MaxAuthTries はホスト内のカウントなので、送信元 IP の連続試行そのものは止められない。PAM (pam_faillock / pam_pwquality) と併用して初めて成立する。

User management

bash
# パスワード変更
sudo passwd <USERNAME>

# ホームディレクトリの他ユーザ読取制限
chmod 701 /home/<USERNAME>

# ロック / 削除
sudo usermod -L <USERNAME>
sudo userdel -r <USERNAME>

# システムアカウントで shell を持つ不整合を検出
awk -F: '$3 < 1000 && $7 !~ /(nologin|false)$/ {print}' /etc/passwd

sshd_config

設定ファイル: /etc/ssh/sshd_config。書き換え後は sshd -t で syntax チェックしてから systemctl reload sshd。

conf
# root 直接ログインの抑止 (公開鍵のみ許可なら prohibit-password)
PermitRootLogin no

# 空パスワードでのログイン禁止
PermitEmptyPasswords no

# 認証試行回数の上限 (総当たり緩和)
MaxAuthTries 3

# パスワード認証を止めて公開鍵のみへ (可能なら)
PasswordAuthentication no
PubkeyAuthentication yes
ChallengeResponseAuthentication no

# プロトコル / 弱いアルゴリズムの明示除外
Protocol 2
KexAlgorithms curve25519-sha256,curve25519-sha256@libssh.org
Ciphers chacha20-poly1305@openssh.com,aes256-gcm@openssh.com
MACs hmac-sha2-512-etm@openssh.com,hmac-sha2-256-etm@openssh.com

# アイドル切断
ClientAliveInterval 300
ClientAliveCountMax 2

# ログイン許可を明示
AllowUsers <USERNAME>

PAM — account lockout (pam_faillock)

SSH の MaxAuthTries はセッション内カウント。IP を変えたり再接続すればリセットされる。pam_faillock は tally file にユーザ単位で失敗を蓄積するので、複数セッションにまたがってロックが効く。

conf
# /etc/security/faillock.conf (systemd 系)

# ロックまでの失敗回数
deny = 5
# ロック解除までの秒数
unlock_time = 900
# root にも適用
even_deny_root
root_unlock_time = 60
# 失敗カウントの集計時間窓
fail_interval = 900
bash
# /etc/pam.d/sshd と /etc/pam.d/login に組み込み (RHEL 系は authselect 経由が推奨)
# 例: /etc/pam.d/common-auth (Debian/Ubuntu) 冒頭付近に
auth    required    pam_faillock.so preauth
auth    [success=1 default=bad] pam_unix.so
auth    [default=die] pam_faillock.so authfail
auth    sufficient  pam_faillock.so authsucc

# 現在ロックされているユーザの確認
sudo faillock

# 手動でロック解除
sudo faillock --user <USERNAME> --reset

PAM — password quality (pam_pwquality)

conf
# /etc/security/pwquality.conf

minlen = 14
minclass = 3          # 大文字 / 小文字 / 数字 / 記号 の 3 種以上
maxrepeat = 3         # 同一文字の連続
maxsequence = 3       # 単純な連続 (abcd, 1234)
dictcheck = 1         # 辞書チェック
enforce_for_root = 1
retry = 3

login.defs

conf
# /etc/login.defs

PASS_MAX_DAYS   90        # パスワード有効期限
PASS_MIN_DAYS   7         # 連続変更間隔
PASS_WARN_AGE   14        # 期限警告開始
UMASK           027       # デフォルト umask
ENCRYPT_METHOD  SHA512    # /etc/shadow のハッシュ方式
FAILLOG_ENAB    yes
Countermeasures
  • 別セッション保持: systemctl reload sshd や PAM 変更の前に必ず既存 SSH セッションを1本残しておく。
  • fail2ban / sshguard の併用: pam_faillock はユーザ単位、fail2ban は IP 単位で、両者は補完関係。
  • ポート変更は防御ではない: 22 → 高位ポートは security-through-obscurity。副次効果としてノイズは減るが、鍵認証必須の代替にはならない。
  • root のロックアウトに注意: even_deny_root を入れる場合は復旧手段(コンソールアクセス、rescue mode)を確保しておく。

IP-based access control

基本形は allow / deny の逐次評価。location ブロックに閉じて評価順を明示する。

conf
# /etc/nginx/nginx.conf
http {
    server {
        location /phpmyadmin {
            allow <CRAWLER_IP>;
            deny  all;
        }
    }
}

Allowlist via geo module

複数拠点の IP を集約管理する場合は geo で bool 化し、if で分岐すると upstream の受け付け先も切り替えやすい。

conf
# /etc/nginx/conf.d/sample_app.conf
upstream sample_app {
    server unix:/tmp/unicorn.sock;
}

geo $allow_ip {
    default 0;
    <OFFICE_IP>    1;  # 自社 IP
    <BRANCH_A_IP> 1;  # 支社 A
    <BRANCH_B_IP> 1;  # 支社 B
}

server {
    location ~ ^/admin/ {
        if ($allow_ip) {
            proxy_pass http://sample_app;
        }
        deny all;
    }
}

Access log with request body

侵害調査時にリクエストボディが取れないと、POST 経由の攻撃 payload が復元できない。標準の combined 相当に $request_body を追加する。

conf
# /etc/nginx/nginx.conf — 元の access_log 行はコメントアウト
log_format  request_body '$remote_addr - $remote_user [$time_local] "$request" '
                          '$status $body_bytes_sent "$http_referer" '
                          '"$http_user_agent" "$http_x_forwarded_for" "$request_body"';
access_log  /var/log/nginx/access.log  request_body;
Note$request_body はパスワード等の秘匿値もそのまま平文で残る。ログの取扱区分・保持期間・アクセス権を先に決めておく。

  • アプリ管理画面へのパス制限: /wp-admin/ や /admin/ は Web 層(Nginx/Apache)で IP allowlist を掛ける。

07Network Controls (UFW)ネットワーク制御

UFW で外向きは default deny incoming / allow outgoing、内向けは「必要なポート・必要な送信元」だけを段階的に開ける。段階を飛ばして一括で締めると疎通確認が困難になる。

Baseline

bash
# 状態確認
sudo ufw status
sudo ufw status verbose

# デフォルトポリシー
sudo ufw default deny  incoming
sudo ufw default allow outgoing

# SSH の到達性を先に確保 (self-lockout 防止)
sudo ufw allow 22/tcp

# 例: クローラ / 特定サービスのみ許可
sudo ufw allow from <CRAWLER_IP>
sudo ufw allow 80/tcp
sudo ufw allow 53
sudo ufw allow 25
sudo ufw allow 81
sudo ufw allow from <CRAWLER_IP> to any port 143

sudo ufw enable
sudo ufw reload

Databases to localhost

bash
# MariaDB / MySQL — 3306 を loopback 限定
sudo ufw deny 3306
sudo ufw allow from 127.0.0.1 to any port 3306
# /etc/mysql/mariadb.conf.d/50-server.cnf
#   bind-address = 127.0.0.1

# PostgreSQL — 5432 を loopback 限定
sudo ufw deny 5432
sudo ufw allow from 127.0.0.1 to any port 5432
# /etc/postgresql/12/main/postgresql.conf
#   listen_addresses = 'localhost'

Tomcat / GroupSession

bash
# 8080 は外に出さない
sudo ufw deny 8080
sudo ufw allow from 127.0.0.1 to any port 8080

# http://<HOST>:8080/host-manager/html の管理画面は特に要確認

Attack source block & watch

bash
# 攻撃元 IP を deny
sudo ufw deny from <ATTACKER_IP>

# 不審な既存ルールの削除 (deny 80 が刺さっていた例)
sudo ufw delete deny 80

# ルール改変の常時監視
watch -n 5 sudo ufw status
Order mattersUFW は登録順に評価される。ufw enable の 前に SSH の allow 22(または実運用ポート)が入っていないと自分自身を締め出す。ufw enable は SSH セッション経由で叩かない、あるいは事前に別セッションを保持する。

08Infrastructure Services周辺サービス

DNS / SMTP / FTP / cache は「listen 範囲」と「送信元制限」が肝。デフォルト設定のまま公開すると DDoS の踏み台や情報漏洩の起点になる。

Unbound (DNS)

conf
# /etc/unbound/unbound.conf
control-enable: no

# 参照許可はセグメント全体ではなくホスト単位に絞る
access-control: <INTERNAL_IP>/32  allow
access-control: <PROXY_IP>/32     allow
access-control: <CRAWLER_IP>/32   allow

# ローカルゾーン例
local-data: "www.example.local. 10 IN A <HOST_IP>"
local-data: "www.example.local. 10 IN NS <HOST_IP>"
local-data: "internal.example.local. 10 IN A <HOST_IP>"
local-data: "mail.example.local. 10 IN A <HOST_IP>"
local-data: "example.local. IN MX 10 mail.example.local."
local-data-ptr: "<HOST_IP> 10 www.example.local."
bash
sudo systemctl status  unbound
sudo systemctl start   unbound
sudo systemctl stop    unbound
sudo systemctl restart unbound

Postfix (SMTP)

conf
# /etc/postfix/main.cf — オープンリレー化を防ぐ
mynetworks = 127.0.0.0/8, <INTERNAL_IP>/32

# SMTP AUTH 前提の運用ならこれも
smtpd_relay_restrictions = permit_mynetworks, permit_sasl_authenticated, reject_unauth_destination
smtpd_helo_required = yes

Proftpd (FTP)

bash
# 設定 / ログ
/etc/proftpd/proftpd.conf
/var/log/proftpd/

# 匿名ログイン禁止 / ルート禁止 / chroot 化を conf 側で確認
grep -E 'Anonymous|RootLogin|DefaultRoot' /etc/proftpd/proftpd.conf

memcached

bash
# /etc/memcached.conf — 全公開を loopback 限定に
-l 127.0.0.1

# 反射増幅攻撃の踏み台にならないよう UDP も落とす
-U 0

Proxy client configuration

bash
# 内部からの外向き通信は明示的にプロキシ経由
export http_proxy=http://<PROXY_IP>:3128
export https_proxy=http://<PROXY_IP>:3128
echo $http_proxy
echo $https_proxy
sudo apt update

# apt 用に永続化する場合
# /etc/apt/apt.conf.d/95proxy
#   Acquire::http::Proxy  "http://<PROXY_IP>:3128";
#   Acquire::https::Proxy "http://<PROXY_IP>:3128";

09Monitoring, Auditing & Cleanup監視・監査・後始末

Hardening は一発反映では終わらず、差分の検出と改竄検知を継続する。watch による軽量監視、auditd による syscall 監査、AIDE によるファイル完全性検証、公開ディレクトリの残置ファイル一掃が実務の中心。

Change detection with watch

bash
# 直近 5 分以内に更新された web 配下のファイル (cache 除外)
watch -n 2 'find /var/www/html/ -type f -mmin -5 | grep -v cache'

# 大量書換の兆候: /tmp や /dev/shm への WebShell 展開監視
watch -n 5 'find /tmp /dev/shm -type f -mmin -5 -ls'

Service status

bash
watch -n 5 '
  echo "Unbound: "; sudo systemctl status unbound       | grep Active:
  echo "Apache2: "; sudo systemctl status apache2       | grep Active:
  echo "Nginx:   "; sudo systemctl status nginx         | grep Active:
  echo "Tomcat8: "; sudo systemctl status tomcat8       | grep Active:
  echo "MariaDB: "; sudo systemctl status mariadb       | grep Active:
  echo "Postfix: "; sudo systemctl status postfix       | grep Active:
'

UFW rule watch

bash
watch -n 5 sudo ufw status

# 不審なルール(例: deny 80 が挿入されていた)を削除
sudo ufw delete deny 80

auditd — syscall & identity audit

認証情報の変更、権限変更 syscall、SSH 設定の書換え、sudoers の改変を追う。-e 2 でルール自体をイミュータブル化すると、再起動まで rule set を凍結できる。

bash
sudo apt install -y auditd
sudo systemctl enable --now auditd

# 実行中のルール確認
sudo auditctl -l
sudo auditctl -s
conf
# /etc/audit/rules.d/hardening.rules

# 認証情報の変更を追跡
-w /etc/passwd    -p wa -k identity
-w /etc/group     -p wa -k identity
-w /etc/shadow    -p wa -k identity
-w /etc/gshadow   -p wa -k identity
-w /etc/sudoers   -p wa -k identity
-w /etc/sudoers.d/ -p wa -k identity

# SSH 設定の変更
-w /etc/ssh/sshd_config -p wa -k sshd

# 権限変更 syscall (人ユーザ発)
-a always,exit -F arch=b64 -S chmod,fchmod,fchmodat -F auid>=1000 -F auid!=4294967295 -k perm_mod
-a always,exit -F arch=b64 -S chown,fchown,fchownat,lchown -F auid>=1000 -F auid!=4294967295 -k perm_mod

# 特権実行の追跡
-a always,exit -F arch=b64 -S execve -F euid=0 -F auid>=1000 -F auid!=4294967295 -k privileged

# ルール自体の変更禁止 (最下行)
-e 2
bash
# ルール反映
sudo augenrules --load
sudo systemctl restart auditd

# ログ検索
sudo ausearch -k identity -ts today
sudo ausearch -k sshd     -ts today
sudo aureport --summary
sudo aureport --auth --summary

AIDE — file integrity

bash
sudo apt install -y aide

# 初期 DB 生成 (時間かかる)
sudo aideinit
sudo mv /var/lib/aide/aide.db.new /var/lib/aide/aide.db

# 変更検知
sudo aide --check

# cron で日次実行
echo '0 4 * * * root /usr/bin/aide --check | mail -s "AIDE report" root' \
  | sudo tee /etc/cron.d/aide

# 設定変更後は DB を作り直す (合法な差分でノイズが増えないように)
sudo aide --update
sudo mv /var/lib/aide/aide.db.new /var/lib/aide/aide.db

Log protection (chattr +a)

bash
# 追記のみ許可 (改竄防止)
sudo chattr +a /var/log/auth.log
sudo chattr +a /var/log/syslog

# 属性確認
lsattr /var/log/auth.log

# 一時解除 (logrotate と競合するので rotate 前後で外す/戻す)
sudo chattr -a /var/log/auth.log
Notechattr +a は root であっても上書き・削除を拒否する。logrotate の prerotate / postrotate で chattr -a / chattr +a を組み込むか、rsyslog で forward 先を別ホストにするのが実務的。

Removing installer / info leaks

公開領域に残ったインストーラや phpinfo は、再インストール実行や環境情報漏洩に直結する。存在確認 → 削除を素直に行う。

bash
# 存在確認
cd /var/www
find -type f -name "install.php"
find -type f -name "phpinfo.php"
find -type f -name "readme.html"

# 一括削除 (EC-CUBE / WordPress / phpMyAdmin / 独立ディレクトリの残骸)
rm ./html/shop/html/install.php
rm ./nginx/blog/wp-admin/install.php
rm ./html/shop/eccube_install.sh
rm ./nginx/blog/readme.html
rm ./html/shop/vendor/swiftmailer/swiftmailer/tests/_samples/files/textfile.zip
rm ./nginx/phpinfo.php
rm ./nginx/phpmyadmin/phpinfo.php
rm ./html/phpinfo.php

Cache invalidation

bash
# EC-CUBE: テンプレート / 設定変更後の反映
cd /var/www/html/shop
sudo php app/console cache:clear

# WordPress: object-cache / transient のクリア
wp --path=/var/www/nginx/blog cache flush
wp --path=/var/www/nginx/blog transient delete --all
Key takeaways
  • Hardening の順序は Assess → Backup → Baseline (kernel) → Harden → Isolate → Monitor/Audit。Backup を最初に取らずに始めない。
  • SSH / UFW の締め直しは 別セッションを保持したまま反映する(自己締め出し防止)。
  • SSH の MaxAuthTries と PAM の pam_faillock は両方入れて意味を持つ。前者はセッション内、後者は永続 tally。
  • DB は bind-address と UFW の両方で localhost に閉じる(片方だけでは設定漏れ時に露出する)。
  • auditd の -e 2 と AIDE の DB は、それぞれ「rule set の凍結」「baseline の凍結」で改竄検知の基準を作る。
  • 公開ディレクトリの install.php / readme.html / phpinfo.php は必ず削除。再インストール攻撃と環境情報漏洩の主要経路。
