伍Web

Web 実装で頻出する弱点を、発見の観点と定石で整理する。列挙 → 仮説 → 検証の順で、各項目に検知/対策を併記する。 前提: 認可を得た検証対象。ペイロードは一般化した例に留める。

01列挙recon

攻撃面はまず「見えている入口」を数える。ディレクトリ・パラメータ・技術スタックを列挙し、フレームワーク特有の弱点へ当たりを付ける。

bash
# コンテンツ探索
ffuf -u https://$TARGET/FUZZ -w /usr/share/seclists/Discovery/Web-Content/raft-medium-directories.txt
gobuster dir -u https://$TARGET -w /usr/share/wordlists/dirb/common.txt
防御 / countermeasures
  • 404 の一貫化、レート制限、WAF で総当たり列挙を抑止。
  • 技術スタックを露出するヘッダ・エラーページを削減。

02認証・セッションauth

弱点は「資格情報の推測可能性」と「セッションの検証不備」に集約される。JWT は署名検証(alg=none・弱鍵)、Cookie は属性(HttpOnly/SameSite/Secure)を確認する。

bash
# ログイン総当たり（許可された範囲で）
hydra -L users.txt -P /usr/share/wordlists/rockyou.txt \
  $TARGET https-post-form \
  "/login:user=^USER^&pass=^PASS^:F=invalid"

# Flask セッション（弱い SECRET_KEY の検証）
flask-unsign --unsign --cookie '<COOKIE>' --wordlist /usr/share/wordlists/rockyou.txt
防御 / countermeasures
  • 強固なランダム鍵、署名アルゴリズム固定、MFA、ロックアウト/レート制限。
  • Cookie に HttpOnly・SameSite・Secure を付与し、失敗ログを監視。

03デシリアライズdeserialization

信頼できない入力をオブジェクト復元に渡すと、ガジェットチェーン経由で任意コード実行に至る。言語・ライブラリで成立条件が異なるため、まずシリアライズ形式(Java/PHP/.NET/Python pickle 等)を同定する。

  • 同定 — Base64 のマジックバイト(Java rO0、.NET AAEAAAD 等)や Cookie/パラメータ形状から推定。
  • 検証 — 公開ガジェット生成器(ysoserial 系)で既知チェーンの成立可否を確認。
注意ガジェットチェーンの実装詳細は載せない。成立条件の同定と、対策側の観点に留める。
防御 / countermeasures
  • 信頼境界を越えるデータをネイティブ・デシリアライズに渡さない(JSON+スキーマ検証へ)。
  • 許可型リスト(look-ahead deserialization)、ライブラリ更新、実行監視。

04SSTItemplate injection

ユーザ入力がテンプレートエンジンに評価されると、式評価から RCE に発展し得る。まずポリグロットで反応を見てエンジンを同定する。

text
# 反応検知用ポリグロット（評価されるか）
${7*7}  {{7*7}}  <%= 7*7 %>  #{7*7}
# 49 が返ればテンプレート評価あり → エンジン同定へ
防御 / countermeasures
  • ユーザ入力をテンプレートに連結しない。ロジックレス/サンドボックス化エンジンを使う。
  • 出力エスケープと、テンプレート評価の監査ログ。

05アップロードfile upload

検証不備は拡張子・Content-Type・マジックバイト・保存先の実行可否の組合せで生じる。狙いは「実行可能な場所に、実行される形式で置けるか」。

検証典型的な不備
拡張子ブラックリスト漏れ(代替拡張子・大文字・二重拡張子)
Content-Typeクライアント申告を信頼
内容検査マジックバイト偽装を未検出
保存先Web ルート配下で実行可能
防御 / countermeasures
  • 許可型リスト検証 + サーバ側でのマジックバイト確認、ランダム名で保存。
  • 保存先を Web ルート外・実行不可に。画像は再エンコードして無害化。
要点 / key takeaways
  • Web は「入口を数える → 技術スタックを同定 → 定石で検証」。列挙が精度を決める。
  • 認証は鍵と署名検証、セッションは属性。デシリアライズ/SSTI はまず形式・エンジンの同定が先。
  • アップロードは単一検証では守れない。拡張子・内容・保存先の実行可否を多層で断つ。
