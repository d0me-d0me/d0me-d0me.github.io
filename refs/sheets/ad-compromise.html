Full AD attack chain from initial foothold to forest compromise, structured by MITRE ATT&CK tactics from Discovery through Impact.初期侵入から DA 取得・Forest 侵害までの完全な攻撃連鎖を、MITRE ATT&CK の Tactic 軸 (Discovery → Impact) で整理。

01Discovery偵察・列挙

ドメイン参加ホスト取得直後から、認証情報なし段階と認証情報あり段階の双方で、ドメイン構造・特権グループ・委任設定・BloodHound 経路を可視化する。ここでの取得情報は後続の全 Tactic の判断材料になる。

重要 Cypher クエリ

cypher
// 自分から DA への最短経路
MATCH p=shortestPath((u:User {name:"USER@CONTOSO.LOCAL"})-[*1..]->(g:Group {name:"DOMAIN ADMINS@CONTOSO.LOCAL"})) RETURN p

// Kerberoastable + 高権限グループメンバー
MATCH (u:User {hasspn:true})-[:MemberOf*1..]->(g:Group)
WHERE g.name CONTAINS "ADMIN" RETURN u.name, g.name

// ASREProastable
MATCH (u:User {dontreqpreauth:true}) RETURN u.name

// Unconstrained Delegation ホスト
MATCH (c:Computer {unconstraineddelegation:true}) RETURN c.name

// RBCD 候補 (Computer への GenericWrite 等)
MATCH (u)-[:GenericAll|GenericWrite|WriteOwner|WriteDacl]->(c:Computer)
RETURN u.name, c.name

// LAPS 読取権限保有者
MATCH (u)-[:ReadLAPSPassword]->(c:Computer) RETURN u.name, c.name

// 自分が書込可能なオブジェクト全て
MATCH (u:User {name:"USER@CONTOSO.LOCAL"})-[r]->(n)
WHERE type(r) IN ['GenericAll','GenericWrite','WriteOwner','WriteDacl','AddMember','ForceChangePassword']
RETURN u.name, type(r), n.name

// アクティブセッション (LoggedOn 収集時)
MATCH p=(c:Computer)-[:HasSession]->(u:User {name:"DA_USER@CONTOSO.LOCAL"}) RETURN p
防御 / countermeasures
  • Event ID 4662 (DirectoryServiceAccess) と 5136 (Directory Service Changes) の異常な LDAP 検索を SIEM で集約、大量 SPN query は BloodHound 収集の signature
  • ハニーポットアカウント (SPN 付き / 実利用ゼロ) を配置、Kerberoast 試行検知に活用
  • Sysmon Event 11 (FileCreated) で SharpHound zip / BloodHound json の生成を検知
  • DC の LDAP query rate baseline を確立、閾値超過をアラート
要点 / key takeaways
  • Phase 1 でロックアウトポリシー・パスワードポリシーを必ず確認 (spray 設計に必須)
  • BloodHound は DCOnly から開始し、Session / LoggedOn は権限取得後に段階的に追加
  • 認証情報なし段階でも RID brute (--rid-brute) / Kerbrute userenum で有効ユーザ列挙は可能
  • LAPS 有効ホスト・Unconstrained Delegation ホスト・Kerberoast 候補は同一 LDAP セッション内で纏めて取得

02Credential Access認証情報取得

ドメイン内で有効な認証情報を得るための一次経路。Kerberoast / ASREProast は Discovery で列挙した候補に対して実行、パスワードスプレーはパスワードポリシー確認後に慎重に、NTDS ダンプは DCSync 権限取得後の最終収穫。

Kerberoasting (SPN 保有ユーザ)

SPN 登録アカウントの TGS はユーザの鍵で暗号化されるため、TGS を取得しオフラインクラック可能。

bash

  -request contoso.local/user:'<PASSWORD>'

# 既存 Kerberos チケットで
GetUserSPNs.py -dc-ip <DC_IP> -k -no-pass -request contoso.local/user
powershell

.\Rubeus.exe kerberoast /user:svc_sql /nowrap

.\Rubeus.exe kerberoast /tgtdeleg /nowrap             # TGT 委任、最低検知
.\Rubeus.exe kerberoast /spn:MSSQLSvc/sql01.contoso.local:1433 /nowrap

  • --continue-on-success で正解を得ても全ユーザ試行を継続
  • 検知回避のため jitter (--jitter 60) + 並列度抑制 (-t 1)

防御 / countermeasures
  • Fine-Grained Password Policy (PSO) をサービスアカウントに適用、長さ 25 文字以上を強制
  • AES-only Kerberos 暗号化に切替、RC4 サポートを削除 (Kerberoast 実効性を大幅低下)
  • Honeytoken SPN アカウントへの Event 4769 (etype 23 = RC4) を SIEM でアラート
  • アカウントロックアウト閾値 5 attempts / 30 min リセット
  • LSA Protection (RunAsPPL=1) + Credential Guard で LSASS メモリ抽出を阻止
  • 特権アカウントを Protected Users グループに配置 (NTLM 拒否、Kerberos AES 強制)
要点 / key takeaways
  • Kerberoast は SPN 保有ユーザで pre-auth 有効でも成立、Event 4769 で etype 23 request が signature
  • ASREProast は DONT_REQ_PREAUTH (UAC 4194304) 属性を持つユーザのみ、AS-REP から直接クラック
  • Password spray は Discovery で lockout policy 取得済み前提、Kerbrute は AS-REQ ベースでログ量少
  • DCSync 権限は非 DA でも保有しうる (delegated principal)、Find-InterestingDomainAcl で事前確認

powershell

防御 / countermeasures
  • Protected Users グループへ特権アカウント配置 (NTLM 拒否、Kerberos AES 強制、delegation 対象化ブロック)
  • 特権アカウントに Account is sensitive and cannot be delegated チェック
  • Unconstrained Delegation は 2019+ で無効化、必要なら Constrained (Kerberos-Only) へ移行
  • ms-DS-MachineAccountQuota を 0 に設定 (Domain User による machine account 追加を阻止 → RBCD 攻撃無効化)
  • LAPS ACL の見直し、Read All / All Extended Rights の付与範囲を最小化
  • CA テンプレートの SAN 属性 (CT_FLAG_ENROLLEE_SUPPLIES_SUBJECT) と enroll permission を audit、ESC1-15 対応
  • Web Enrollment で EPA (Extended Protection for Authentication) を強制、LDAP + SMB signing 必須化
要点 / key takeaways
  • ACL パスは BloodHound で発見、WriteDACL on domain object は DCSync 権限化の直行経路
  • Unconstrained + Spooler / PetitPotam は DC$ TGT 捕獲 → DCSync の高速経路
  • RBCD は ms-DS-MachineAccountQuota 未 0 化ならほぼ確実に成立、Computer への GenericWrite で完結
  • AD CS ESC1 は SAN 指定可能 vulnerable template を発見した時点で任意ユーザ化可能
  • ESC8 (Web Enrollment Relay) は coercer と組み合わせ、DC 自身を認証源として利用可能

04Lateral Movement横展開

取得済み hash / ticket / plaintext を用いた別ホストへの実行手段。プロトコル別 (SMB / DCOM / RDP / WinRM) の選択は検知率・要件・admin 権限有無で決まる。

Pass-the-Hash (PtH)

bash
# psexec (ADMIN$ + service install、最高検知率)

防御 / countermeasures
  • Credential Guard (VBS) を Enterprise SKU 上で有効化、LSASS メモリ抽出防止
  • Restricted Admin Mode は既定無効を維持、必要なホストのみ許可
  • LAPS で local admin パスワードを一意化、横展開の hash 再利用を阻止
  • Tier 0 (DC / DA) と Tier 1 (Server) の管理境界を厳格化、DA を workstation に Session させない
  • SMB signing 強制 (RequireSecuritySignature=1) で SMB relay 阻止
  • Windows Firewall で 445 / 5985 / 3389 の inbound を Tier 内に限定
  • Sysmon Event 1 で psexec.exe / wmic.exe の unusual parent を検知、Event 4624 の Type 3 (Network) + NtLmSsp を SIEM で aggregate
要点 / key takeaways
  • 実行方法選択: psexec (最高検知) < smbexec < wmiexec < atexec < dcomexec (最低検知)
  • NetExec の bulk 実行は探索フェーズで有効、--local-auth は local hash 再利用検証にも使用可能
  • PtT は -k -no-pass + /etc/hosts FQDN 登録が前提、時刻同期は必須
  • Restricted Admin Mode RDP は NTLM hash で GUI ログオン可能、レジストリで有効化
  • Over-PtH は新プロセスで Kerberos 経路に切替、以降の hop は NTLM 痕跡を残さない

05Persistence永続化

DA / DC 取得後、krbtgt / trust key / DSRM / AdminSDHolder を使った長期アクセス確保。DCSync で krbtgt を取得すれば Golden Ticket による forest-scope の永続化が完成する。

防御 / countermeasures
  • krbtgt account パスワードを年 1 回 + 侵害疑いの都度 2 回連続でリセット (Golden Ticket 無効化)
  • Event 4769 で Ticket Options: 0x40810000 + etype 0x17 (RC4) の異常を SIEM で aggregate
  • Event 4662 で AdminSDHolder GUID (bf967a86-0de6-11d0-a285-00aa003049e2) への ACL modification を alert
  • LSA Protection (RunAsPPL=1) + Credential Guard で DSRM / mimikatz 系メモリ抽出阻止
  • Golden / Silver Ticket 検知は Rubeus / mimikatz プロセス signature + Kerberos anomaly (ticket lifetime 10 years、AS-REQ なしの TGT) の組み合わせ
要点 / key takeaways
  • DCSync 権限保有ユーザは非 DA でも存在しうる (Find-InterestingDomainAcl で発見)
  • Golden Ticket は AES256 (etype 18) 使用で RC4 検知 (Event 4769 etype 23) を回避
  • Silver Ticket は KDC 通信なし (AS-REQ / TGS-REQ ログなし)、単一サービスへのアクセスに最適
  • Skeleton Key は DC 再起動で消失、Golden Ticket 併用で永続化担保
  • AdminSDHolder Persistence は SDPROP (60 分間隔) の伝播で全特権 principal に ACL 拡散、検知極めて困難

06Impact侵害拡大

単一ドメインの侵害を child → parent / cross-forest に拡張する層。SID History injection と Trust Key 抽出が主軸で、fully-forest 侵害 (Enterprise Admins 化) が目標。

SID History 注入 (Child → Parent)

ACL Cross-Domain Abuse

Foreign Group Membership / Trust ACL 悪用。BloodHound の Cross Domain クエリで可視化。外部 principal に付与された ACL は trust 越えの侵害経路になる。

防御 / countermeasures
  • Forest boundary を security boundary として扱い、不要な外部 forest trust を撤廃
  • SID Filtering (netdom trust /quarantine:yes) を全 trust で有効化 (external では既定有効、intra-forest では無効)
  • Selective Authentication に切替、必要 principals のみ trusted domain resource へアクセス許可
  • Trust password の定期ローテーション (30-60 日)、nltest /trust_kerberos で監査
  • Enterprise Admins グループの Event 4728 / 4729 (add/remove member) を SIEM で alert
  • 各 domain の krbtgt を分離管理、child 侵害が forest root DA に伝播しない設計を検証
要点 / key takeaways
  • SID History (/sids) はフォレスト内 trust で既定 SID Filter 無効化のため成立
  • Trust key ベースの inter-realm TGT 偽造は cross-forest でも有効、Event 4769 で異常な service ticket (source domain ≠ requesting user domain) を検知
  • Foreign Group Membership は BloodHound の cross-domain クエリで可視化、trust 関係の可視化不足が典型的侵害ルート
  • Enterprise Admins 化 = forest root DA、Schema Admins は既定でメンバー 0 だが Schema modification 権限保有
