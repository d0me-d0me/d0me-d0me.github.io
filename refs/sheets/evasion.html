弐Evasion

Windows 上の可視化レイヤ(AMSI・ETW・スクリプトブロックログ)がどこで何を見ているか、そして回避が成立/失敗する境界。手法は概念と公開ツール参照に留め、検知観点を厚く扱う。 目的は「見えている場所」を理解すること。丸写しのバイパス文字列は載せない。

01可視化レイヤwhat is watching

回避を語る前に、Windows がどこで挙動を観測しているかを分けて考える。層ごとに「何が見えるか」「無効化がどれだけ痕跡を残すか」が違う。

層見えるもの回避時の痕跡
AMSIスクリプト/マクロの実行前バッファプロセス内改変は EDR のメモリスキャンで露見しうる
ETW.NET ロード、API 呼出等のテレメトリプロバイダ無効化自体がシグナル
Script Block LoggingPowerShell の逆難読化後スクリプトEvent 4104。無効化は監査対象
EDR user-mode hooksNt* API 呼出のフックアンフック操作が検知される場合あり
前提ここに載る手法はラボ/CTF/公開 CVE の再現が前提。実運用の EDR 回避データは公開ノートに含めない。

02AMSIAntimalware Scan Interface

AMSI は PowerShell・WScript・.NET などが「実行しようとしている内容」をスキャンエンジンへ渡すための橋。スクリプトはメモリ上で復号された後にスキャンされるため、単純な文字列難読化だけでは通り抜けない。

公開されている回避カテゴリ(概念)

  • 初期化失敗の誘発 — ロード済みマネージド状態のフラグ操作で以降のスキャンを無効化する系統。
  • スキャン関数のメモリパッチ — AmsiScanBuffer 相当をプロセス内で書換え、常に「clean」を返させる系統。
  • プロバイダ側の回避 — そもそも AMSI 経由で実行しない(ネイティブロード等)系統。
重要これらの実装文字列は世代ごとに署名され陳腐化する。AMSI bypass 実装は検証環境で都度確認するのが実際的で、公開ノートに動作コードを固定掲載する価値は低い。
防御 / countermeasures
  • AMSI をバイパスするプロセス内メモリ改変は、EDR のランタイムメモリスキャンや RWX 領域監視で捕捉しうる。
  • PowerShell の言語モード制限(CLM)+ WDAC/AppLocker で「そもそも任意コードを走らせない」層を足す。
  • AMSI 無効化に伴う異常(scan 結果の急変、既知パッチのバイトパターン)を検知ルール化。

03ETW とログtelemetry

ETW は .NET アセンブリのロードや各種 API をイベントとして流す。EDR の多くはここに乗る。無効化(プロバイダの停止・パッチ)は有効だが、無効化そのものが強いシグナルになる点が要点。

  • Script Block Logging (4104) — 逆難読化後のスクリプトを記録。難読化は無力で、むしろ復号後が残る。
  • Module Logging / Transcription — パイプライン全体を記録。
  • PPL 保護 — ログ/AV プロセスの改変耐性を上げる。
防御 / countermeasures
  • Script Block / Module Logging を GPO で強制し、集約 SIEM へ転送(ローカル改変の影響を減らす)。
  • ETW プロバイダの停止・改変を、プロバイダ状態の欠損として検知。
  • PowerShell を Constrained Language Mode + 署名要求で運用。

04インメモリ実行disk を触らない

ディスク書込を避けると静的検知の面は減るが、メモリ側の監視面が増える。トレードオフの位置取りが要点。公開ツールの用途整理:

道具用途
donutPE/.NET/スクリプトを position-independent シェルコード化
SharpShooter各種ステージャ生成(検証用テンプレート)
reflection load.NET アセンブリをディスクに置かずロード
観点「disk 静的 → memory 動的」へ検知面が移るだけ。どちらの面が薄い環境かを列挙で見極めてから選ぶ。
防御 / countermeasures
  • RWX メモリ確保、リフレクティブロード痕跡、未バックド DLL のメモリ実行を EDR で監視。
  • Attack Surface Reduction ルールでスクリプト系ステージャの実行を抑止。

05難読化の考え方署名を外す

難読化はシグネチャ回避の手段であって、振る舞い検知には効かない。有効なのは「静的署名を外す」局面で、「実行後のログ・メモリ」には無力という切り分けが実務上の要点。

  • 文字列分割・エンコード(XOR やカスタムエンコーダ)は AMSI 前段では有効、4104(復号後)では無力。
  • API 名の間接解決、シンボル除去はホーリスティックな静的検知に効く。
  • 振る舞い(親子プロセス関係、LOLBIN 連鎖)は難読化で消えない。ここは別問題として扱う。
要点 / key takeaways
  • 回避は「無効化」ではなく「観測面の選択」。層ごとに見えるものと痕跡が違う。
  • AMSI/ETW の無効化は有効だが、無効化した事実自体が検知シグナルになりうる。
  • 難読化は静的署名向け。Script Block Logging(復号後)と振る舞い検知には効かない。
  • バイパス実装は世代で陳腐化する。固定文字列を覚えるより、検証環境で都度確認する運用が正しい。
